# NOTES.md — background & rationale

Forensic detail and "why we did it this way" narrative moved out of `CLAUDE.md` to keep that file to actionable rules. Nothing here is required to follow the rules; consult it only when you need the history behind a decision. Rules live in `CLAUDE.md`; this is the "why".

## Codebase-health F5 — split send_digest into a `fantasy/` package (2026-07-18)

The last refactor step: break the 7,334-line `send_digest.py` into a bottom-up package with `send_digest.py` kept as a re-export facade (`from fantasy.<mod> import *`), so every downstream caller (dashboard/trade_lab/weekly_recap/CI) is zero-touch. Shipped **ui.py** (presentation leaf, ~551 lines) + **scoring.py** (scorers + calibration, ~592 lines); send_digest.py 7334 → 6191.

- **Method — byte-preserving AST move, not hand-editing.** Each module was built by a script that parses send_digest with `ast`, slices each moved node by `lineno..end_lineno` (exact source bytes, CRLF + BOM preserved), and reassembles them; then a second script deletes those ranges from send_digest and inserts the facade import. A static undefined-name pass (ast, minus builtins + the lower layer's exports) caught the one real missed dependency (`_ERA_REG_PRIOR_IP`) before running anything. The render-diff harness is the acceptance test — all five previews byte-identical.
- **The `import *` + underscore problem.** `from module import *` skips underscore names unless `__all__` says otherwise. Each `fantasy` module ends with `__all__ = [n for n in dir() if n not in _EXCLUDE and not n.startswith('__')]` where `_EXCLUDE = set(dir())` is captured right after the imports — so `__all__` auto-includes every *defined* name (incl. single-underscore helpers like `_hit_badge`) and excludes imported names. No hand-maintained export lists to drift.
- **The one real footgun — mutable module globals across the facade (why scoring was the risk).** `from fantasy.scoring import *` binds send_digest's names to scoring's objects *at import time*. For a global that's **rebound** (`global X; X = ...`), send_digest's binding then goes stale. For a global **mutated in place** (`X.clear(); X[k]=...`), send_digest's binding is the same object and sees every update. The four calibration dicts (`_LG`/`_PIT_BENCH`/`_AB_BENCH`/`_SCORE_CALIB`) are all in-place → safe to read from send_digest across the boundary; the two rebound globals (`_XERA_OFFSET`, `_POS_SCARCITY`) are each read by exactly one function, so co-locating each with its reader avoids any fork. This is *the* invariant to preserve when touching the split (documented in CLAUDE.md). The render-diff would catch a fork (it changes nearly every score), but the in-place-vs-rebind reasoning is what let it ship correct the first time.
- **Why the split paused at ui + scoring.** A dependency-closure analysis of the trade engine (seeded from `find_trades`/`_grade_pending_trades`/etc.) came back at **75 members**: it pulls in the shared analytics (percentiles, `positional_breakdown`, roto ranks, `_on_il`) AND the whole score-breakdown-narrative subsystem — but **zero `build_*` section renderers** (no upward dependency), so a further extraction IS feasible/verifiable. The open call is how to slice it: one large `trades.py`, or a separate `analytics.py` (percentiles/positional/roto) under a leaner `trades.py`, then `sections.py` (which also wants `build_email`'s 8 inline sections extracted first). Left as a follow-up because the boundary is a design choice, not a mechanical move.

## Codebase-health F4 — one canonical `_name_key` (2026-07-18)

Step 3 of the codebase-health plan. There were **three** name-normalization functions that disagreed on edge cases: `fetch_data._name_key` (the upstream FantasyPros↔ESPN merge key — accent/suffix strip, `.`/`,`→space, keeps apostrophes/hyphens, ≥2-token guard), `weekly_recap._name_key` (dropped ALL non-alphanumerics), and `send_digest._badge_name_key` (`_ascii_lower` + drop `.`/`'`, keep `-`). They could produce different keys for `A.J.`/`O'Neill`/`Kiner-Falefa`-type names.

- **Fix:** promoted fetch_data's (the most defensive) into a **stdlib-only leaf module `name_utils.py`** that `fetch_data` and `send_digest` both import (neither imported the other, so a leaf module is the only clean shared home — the producer must not depend on the reader, and pulling fetch_data's heavy espn_api/pybaseball deps into the reader would break the instant `--dry-run`). `send_digest._badge_name_key` is now a **back-compat alias** of `_name_key`; `weekly_recap` imports `_name_key` from send_digest and deletes its local copy. fetch_data's own behavior is byte-unchanged (it keeps driving the snapshot merge), so the snapshot itself is untouched.
- **Verification:** the render-diff harness proved digest / briefing / dashboard / Trade Lab **byte-identical**. An empirical pass over the 359 ESPN-side lookup names in the live snapshot found **0** changed lookup outcomes, and the only key collision under the canonical key is the already-known-ambiguous `Luis Garcia` / `Luis Garcia Jr.` pair (identical to the old functions — no NEW collisions).
- **The one intentional diff — and a real display bug it surfaced (`Ryan O\'Hearn`).** The recap changed by exactly one character. Root cause: `recent_hitting`/`prev_week_*` names arrive from the pybaseball/Baseball-Reference scrape with a **literal backslash-escaped apostrophe** (`Ryan O\'Hearn`) that `_fix_mojibake` didn't clean (it only handled `\xNN`). The OLD recap key masked this by stripping ALL punctuation (incl. the stray `\`), so O'Hearn matched his roster row — but his **displayed** name still showed the ugly `O\'Hearn`. The shared canonical key PRESERVES apostrophes, so an uncleaned `o\'hearn` no longer matched `o'hearn` → he'd have wrongly dropped to FA. The correct fix is separation of concerns: `_fix_mojibake` cleans the dirty source name (now handles `\'` too, before the `\x` early-out), and `_name_key` normalizes a clean name. Net effect: O'Hearn still matches AND his name now renders correctly (`Ryan O'Hearn`) — a strict improvement. Five names carry the `\'` artifact (O'Hearn/O'Hoppe/O'Neill/O'Brien/Ke'Bryan Hayes); only O'Hearn currently reaches a rendered table.


## Trade Radar — design rationale (2026-07-10)

New FREE-AGENTS-band section (`find_trades` + `build_trade_radar`, send_digest.py). The digest was already exhaustive on the two levers a manager pulls solo — roster moves and FA pickups — but blind to trades, even though every input a trade finder needs was already in the snapshot (all 12 rosters via `FantasyTeam`, `season_cat_totals`/`roto`, and team-agnostic scoring/percentile helpers). Trade Radar mines that.

Design choices (all confirmed with the user before building):
- **Mutual-benefit only, not "helps me."** A one-sided "give me your closer" list is noise — nobody accepts it. Requiring `my_surplus ∩ their_need` AND `their_surplus ∩ my_need` both non-empty means every card is a deal the other manager has a reason to say yes to. Fewer cards, but each is proposable. This is the whole point of the feature.
- **Season-category-fit ranking, not current-matchup win-prob.** A trade is a season-long asset move; ranking by the `_project`/`_cat_win_prob` swing vs *this week's* opponent would over-fit to a single matchup and flip week to week. Instead: need-depth (the roto rank number itself — rank 12 = deepest need = highest weight) of the cats the deal addresses, plus a half-weighted their-side term to enforce realism, minus a value-gap and a mild package-size penalty.
- **Surplus-POSITION gate on the outgoing side.** Value parity alone would happily suggest trading a stud you can't afford to lose. Gating outgoing players to positions where I rank top-third (reusing `pos_data` — same surplus test as `_roster_suggestion`) means a trade never opens a hole. This is why the engine correctly refused to trade from teams whose only surplus category had no strong player at a deep position.
- **QS and B_SO can't be traded for.** Neither is a per-player stat (QS is a team pitching cat; B_SO is team batter-strikeouts), so `player_cat_strengths` has nothing to match — a team need in those simply finds no candidate. Same intentional limitation as the FA "Cats" column, not a bug.
- **Value parity uses a CROSS-ROLE trade currency, NOT the role-score badge** (fixed same day, see below).

### Trade value: cross-role currency, not role-score parity (the "James Wood for a top closer" bug)
The first cut gated value parity on the canonical role score (`_tscore` = `_blend(hitter_score)` / `_score_p`). That was wrong: those scores are calibrated WITHIN role (p50→50/p90→80 among hitters, among closers, etc.), so a 95 closer and a 98 hitter look "equal" — and the engine happily offered James Wood / Munetaka Murakami (everyday smashers) straight up for a top-3 closer. Insane in roto: an everyday bat contributes to 5 categories every day; a closer touches one PUNT category (SV+H, deliberately de-emphasized in our whole build) plus a sliver of K/ERA/WHIP in ~60 IP. `_trade_value` replaces it: `Σ max(0, pctile(c) − 0.5) × w` across the player's role cats, saves weighted `_TRADE_SVHD_W=0.35`. Two properties make it right: (a) percentiles are vs the WHOLE qualified pool, so a reliever's low K/W **volume** ranks low (a starter's high K ranks high) — cross-role volume is captured for free; (b) hitters sum across 5 cats, closers across ~2 meaningful ones, so a smasher's value (~1.8) dwarfs a closer's (~0.7). Parity gate `_TRADE_VAL_PARITY=0.45` (×1.5 on 2-for-2 sums). The role-score badge is still DISPLAYED (familiar), but fairness is judged on `_tval`. After the fix, Wood/Murakami-for-a-closer vanished; elite bats only move in balanced 2-for-2s.

### Trade needs include POSITIONAL holes, not just categories (hitting is roster-filling)
User's second point: "hitting is about filling all those [positional] holes, whereas pitching is about balancing QS/SP and SV+H/K." Category-rank needs alone are too coarse on the hitting side — with SV+H as my only bottom-third category, the engine could ONLY ever hand me closers. So the get side also matches `_fills_need_pos`: a hitter at one of my thin hitter positions (`need_pos` = `pos_data` bottom-third rank) whose `_tscore` beats my current average there (a real upgrade, not a lateral move). Positional need is **hitter-only** by design — pitching need is genuinely category-shaped (the QS/SP-vs-SV+H/K balance), not roster-slot-shaped. The team gate relaxed from "both send_cats AND get_cats non-empty" to "send_cats non-empty (they benefit) AND the get side yields a category OR positional fill." This is what finally gave the radar variety — my thin position (C) now surfaces catcher targets alongside the saves help.

### Favor-me + good-not-great + sell-high (2026-07-10, third pass)
User pushback on the even-value model: "I'll be hard-pressed to move my ace batters — HRs are too fun and hard to replace. Find good rather than great trades. Help me more than both teams. And leverage my sell-high guys." Three coupled changes:
- **Good-not-great / protect my studs:** a give-side value ceiling `_TRADE_MAX_VAL=1.35` — my elite bats are never offered, EXCEPT a sell-high regression candidate (whom I *want* to move, so he's exempt). Mid-tier ↔ mid-tier deals result.
- **Favor-me (not even):** symmetric parity → directional `_favor_me`: I never overpay by more than `_TRADE_GIVE_SLACK`(0.15) and may extract up to `_TRADE_MAX_EDGE`(0.45). The rival still accepts because the deal fixes a category NEED of theirs (`send_cats` required) — acceptance is need-driven, so value can tilt to me. Ranking down-weights their benefit (0.5→0.3) and adds `+5·net_val`.
- **Sell-high / buy-low is a DOUBLE ARBITRAGE, and it's DIRECTIONAL.** `_regression_flag` (same xBA/xSLG thresholds as the `$`/`▼` hitter badges) → `_tsell`/`_tbuy`. Selling-high on the GIVE side + buying-low on the GET side both help me: my sell-high guy's actual stats OVERstate his real value, their buy-low guy's UNDERstate his — so a deal that's even on `_tval` actually favors me. The REVERSE is a trap (dealing a riser / acquiring a regressor), so `timing = sell_out + buy_in − buy_out − sell_in` and the card chips the trap direction ORANGE ("selling low?" / "regression risk"). A first cut rewarded any sell/buy leg regardless of side and surfaced a "buy Liam Hicks [SELL]" deal (acquiring a regressor) — the directional fix penalized it out. Pitcher buy/sell is a FUTURE feature (`_tsell`/`_tbuy` computed for hitters only) — the ERA-vs-xERA luck analog would let the radar sell-high a lucky reliever too.
- **Validation (2026-07-10 snapshot):** top card nets **+0.54** giving only mid-tier (Kody Clemens `_tval` 1.06 + Ashby) for a closer + my catcher hole; clean 1-for-1s **Brandon Marsh [sell-high] → Hunter Goodman (C)** and **Jake McCarthy [sell-high] → Shea Langeliers (C)** both net positive AND sell high. No elite non-sell bat leaves; the buy-a-regressor trap no longer surfaces.

### Pending-trade evaluator — grading REAL offers (2026-07-12)
User gets actual trade offers in ESPN and wanted the tool to say accept/decline/counter. Two facts drove the design:
- **The data wasn't anywhere.** `espn_api` doesn't expose pending trades and the snapshot never fetched them. The raw endpoint `.../leagues/{id}?view=mPendingTransactions` returns them — but under the top-level **`pendingTransactions`** key, NOT `transactions` (my first probe keyed on `transactions` and got 0 rows; the `mPendingTransactions` view's `transactions` key is empty — the data is in `pendingTransactions`). `memberId == my SWID` is the reliable "I proposed it" signal (→ `incoming=False`); direction per player comes from `fromTeamId`/`toTeamId` vs my team id.
- **Don't re-derive grading — reuse Trade Radar.** The radar already had the right currency (`_tval`), role scores, buy/sell timing, and need-fit logic; the only new thing is evaluating a GIVEN set of players instead of searching for hypothetical ones. So `_trade_value` and the player-`enrich` step were promoted out of `find_trades`' closures to module-level (`_enrich_trade_player`) — one implementation, no drift. `build_pending_trades_section` resolves each traded name to its snapshot YEAR row (`_badge_name_key`, 30→15→7 fallback, like Today's Games), enriches, and grades on `net_val = get − give` with the same ±0.1 value-tilt thresholds.
- **Verdict is need-AND-value, not value alone.** ACCEPT when I win value (or roughly even AND it fills a real category/positional need) and timing isn't a net trap; COUNTER when the direction is right but I pay up or would be selling a riser / buying a regressor; DECLINE when nothing is addressed and I don't gain value. Incoming-only — an outgoing offer I proposed just shows "awaiting {partner}" since it's their call.
- **Validation (2026-07-12, real offers):** the live incoming offer — Gabriel Moreno (C, `_tval` low) for Grant Taylor (RP, `_tscore` 87) — graded **COUNTER**: it fills my thin C slot (right direction) but Taylor is the more valuable chip, so "you pay up — ask for more." That matches the intuitive read, so the reused currency transfers correctly to real offers.

### Pending trades — surface HIGH + suggest a counter (2026-07-12, second pass)
User feedback after seeing v1: "I had to search a little for it — the heads-up belongs in Week at a Glance or higher since offers expire; and is it in the email body? And if it says COUNTER, can it suggest a counter that makes sense?"
- **Placement was the miss.** v1 put the incoming-offer nudge only in the roster-alerts box (under the MY ROSTER band, well below the fold) and NOT in the Briefing (the inline email body) at all. Fix: grade the offers ONCE (`_grade_pending_trades`, right after `pos_data`) and feed the headline (`_pending_headline`) into the two highest-visibility surfaces — the top of **Week at a Glance** (`trade_bullets`) and the **Briefing "⚡ Act today"** list (`pending_incoming`, as the FIRST items since they expire). Dropped the alerts-box nudge (redundant once it's higher). The expiry ("expires in Nd", RED) rides along everywhere so the time pressure is visible.
- **Counter = the cheapest realistic add-on that evens it.** For a COUNTER (I'm overpaying by `gap = −net_val`), `_counter_suggestion` scans the partner's roster for the best add-on to request: a piece whose `_tval` closes the gap (`≥ 0.7·gap`) without being an unrealistic overreach (`≤ gap + _TRADE_MAX_EDGE`), preferring one that helps a category/positional need of mine (and buy-low timing). Validated: Moreno-for-Taylor → "ask them to add **Tyler Wells** (adds SV+H)" — a spare reliever of theirs that both closes the value gap and helps my saves need. Deliberately an *add-on* ask, not a give-side swap (simpler, universally applicable; give-side reshaping is a bigger negotiation).
- **Grade-once was the enabling refactor** — three surfaces reading one computation means the verdict/counter/expiry can never drift between the email body, the headline, and the section card.

### Pitcher buy/sell badge (ERA vs xERA) + the de-bias offset (2026-07-10, fourth pass)
User asked whether the hitter `$`/`▼` buy-low/sell-high badges should apply to pitchers too, and whether the ▼ sell would make the ⚠ blowup-risk flag redundant.
- **Not redundant — orthogonal axes.** ▼ sell-high = MEAN regression / luck (ERA better than the peripherals deserve, due to rise); ⚠ = single-start TAIL risk / low floor (variance). A soft-contact arm riding a low BABIP is sell-high but not blowup-prone; a hard-contact arm whose ugly ERA already matches an ugly xERA is blowup-prone but not sell-high (no luck to give back). They share the ERA/xERA input via `_effective_era`, so they sometimes co-fire — which is a *stronger* "move him" signal, not duplication. Kept separate; glossary draws the line.
- **The de-bias offset was the real subtlety.** Raw `xERA − ERA` over the YEAR pool has a systematic **+0.33 median** (xERA runs high across the board in this data). A symmetric raw threshold therefore over-fired "sell" (37% sell vs 17% buy at a 0.60 gap) — useless as a selective glance flag. Fix: `compute_xera_offset` sets `_XERA_OFFSET` = league median gap; the flag tests `(xERA − ERA) − _XERA_OFFSET` (relative luck) against `_XREG_ERA=1.00` → ~12% sell / 15% buy, matching the hitter-badge selectivity. Same data-derived-calibration pattern as `_LG`/`_SCORE_CALIB`, computed in `build_email` right after `compute_league_averages`. Validation names were exactly right: buy-low = Strider (5.31/3.49), Devin Williams, Nola; sell-high = Eduardo Rodríguez (2.25/5.01), Robert Suárez (0.56/3.27).
- **Feeds Trade Radar:** `_tsell`/`_tbuy` now computed for pitchers too, so the radar can sell-high a lucky reliever and buy-low an unlucky arm (a card now acquires Morejón [BUY]). Timing-chip tooltips made role-neutral ("results running ahead/behind expected") so they read right for a pitcher. NOT yet on the dashboard — possible follow-up.

### Two-sided trade acceptance: star reluctance + demand-side need value (2026-07-15)
User: some radar deals tagged "realistic" would be rejected on sight, and the Lab should show whether BOTH managers would say yes. Two coupled ideas, both kept in the **acceptance layer only** — `_tval` stays a pure category-value currency (real ACCEPT/COUNTER/DECLINE verdicts, fair-lane gating, and scarcity math all depend on that purity; convex-ing `_tval` would corrupt them).
- **Graduated star reluctance (supply side — won't ship quality cheap).** `_tval` COMPRESSES stars: a category compiler (e.g. Otto Lopez, `_tval` 1.61 / score 76) can out-total a genuine star (Witt, 1.45 / 87), so acquiring a star at even `_tval` read "realistic" when no manager would ship their crown jewel at par. This is a market-perception / endowment effect, not a value error — hence the acceptance layer. First cut was a binary cliff (score ≥ 80, must pay up 0.25) which gave a very good 76–79 player *zero* reluctance. Replaced with a smooth premium rising with role SCORE (`_star_reluctance`: 74→0.00, 82→0.40, 94→1.00, cap 1.10). `_deal_star_reach` (rival balks — required overpay = their-best-acquire premium − my-best-give premium) and its mirror `_deal_star_surrender` (I hold out) both run on **base `net_val`**. Relievers excluded from both sides (cross-role incomparability — the Witt-for-McCarthy+reliever bug). `_deal_star_surrender` downgrades an incoming ACCEPT → COUNTER when a rival pries my star at par.
- **Demand-side team-need value (`_need_mult` → net_me / net_them).** The star curve is supply-side; this is the complement — the SAME player is worth more to a team that needs his position/category than to one already set there, which is what makes trades mutually agreeable. `_need_mult` scales `_tval` by need coverage (per need cat, thin position, minus surplus-only help; clamped). NOT folded back into `_tval` or the `find_trades` ranking (which already rewards need coverage — would double-count). Used for the acceptance read only: `net_them` drives the rival tilt ("Would they do it?"), `net_me` drives the roster-aware incoming-offer verdict. Displayed value phrase + counter-gap stay on base `_tval` (the universal read). Trade Lab surfaces both POVs via a three-row Base / My value / Their value block, so a base-negative but mutually-beneficial deal shows both team rows positive. All tuning constants baked into `DATA.tune` from the send_digest globals so the Lab JS mirror can't drift. A board reading only base `net_val` had tagged a partner-LOSING deal "realistic/BEST" while the builder correctly said "aggressive ask" — fixed by threading `net_them` into `_trade_tilt` at all three callers.

### Both-parties roster-depth floor — "their only catcher for 2 OF" (2026-07-16)
User: the radar tags deals "realistic" that leave a team short at a scarce position — the flag was *a rival's only catcher for two of my outfielders*, which no manager accepts (C vs OF scarcity). Root cause: **every positional guard was computed from MY roster only.** My give side is gated by `surplus_pos` and `_non_redundant_get_pos` stops me over-stacking, but the partner's give side had no equivalent, and `net_them` called `_need_mult` WITHOUT the positional args (the code comment even admitted "rival positional need isn't plumbed into the digest"). `_POS_SCARCITY` boosts a catcher's `_tval` league-wide but is capped ×1.50 and can't represent going 1 C → 0 at a mandatory slot; because catcher category `_tval` collapses to ~0, two productive OF out-value the lone C and the deal nets out "realistic."
- **The fix is a body-count depth floor INDEPENDENT of `_tval`.** `_leaves_position_short(team_counts, give_players, get_players)` → hitter positions where giving/receiving drops a team below `POS_STARTERS[P]` bodies with no same-position body back (SP/RP excluded — pitching need is category-shaped, and SP/RP floors of 4/3 would falsely veto reasonable arm swaps). This is a structural check the value currency can't express, so it lives OUTSIDE `_tval` (same discipline as star reluctance / need-mult — the acceptance layer, not the currency). No new plumbing: `_my_position_counts` was already team-general, just renamed `_team_position_counts` and called per rival.
- **Two enforcement levels (user picked both).** HARD VETO in `find_trades` (both parties) — a package that leaves either team short is never generated. HONEST READ for survivors: a deal leaving the RIVAL at exactly the floor at a single-slot position (`thin_them`) subtracts `_TRADE_THIN_POS_PENALTY`=0.15/slot from `net_them` so `_trade_tilt` flips to "aggressive ask," + a `thin_note` ORANGE line ("leaves them without a backup at C"). Real pending offers aren't vetoed (they exist) — instead `_pending_verdict` gets a `leaves_me_short` flag that downgrades an ACCEPT→COUNTER when accepting would thin ME out. **Floor = `POS_STARTERS` (startable), so acquiring a team's only catcher requires sending a catcher back** — matches real manager behavior (deliberately blocks ALL cross-position acquisition of a one-deep slot, which is correct: you trade C-for-C, not C-for-OF).
- **Validation (2026-07-16 snapshot):** the two Elly De La Cruz (SS) radar deals for Blubaugh's now carry "leaves them without a backup at SS" + read "aggressive ask" (Blubaugh's has 2 SS → post 1 = floor → thin, not vetoed). Python `_leaves_position_short` unit-tested (only-C-for-2-OF → {C}; C-for-C swap → clean; multi-elig depletes each slot; SP ignored). Trade Lab JS mirror (`leavesShort`/`thinPos`) exercised in headless Chrome against the real rendered functions — all 5 cases pass. **JS keeps the value-display rows pure** and applies the thin penalty only to a read-only `netThemRead` (the digest bakes it into `net_them` because it has no separate value display but a `thin_note`; the Lab shows the value block explicitly, so purity + a gated read is the better UX — same behavior, different surfacing).

### Trade Lab Partner Fit board — why a server-side engine call in a JS-only tool (2026-07-15)
The Lab's rule is "JS only sums pre-computed per-player numbers" (keeps it emailable-adjacent and driftless). Partner Fit ("who should you be trading with?") deliberately breaks that: it calls the real `find_trades_combined` **server-side at build time**, once per team's POV, to surface one concrete engine-graded deal per rival in plain English. Reason: ranking every rival by how landable a deal is, and picking the single best deal each, is exactly what the engine does — re-implementing that in JS would duplicate hundreds of lines and inevitably drift from the digest. `dashboard.py` already makes the same contained exception (it calls `find_trades_combined` too). Caps are temporarily raised (400/6, restored in a `try/finally`) so every rival surfaces its best deal, not just the digest's league-wide top ~6. Per-POV serialization (`DATA.partnerFit[pov_key]`) keeps the board correct when the LEFT dropdown switches teams.

### Pocket mode — phone Trade Lab with one-tap refresh (2026-07-15)
Goal: run the Trade Lab on the user's phone and refresh the data on demand, with no laptop and no schedule. The page is already self-contained (data baked in, JS only sums), so the only new machinery is the refresh trigger. Design: GitHub Pages hosts `public/index.html`; an in-page ↻ button POSTs to a Cloudflare Worker (the ONLY holder of a fine-grained GitHub `Actions:write` token — a token can't be safely baked into a public page); the Worker fires a `repository_dispatch` that runs `pocket-tradelab.yml` (`trade_lab.py --refresh --out public/index.html`) and deploys via Pages; the page polls a sibling `build.json` and reloads when the build timestamp changes. Chose a Worker proxy over baking a token (public page) and over a schedule (user wanted on-demand, and the data only matters when he's actually looking). One-time setup (Pages source, fine-grained PAT, Worker deploy, `POCKET_REFRESH_URL` repo variable) lives in `worker/README.md`. Live URL + token-renewal gotcha are in the [[pocket-tradelab]] memory.

## Single-viewport dashboard — layout rationale (2026-07-09)

Why the desktop layout is pinned the way it is (`dashboard.py`):

- **`#wrap{position:fixed;inset:0}` is load-bearing.** The first attempt used `height:100vh` + `flex:1` on the grid; that collapsed the grid to ~half viewport height. Pinning `#wrap` to the viewport fixed it.
- **Design for a REAL viewport, not 900px.** Browser chrome (tabs/URL bar) shaves usable height to ~768–820, so a tile tuned to exactly fill 900 clips its bottom rows on a real screen (this is why FA-radar hitters "disappeared" in testing). Keep the vertical rhythm compact (small gaps, `line-height` ≈ 1.4) so everything survives ~768 usable — verify with a headless screenshot at 1440×768, not just 900.
- **Content-heavy tiles get more `flex`** (Weakest/Lineup shares col 1 with only Category Pulse; Opponent → col 2). Don't equalize the weights blindly.

## Lineup efficiency / bench leakage (2026-07-08)

Answers a user question: *can we see which player stats actually counted toward the matchup, so we can catch (a) a bat that homered on my bench and (b) a starter I let implode in my active lineup then dropped?* Yes — but not through the obvious path.

**Why `box_scores` isn't enough:** this is a **categories** league, so `league.box_scores()` returns `H2HCategoryBoxScore`, which carries only team category totals — no per-player lineup. The `home_lineup`/`away_lineup` `BoxPlayer` list only exists on the **points**-league box class. The per-player, per-day slot instead comes from `league.espn_request.league_get(params={"view":"mRoster","scoringPeriodId":<day>})`. Verified experimentally that ESPN **preserves the historical daily lineup**: querying a past scoring period returns each entry's `lineupSlotId` *as it was set that day* (a player's slot genuinely varies day to day — e.g. James Wood `OF,OF,OF,BE,OF...`), plus that day's actual stat split (`statSourceId==0`, matching `scoringPeriodId`). ESPN baseball scoring periods are daily and increment by 1, so a matchup week = 7 consecutive scoring-period ids ending the day before this Monday's id.

**Opportunity-cost correction (the key nuance, from the user):** raw "he produced X on the bench" overstates the miss — with a full active lineup, starting him means benching someone else. So bench leakage is reported **net** of the weakest startable bat you'd have sat. Whether he could have been slotted *without* benching anyone (open slot **or** a legal reshuffle) is a **max-bipartite-matching feasibility** check (`_full_match`, augmenting-path) over the day's active hitters + the candidate, against `lineupSlotCounts`-expanded hitter-slot instances, using each player's `eligibleSlots`. If feasible → displaced = nobody. Else → displaced = the min-value (`TB+BB+SB`) active hitter sharing an eligible slot. Net uses only his *notable* days (HR/SB/2+R/2+RBI) — a quiet 0-4 bench day has nothing to recover, and netting it would cherry-pick negatively. **OPS is never netted** — it's a rate, and benching an 0-4 line actually *helps* OPS, so "recovering" it is ambiguous.

**Pitcher side (user's example 2):** an active-slot start of 5+ ER (or 4+ ER in <3 IP) already counted toward ERA/WHIP; cross-referenced against `recent_activity(size=150)` drops to flag "imploded then dropped" — the damage is banked, the tactical lesson is sit-don't-start next time. (Once a player is off the roster his future stats don't attribute to you, so there's nothing to recover there — only the pre-drop damage is visible.)

**Split across two surfaces:** `mode="prev"` (completed Mon–Sun) feeds the Monday recap's fuller **Lineup Efficiency** band (post-mortem); `mode="current"` (Mon→yesterday, today excluded as incomplete) feeds the daily digest's compact **Lineup Watch** callout — deliberately mid-week, because that's when a benched masher is *still fixable* for the remaining days, unlike the Monday view. Both stored in the snapshot (`lineup_efficiency`, `lineup_efficiency_current`) so the render scripts stay snapshot-only; the live daily fetch lives in `fetch_data.get_lineup_efficiency`. Standalone console version with the opponent + head-to-head comparison: `bench_leakage.py` (the user cared less about the opponent view, so only my-team surfaces in the digest/recap). First real run caught Riley Greene's 4 HR / 9 RBI net stranded on the bench (started cold Moniak over him) and Dustin May imploding (0.2 IP, 5 ER) then getting cut same day.

### Full-matchup span + idle check (2026-07-10)

Two later changes. **(1) Both modes now span the full matchup period, not the current calendar week.** The original code assumed "matchup week = 7 consecutive scoring-period ids ending the day before this Monday" — which silently breaks for the 14-day All-Star/playoff matchup. `matchup_periods` can't fix it because ESPN encodes the break as a single "week" (`len(mps['15'])==1`) despite being 14 days; a length-based guess (my first attempt: cumulative `1+Σ7·len(mps[k])`) was misaligned (`cand=99≠this_mon_sp=104`) and would have failed the exact case it targeted. The robust source is a matchup entry's **`home.pointsByScoringPeriod` keys — they ARE the exact days that count** (verified live: matchup 14 → SPs 97–103, matchup 15 → 104–108). One `mMatchupScore` fetch builds `sp_span[matchupPeriodId]=(min,max)`; current uses `sp_span[cur_week][0]→yesterday`, prev uses `sp_span[cur_week-1]`. Guarded → falls back to the old this-Monday±7 window on fetch failure; a normal 7-day week is byte-identical (`min_sp==this-Monday`). **(2) Idle "wasting space"** — a third finding, an active-slot hitter not accumulating stats (0 AB), but **counting only games his MLB team actually played** (schedule-gated: a ranged MLB StatsAPI call → `played_days={(mlb_team_id,date)}` of Final games; ESPN `proTeamId`→MLB id via static `_ESPN_PROID_TO_MLBID`; fetch-fail → count all, ungated). Fires only on a **pattern** (user's spec): 3+ idle games in a row OR an AB in <50% of active games (min 4) — an occasional day off stays silent.

## Matchup W-L: the ratio-stat IP floor (the 10-2 vs 11-1 bug)

`get_all_matchups` / `get_all_prev_matchups` read ESPN's own `box.home_stats[cat]["result"]` (`"WIN"`/`"LOSS"`/`"TIE"`) rather than comparing raw values. Why: ESPN applies a ratio-stat innings-pitched minimum (~25 IP) before ERA/WHIP count. A team with the *better* WHIP that is still under the IP floor **loses** that category. A naive `h_val < a_val` comparison reported 10-2 when ESPN showed 11-1 — my WHIP 1.55 "lost" to the opponent's 1.37 on raw value, but the opponent had only ~22.7 IP (68 OUTS) < 25, so ESPN scored it a WIN for me. The box-score stat dict exposes `OUTS` (IP×3) and a per-cat `result`; discovered via live diagnostic. The 25-IP-min insight came from the user. Raw-value comparison remains only as a fallback when ESPN supplies no `result` (old snapshots).

Intended consequence: the current result (honors the live IP floor) can legitimately differ from the projected result (Category Pulse / `classify_categories` compare raw projected values, since both teams usually clear the IP min by Sunday). So a category you're "winning" only because the opponent is under the IP min will correctly show a ▼ flip arrow projecting the loss once they qualify. Don't make the projection honor the floor — the divergence is informative.

## SP blowup-risk flag: why skill-only (2026-07-10)

Motivation: the user rostered Trevor McDonald off good QS-reporting, he imploded (2.1 IP, 8 ER) and was dropped a day later (ER/WHIP already counted). He wanted to avoid low-floor streamers — chose **flag + steer**.

**Why skill-only, no realized blowup RATE.** A pitcher's own past blowup frequency is near-pure binomial noise at ~20 starts (p≈0.17 → SD≈8.5%) — folding it in HURT decile lift in the walk-forward backtest (1.34x vs 1.38x). So `blowup_risk` uses four league-anchored skill drivers (WHIP, K-escape, `_effective_era` = ERA regressed toward xERA, hard-hit%) with recent cold form as an **additive-only escalator**. Additive-only because a first symmetric-blend attempt wrongly de-risked bad-skill/hot-week arms (e.g. Merrill Kelly 78→59): a good week doesn't cure a shaky arm. `_effective_era` was chosen over pure xERA because a genuinely high ERA means runs actually scored (lower floor), and over raw ERA because tiny samples need the pull.

**Validation is soft-but-real, and that's the ceiling.** Backtest BLOWUP-RISK panel: n=1324, base 16.2%, AUC 0.522, top decile 1.26x, bottom decile 0.93x (safe arms genuinely safer), flagged 1.19x. Blowups are variance-dominated, so AUC tops out ~0.52–0.53 — 6 ERA-treatment variants probed offline all moved AUC <0.01. Don't chase a better per-start classifier; the only real lever left is a coarser target (per-pitcher season blowup count or a multi-start floor).
