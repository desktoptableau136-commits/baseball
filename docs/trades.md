**`trade_lab.py`** reads the same snapshot and builds an **interactive, browser-only "Trade Lab"** — `previews/tradelab_{team_slug}.html`, a three-column `My Team | Trade | Trade Partner` builder: pick any two teams, see each roster grouped by role (Hitters / SP / RP), click players to assemble a deal, watch it graded live (value tilt, verdict, categories/positions gained-lost, timing, partner-fit). **The ONLY tool in the repo that uses JavaScript** — so it is **NOT emailable** (no `--email` flag). Like `dashboard.py` it **imports `send_digest` and reuses its scoring/trade functions verbatim**. A header **freshness badge** (`_freshness`, from the snapshot's `refreshed_at`) shows the refresh time in ET with a green/yellow/red dot (today / 1-day / 2+-day stale → run `--refresh`). **The digest deep-links here:** every Pending Trades / Trade Radar card carries a "Build in Trade Lab" button (`send_digest._tradelab_button`, `TRADE_LAB_URL` constant = the hosted pocket URL below) pointing at `#partner=&give=&get=`, which `preloadFromHash()` (below) parses on load.

The page's `checkBuild()` polls `build.json?t=…` every 15s (≤6 min) and reloads when `built_at != DATA.builtAt`. **Pocket UI (additive `@media (max-width:640px)`, desktop untouched):** single column, big tap targets, sticky `.sidehead`, a fixed-bottom **`#dealbar`** (`display:none` on desktop) filled by `setDealBar()` (Give N · Get N · net · verdict), tap (`jumpToDeal`) scrolls to `#mid`. **Opens fully folded on the HOSTED build (any device):** `pocketCollapseDefaults()` (in the bootstrap IIFE, run BEFORE the first render) pre-seeds `collapsed`/`collapsedPos` for both sides (every role section + hitter position sub-group) and closes the `#fitboard` `<details>` — so the hosted page shows a short list of expandable headers, not a wall of players, on phone OR desktop. **Gated on `DATA.refreshUrl`** (baked ONLY into the `--refresh-url` pocket/Pages build), NOT the viewport — so local `python trade_lab.py` desktop dev runs stay fully expanded (no `refreshUrl` → the check returns early). Taps still toggle live; a preloaded deal still grades in the center even with its rows folded.
- **Server side (`build_data(snap, my_team)`):** calls the SAME shared prelude helpers as `build_email`/`dashboard.build_context` (`sd.prepare_scoring` + `sd.build_recent_indexes` — so `hit_pctile`/`pit_pctile`/`_tval` match the digest exactly). For **every** team it enriches each YEAR roster row via **`sd._enrich_trade_player`** (attaches `_tscore`/`_tcats`/`_tval`/`_tsell`/`_tbuy`/`_tgroups`) and `_serialize`s a JSON-safe record per player: name, MLB logo, role (`hit`/`sp`/`rp` via `_is_sp`), stat line, score, pre-rendered badges HTML (`hitter_badges` / `blowup_badge`+`pitcher_regression_badge` / `pitcher_regression_badge`), pre-rendered score-breakdown prose (`_hitter_score_breakdown` / `_pitcher_score_breakdown`), `tval`/`tcats`/`tgroups`/`sell`/`buy`. Per-team meta = `needs`/`surplus` cats (bottom/top third of `team_category_ranks`) + thin-hitter `need_pos` + `surplus_pos`. All → one `json.dumps` blob (`</` escaped for `</script>` safety).
- **Client side (embedded JS):** renders both rosters from the blob (score pills color-coded like `_mini_badge`; clicking a pill toggles its breakdown; clicking a row toggles the player into that side's package). The center panel recomputes live: base `netVal = Σtval(get) − Σtval(give)`, plus **demand-side `netMe`/`netThem`** (each side re-values by its own needs via `needMult`/`sumEff`, JS mirrors of `_need_mult`), `timing` = sell-out + buy-in − buy-out − sell-in, category gain/lose chips (green = fills a LEFT-team need, red = loses a non-surplus strength), positional upgrades. **CALM two-rail progressive-disclosure layout (`#cols` = `3fr 4fr 3fr`, middle wider):** the resting `#mid` = verdict pill · a give/get **ledger** where each `ledgerItem` renders the player's `tval` + an **arb-marker echo** (amber ▸ / green ◂, see roster layout) + a per-rail **subtotal** (`#giveSub`/`#getSub`), the name/value **tappable (`openVs`) to a per-player value story** — `valueStory(p)`: base drivers (`tcats`/`tgroups`) then the SAME player re-priced `To you` / `To <partner>` via `needMult`, with ± reasons from **`multParts`** (a faithful client mirror of `_need_mult`'s branches, so the story can't drift from the value math) · a **fairness bar** (`#fairbar`: a give↔green-get split sized by BASE value + a one-line tilt caption) · the `dealSummary(...)` **takeaway** (`#dealsum`) · then `#reads` = ONLY the two acceptance lines. **Everything numeric moves behind a collapsed `<details id="vdetail">` ("Value detail")**: the **three-row Base / My value / Their value give-get-net matrix** (`_vrow`) + the category gain/lose chip lines. **Column convention: every row's Net = Get − Give** (matching the "Give"/"Get" headers literally). The "Their value" row therefore passes `_vrow('Their value', thGet, thGive, netThem, ...)` — `thGet`/`thGive` swapped relative to their variable names (which are MY give/get value re-priced by the partner) so the DISPLAYED Give/Get columns read as *their* give/get and `Get − Give` still lands on `netThem` (`= thGive − thGet`, "+ = they win"). Passing `thGive, thGet` in that order (the naive reading) silently breaks the Get−Give pattern the two rows above establish — don't revert it. `dealSummary(netVal, netMe, netThem, addressesNeed)` = a base clause off `netVal` (roster-blind raw value) + a tail off the two demand-side nets (e.g. "A slight overpay on paper, but it fills both sides' needs — a win-win"). The **verdict** mirrors `_pending_verdict` keyed on `netMe` (`trap = timing<0` → ACCEPT/COUNTER/DECLINE), downgrading an ACCEPT to COUNTER on a `dealStarSurrender` OR a **my-side depth short** (`leavesShort(myMeta.pos_count, giveArr, getArr)`); **"Would they do it?"** mirrors `_trade_tilt(net_them)` — AGGRESSIVE (`netThem ≥ -realisticMax`) + graduated `dealStarReach`, but **hard-balks first when `leavesShort(partnerMeta.pos_count, getArr, giveArr)`** (they won't gut a position) and applies a read-only `netThemRead` penalty (`DATA.tune.thinPosPenalty` × `thinPos` single-slot count) so a thin loss flips the read (`pfTier` no/maybe/yes/na). **OVERALL-DEAL check (the pill can't contradict "Would they do it?" below it):** an otherwise-ACCEPT is downgraded to COUNTER whenever `pfTier !== 'yes' && pfTier !== 'na'` — i.e. a deal that's good for ME but that the rival would balk at (star reach, a depth gut, or a bad-for-them net) isn't landable as ACCEPT no matter how good it looks on my side; `why` becomes "good for you, but " + the partner-side reason. **"Would you do it?"** uses `dealStarSurrender`, balks on a my-side short, AND now checks `trap` too (a need-filling deal with bad timing reads `maybe`, not `yes`) — the same trap gate the pill above already uses, so this line can no longer disagree with it. `leavesShort`/`thinPos` are the JS mirrors of `_leaves_position_short` (body-count depth floor, hitter slots only, keyed off per-team `pos_count` + `DATA.posStarters`) — value-display rows stay pure, the penalty is read-layer only. **Both reads render with a sentiment marker** (`accLine`/`ACC`: 👍 green likes / 🤔 yellow stretch / 👎 red balks) + short reason. All tuning (`DATA.tune`: star curve + realistic band + need-mult constants + `thinPosPenalty`) is baked from the send_digest globals so the Lab can't drift. LEFT defaults to my team (or `--team`); either dropdown can pick any of the 12 teams.
- **Roster layout (`renderRoster` + `groupHitters`/`playerRowHtml`/`gaugeHtml`/`toggleSection`):** three role sections (Hitters / SP / RP). **Hitters are grouped under EVERY eligible position** (`groupHitters` buckets a bat into each `POS_GROUPS` slot in `posTokens`, so a 2B/SS/OF bat appears under all three — a per-slot "how strong here?" scan; SP/RP stay flat). A multi-eligible bat therefore has multiple DOM rows — `playerRowHtml` gives each a `gkey`-unique id but tags it `data-pid`, and **`toggle(side,id)` syncs `.sel` across EVERY copy** by `data-pid` (selection registers once in `picked`). A hitter with no recognized slot lands in a **UTIL** bucket (nobody dropped). **Both role sections AND hitter position sub-groups are collapsible** — the `.rolehdr` calls `toggleSection(side,role)` (fold state in the `collapsed` global, side→role→bool) and each `.possubhdr` calls `togglePos(side,pos)` (the `collapsedPos` global, side→pos→bool); both persist across re-renders (caret ▼/▶, `.secbody`/`.possecbody` `display:none`). **Rank tag (`rankText`) — plain COLORED TEXT, NOT a pill** (a pill read like a player's score badge → confusing): each header shows a right-aligned `#N` = the RENDERED team's **league rank** (1 = best), so a FOLDED section says "worth digging / trading for?" at a glance. Three sources: **hitter position sub-headers** → `teamsMeta[tk].pos_rank[pos]` (per-slot); **SP/RP role headers** → `pos_rank['SP']`/`['RP']` (crew rank); **Hitters role header** → `teamsMeta[tk].hit_rank`, the team's **AGGREGATE lineup-hitting rank** (so a folded Hitters section says "Xth-best hitters in the league"). `pos_rank` is built server-side in `build_data` from `sd.positional_breakdown` (C/1B/2B/3B/SS/OF/SP/RP); `hit_rank` ranks teams by Σ per-position `my_avg × POS_STARTERS` (same starter-avg values shown per position). Color: top-third GREEN / middle YELLOW / bottom-third RED; tooltip = plain-English `rankPhrase` ("2nd-best of 12" / "2nd-worst of 12", via `ordinal`). Unranked buckets (UTIL) show no tag. `openBd(bid)` takes the row's unique breakdown-element id.
- **Player rows** show a compact **position chip** after the name (hitters only — `_pos_tokens` collapses OF variants; pitchers already grouped SP/RP), and **partner-side players get a 🎯 target marker** when they'd help MY (left) team — via `targetReasons(p, meta, poss)` (strong in one of my need cats, or a hitter upgrading a thin `need_pos`). `targetReasons` is reused from BOTH perspectives; the `poss` arg flips "your"↔"their". A 🎯 target's row edge is a **soft-WHITE** hint (`.prow.target`) — **GREEN is reserved for the arb marker**, so a green edge always means "value edge", never merely "fills a need".
- **Value-asymmetry ("arb") marker — a SEPARATE signal that lives ALONGSIDE 🎯, on BOTH rosters + the center ledger.** Answers "is there a value EDGE?" (does one side price him higher than his current owner?), distinct from 🎯 ("does he fill my need?") and from the `$`/`▼` Statcast luck badges (a different axis). Computed client-side: `arbMarker(p, holderMeta, otherMeta)` on the **gap** `needMult(other) − needMult(holder)`; fires when `gap ≥ DATA.tune.needCat` (glyph), doubled when `gap ≥ needCat + needSurplus` (strong). Direction points **inward toward the trade**: LEFT roster (holder = my team) → **amber ▸/▸▸ "send"** (my surplus the partner values more; `.arb.bait`, optional right-edge tint `.prow.abait`); RIGHT roster (holder = partner) → **green ◂/◂◂ "grab"** (their surplus I value more; `.arb.grab` + green edge `.prow.agrab`, which **wins over the soft-white target edge** when a player is both). No arb = a keeper. `renderRoster` passes the OTHER dropdown's meta as `otherMeta`; both dropdowns already re-render both rosters (`initSide`) so markers recompute on either team change. Glyphs are **HTML entities** (`&#9656;`/`&#9666;`) not `\uXXXX` — `_JS` is a raw string so `\u` would survive literally. `arbReasons`/`multParts` build the hover tooltip + value-story reasons. **Display/read-layer only — never touches `_tval`, `needMult`'s use in the verdict, or any score.**
- **Deal Coach (`renderCoach`, center column):** match-up context (You need / They need / Your leverage = my surplus cats ∩ their need cats) + two value-ranked **clickable** suggestion lists (**Add to get** = partner players filling my needs; **Offer them** = my players filling theirs; each click `toggle`s the player) + a live **balance nudge** off the running `net`. A **`Fair | Favor me | Fleece` strategy toggle** (`strategy` global; `TARGET_NET`/`STUD_CEIL`) reshapes the COACH ONLY (never the objective verdict): moving right protects my studs from the Offer list (value ceiling), leads with cheaper need-fillers (ascending `tval`), steers the nudge toward a bigger edge. **A COUNTER driven by overpaying NAMES the specific partner add-on to request** (`counterAddon` — mirrors `_counter_suggestion`): a spare partner piece worth `[0.7·gap, gap+0.45]`, preferring one that fills a need of mine (then buy-low, then closest to the gap), rendered as a **clickable** `.counteradd` name that `toggle`s it into GET. The center Trade column is **wider than the two team columns** (`#cols` grid `3fr 4fr 3fr`).
- **Partner Fit board ("Who should you be trading with?", `#fitboard` — a collapsible `<details open>` ABOVE `#cols`):** ranks EVERY rival by how landable a deal is, showing one concrete engine-graded deal each in plain English (no raw `tval`). **Uses the real trade engine SERVER-SIDE** (a contained departure from the Lab's "JS only sums" rule, like `dashboard.py` — see NOTES). `build_partner_fit(pitchers, hitters, roto, team_keys, ranks, n, best_recent_p, best_recent_h, hit_pctile, pit_pctile, pos_data_by_team)` (end of `build_data`, reusing per-team `pos_data` stashed in `pos_data_by_team`) runs `sd.find_trades_combined` **from EVERY team's POV** with caps temporarily raised (`sd._TRADE_MAX_CARDS`/`_TRADE_PER_TEAM_CAP` → 400/6 in a `try/finally`). Phrasing via `_fit_deal_words`; need tags via `_fit_get_tags`. Serialized as `DATA.partnerFit = {pov_key: [records...]}`; a record is ACTIONABLE (`tier`, `get`, `give`, `verdict`, `whyOffer`, `whyGet`) or a DIAGNOSIS (`tier`, `why`). **Client:** `renderFitBoard()` reads `DATA.partnerFit[selL.value]` + fills the holes summary from `teamsMeta[myTk]`; re-rendered on LEFT-team change + bootstrap. Each card's **"Build this ▶"** (`loadDeal(el)`, reading `data-partner`/`data-give`/`data-get`) `clearAll()`s, sets `selR`, selects via `pickByName`, **folds the `#fitboard` `<details>` shut** (so the builder isn't buried under the board), and `scrollIntoView`s the builder. Tier colors in `FIT_TIER`; CSS scoped under `#fitboard`/`.fb*`.

### Trade Radar (`find_trades` + `build_trade_radar`) — cross-team, FAVOR-ME
Surfaces trade ideas that **fix a rival's category need** (their reason to accept). Two lanes (see TWO LANES): a **FAIR** lane (realistic, near-even/pay-up deals a rival would accept — the ones that land a scarce C/SS upgrade) and a **FAVOR-ME** lane (good-not-great pieces, my elite bats protected, value tilted to me, buy-low/sell-high timing); `find_trades_combined` blends them, leading with fair. Perspective-driven via `my_team`, so it works under `--team` for free — **not gated** on `not override_team`. Renders in the TRANSACTIONS band (anchor id stays `band-fa`) after FA Hitters (`build_email` body_parts, section 13b); returns `""` when no candidates exist.
- **`realistic_only` — a GENERATION-TIME hard gate, not just a display label.** `find_trades(..., realistic_only=False)` / `find_trades_combined(..., realistic_only=False)`: when `True`, a candidate package is dropped (`continue`, right after `net_them` is finalized) unless `_trade_tilt(net_val, ins, outs, net_them=net_them)[1] == "realistic"` — so an "aggressive ask" idea can never win a card slot to begin with, not merely get colored yellow. `build_trade_radar` (digest) and dashboard `render_trade_radar`/`build_context` both pass `realistic_only=True` — Trade Radar is prescriptive ("go send this"), so it should never suggest a deal it already knows the rival would balk at; some days a rival may surface zero cards, which is the intended behavior, not a bug. Trade Lab's Partner Fit Board (`build_partner_fit`) deliberately leaves it `False` (default) — its BEST/REACH/NOFIT/ONEWAY/SLIM tiering wants to show "worth a shot" reach deals too, a different, informational use case from a card that reads as an actionable suggestion.
- **Category matrix:** `team_category_ranks(roto)` generalizes `category_ranks` to `{team_key: {cat: rank}}` (rank 1 = best; same `{cat}_Points` aggregation, so `_LOWER_BETTER` is baked in). A team's **need cats** = bottom-third (`rank ≥ n − third + 1`), **surplus cats** = top-third (`rank ≤ third`), `third = round(n/3)`.
- **Acceptance gate:** for rival T, `send_cats = my_surplus ∩ T_needs` (cats I help them in) and `get_cats = T_surplus ∩ my_needs`. **`send_cats` must be non-empty** (their reason to say yes) or the pair is skipped; the get side can be a category need OR a positional need.
- **PERSONAL STRATEGY OVERLAY (`_TEAM_PUNT_CATS` / `_TEAM_TARGET_POS`, keyed by canonical team name):** the rank-derived needs are strategy-blind — they don't know a manager is *punting* a category (so chasing it is noise) or *wants* a position the ranks don't flag as a deficit. Two overlays, applied **only when `find_trades` runs from that team's perspective** (`my_key` lookup; every `--team` view falls through to `{}` → default behavior, so another manager's read is never distorted): **PUNT** = cats subtracted from `my_needs` (`Guerrero Warfare: {SVHD}` — I punt saves, so the radar stops flooding RP/closer cards for a category I don't contest); **TARGET** = positions folded into `need_pos` even when NOT bottom-third (`Guerrero Warfare: {C, SS, SP}`). A target still has to clear my starter avg there (the same upgrade gate as a real `need_pos`), so it surfaces a genuine upgrade, never a phantom need. **SP is an honorary `need_pos` entry** (rank/`my_avg` from `pos_data`) so its downstream — the ranking's `need_pos[p][0]`, the `_fills_need_pos` upgrade gate — works; `_need_mult` skips the positional term for pitchers, so the SP key never distorts a pitcher's demand-side value. **SP target = knowingly paying a little value for rotation stability** (fewer streamer swaps): my pitching cats (K/QS/W) are already a surplus, so a solid SP helps where I'm deep → the value math rates it a low-value acquisition and the card honestly reads "you pay up / even value · Upgrades your SP slot". **Depth-cap caveat:** a target position where I'm already at the redundancy cap of *playable* bodies (`POS_STARTERS[P] + _POS_DEPTH_SLACK`) still gets its positional credit STRIPPED by `_non_redundant_get_pos` unless I shed a same-position body. **IL-slot bodies no longer count toward that cap** (`_team_position_counts` excludes `_on_il`), so the common case that blocked C — an *apparent* 2-catcher stack that is really 1 active + 1 injured (e.g. Alvarez active + Will Smith on the 60-day IL) — now surfaces a C upgrade normally (verified: Drake Baldwin lands as a "Upgrades your C slot" get). The only residual block is a stack of **two genuinely-healthy** bodies at a target position: my bats there aren't in the give-pool (that position isn't a *surplus*), so a pure *add* can't form; a true upgrade-*swap* (give my weakest body at the target + a piece → get a better one) would need the give-pool to also offer that body (not yet built).
- **CROSS-ROLE TRADE VALUE (`_trade_value` → `_tval`) — the parity currency, NOT the role-score badge** (why closers ≠ hitters at equal role-score → NOTES, "James Wood" bug). `_trade_value` = `Σ max(0, pctile(c) − 0.5) × w` over the player's role cats, `w = _TRADE_SVHD_W`(0.35) for SV+H else 1.0. Percentiles are vs the WHOLE qualified pool (hitters `hit_pctile`; pitchers `pit_pctile` = `build_cat_percentiles(all YEAR pitchers, _FA_RP_CATS)`, built in the prelude next to `hit_pctile`), so a reliever's low K/W volume ranks low and an everyday smasher outweighs a one-category closer. **The role-score badge (`_tscore`) is still DISPLAYED; parity/labels use `_tval`.**
  - **INJURY DISCOUNT (`_health_value_mult` → `_IL_TVAL_MULT`):** an IL/injured player trades for LESS but never 0 — he returns. `_enrich_trade_player` multiplies `_tval` by a severity-tiered factor keyed on ESPN's status vocabulary (`ESPN_Status` first, `FreeAgentInjuryStatus` fallback): `SIXTY_DAY_DL`/`OUT` ×0.40–0.45, `FIFTEEN/TEN_DAY_DL`/`IL` ×0.60–0.62, `DAY_TO_DAY` ×0.90, healthy ×1.0; a player in an IL lineup slot (`_on_il`) whose status is blank/stale-`ACTIVE` takes the mid-tier `_IL_TVAL_DEFAULT` (0.60). Applied in `_enrich_trade_player` **AFTER** `_trade_value`, NOT inside it, so `compute_position_scarcity`'s league starter-baseline stays on healthy value. **Symmetric across both POVs** (shared `_tval`): an injured stud a rival ships is worth less to them, and one I'd acquire off the IL is a discounted buy-low to me (e.g. Bobby Witt Jr. on the 10-day IL: raw 1.44 → 0.89, so acquiring him reads "you pay a real price" not a phantom steal). Auto-mirrored into Trade Lab (the serialized `.tval` already carries the discount — no JS change).
  - **POSITIONAL SCARCITY (`compute_position_scarcity` → `_POS_SCARCITY`, HITTERS ONLY):** the raw percentile sum is position-blind, so it scales a hitter's `_tval` by `mult[P] = baseline / starter_avg_tval[P]`, where `starter_avg_tval[P]` = mean raw `_tval` of the league's top-`POS_STARTERS[P]·n_teams` eligible bats at P, and `baseline` = the mean of those per-position averages. Weak-starter positions (e.g. **C**) → `mult > 1` (SCARCE); deep positions (e.g. **1B**) → `mult < 1`. Clamped to `_POS_SCARCITY_CLAMP`(0.75–1.50). A **multi-eligible** bat takes his **scarcest** position's multiplier. Set by `prepare_scoring` (which calls `compute_position_scarcity(hitters, hit_pctile)` right after building `hit_pctile`/`pit_pctile`) — all three trade surfaces (`build_email`, `dashboard.build_context`, `trade_lab.build_data`) go through it. **Reset to `{}` at the top of the compute fn** so the internal `_trade_value` calls return RAW value while the scale is being derived (empty global = no-op passthrough = cold-start behavior). **Pitchers are NOT scaled** — pitcher value is already punt-saves-shaped, a separate axis. Typical: C ~×1.28, 1B ~×0.78, 2B/3B/SS/OF ~×1.0.
- **Outgoing pool (GOOD-not-GREAT, my studs protected)** = my players strong in a `send_cats` cat, **at a surplus POSITION** (`pos_data` rank top-third, no hole opened), **AND `_tval ≤ _TRADE_MAX_VAL`(1.35) UNLESS `_tsell`** — elite keepers never offered, except a sell-high regression candidate. **Incoming pool** = T's players that either help a `get_cats` cat OR **`_fills_need_pos`** — a hitter at one of my THIN hitter positions (`need_pos` = `pos_data` bottom-third rank) whose `_tscore` clears my positional avg there. Positional need is hitter-only (pitching need stays category-shaped). "Strong in" reuses `player_cat_strengths`. **QS/B_SO aren't per-player stats**, so a team category-need in those finds no player — intentional.
  - **ROSTER-REDUNDANCY GUARD (`_non_redundant_get_pos` + `_team_position_counts` + `_POS_DEPTH_SLACK`=1):** an incoming hitter's positional-need credit is **stripped when the deal would leave me over-depth at that slot** — more eligible bodies than `POS_STARTERS[P] + _POS_DEPTH_SLACK` — **UNLESS I shed a body eligible there in the same deal** (a swap/upgrade still counts, `shed>0`). `_team_position_counts(hitters, team_key)` (formerly `_my_position_counts`, now team-general) counts a team's rostered YEAR bats per `_trade_pos_groups` label (multi-eligible bats count at each slot; **EXCLUDES IL-slot players** (`_on_il`) — a body on the IL isn't PLAYABLE now, so it can't fill a startable slot. Counting it made a 1-active-plus-1-IL catcher read as C-deep, wrongly suppressing C upgrades and letting "give your only active C" pass the depth floor; the IL body still returns eventually, but for near-term playability it doesn't count). Applied everywhere positional need is credited: `find_trades` (the 2-for-2 `benefits` gate AND final `gpos`), `_grade_pending_trades` (`gpos`), and Trade Lab (`posStacked` JS mirror). Suppresses the *positional* credit only; a genuine category need the same player fills still counts.
  - **BOTH-PARTIES DEPTH FLOOR (`_leaves_position_short` + `_TRADE_THIN_POS_PENALTY`=0.15) — the fix for "their only catcher for 2 OF":** a body-count floor INDEPENDENT of `_tval` (catcher category value collapses to ~0, so two productive OF out-value a lone C and the value math alone reads it "realistic"). `_leaves_position_short(team_counts, give_players, get_players)` returns the **hitter** positions where GIVING `give_players` and RECEIVING `get_players` drops a team below `POS_STARTERS[P]` bodies with no same-position body back (SP/RP excluded — pitching need is category-shaped). **HARD VETO in `find_trades`:** a package is skipped when `_leaves_position_short(t_pos_count, ins, outs)` (rival left short) OR `_leaves_position_short(my_pos_count, outs, ins)` (my side). **HONEST READ:** a surviving deal that leaves the RIVAL at exactly the floor at a single-slot position (`thin_them`) subtracts `_TRADE_THIN_POS_PENALTY` per slot from `net_them` (so `_trade_tilt` flips to "aggressive ask") and stashes `thin_note` ("leaves them without a backup at C") on the trade dict → rendered ORANGE in the digest + dashboard cards. **Pending offers (`_grade_pending_trades`) aren't vetoed** (they exist) — instead `_leaves_position_short(my_pos_count, outs, ins)` feeds `_pending_verdict(..., leaves_me_short=…)`, downgrading an otherwise-ACCEPT to COUNTER ("leaves you thin at SS — get a replacement first"). **Trade Lab mirrors it in JS** (`leavesShort`/`thinPos` next to `posStacked`; per-team `pos_count` + `DATA.posStarters` + `DATA.tune.thinPosPenalty` baked): the live verdict downgrades on a my-side short, "Would they do it?" hard-balks on a partner short (+ thin note via a read-only `netThemRead`), "Would you do it?" balks on a my-side short. Value-display rows stay pure — the penalty is read-layer only.
- **Buy-low / sell-high timing (`_tsell`/`_tbuy`):** hitters via `_regression_flag` (xBA/xSLG-vs-actual), pitchers via `pitcher_regression_flag` (ERA-vs-xERA, de-biased — see the pitcher-badge gotcha). **Directional:** selling-high on the GIVE side and buying-low on the GET side are the arbitrage; the reverse (dealing a riser, acquiring a regressor) is a TRAP, penalized in ranking and chip-warned ORANGE. Timing chip tooltips are role-neutral ("results running ahead/behind his Statcast expected").
- **TWO LANES (`find_trades(..., mode)`) — both grade on the SAME scarcity-weighted `_tval`.** `mode="favor"` (default) tilts value to me; `mode="fair"` finds realistic, near-even deals a rival would ACCEPT (favor-me by construction can't surface a fair-priced scarce-position deal). Each trade is tagged `"lane"`. **`find_trades_combined`** runs both lanes and **interleaves them leading with FAIR**, deduping identical packages and keeping the per-team (`_TRADE_PER_TEAM_CAP`=2) + total (`_TRADE_MAX_CARDS`=6) caps joint across lanes. The digest `build_trade_radar` and dashboard `build_context` both call `find_trades_combined` (NOT `find_trades` directly).
  - **FAVOR lane gate (`_favor_me`):** I don't overpay (`give − get ≤ _TRADE_GIVE_SLACK`(0.15)) and may extract up to a steal ceiling (`get − give ≤ _TRADE_MAX_EDGE`(0.45)); give-side capped at `_TRADE_MAX_VAL`(1.35) (sell-high exempt). Ranking `score = Σ my-need-depth(gcov+gpos) + 0.3·Σ their-need-depth(scov) + 5.0·net_val + 4.0·timing − 0.5·(pkg−1)`.
  - **FAIR lane gate (`_fair`):** near-even — I keep at most `_TRADE_FAIR_BAND`(0.20) edge and may PAY UP at most `_TRADE_FAIR_PAYUP`(0.35) to land a scarce need-fill. Give-side ceiling RAISED to `_TRADE_FAIR_MAX_VAL`(1.90) (deal a good chip for fair return, but still protect a franchise anchor). Ranking `score = Σ my-need-depth + 0.5·Σ their-need-depth + 4.0·timing − 4.0·max(0,−net_val) − 0.5·(pkg−1)` — their benefit UP-weighted (0.5) and paying up PENALIZED, so closest-to-even need-fills rank first. Both gates: 2-for-2 scales the value slack by ×1.5; shapes 1-for-1 + 2-for-2 (top-`_TRADE_POOL_WIDTH`=4 each side; incomers cover ≥ 2 distinct needs). `net_val` = my `_tval` edge (`get − give`), `timing` = `sell_out + buy_in − buy_out − sell_in`.
  - **Acceptance read (`_trade_tilt(net_val, ins=None, outs=None, net_them=None)` → `(value_phrase, accept_phrase, color)`):** shared by the digest card header + the dashboard tile. `value_phrase` is my-POV (*you win the value / even value / you pay up*, base `_tval` ±0.1); `accept_phrase` is the RIVAL's-POV — **realistic** (GREEN) vs **aggressive ask** (YELLOW), the rival must not *clearly lose*. When `net_them` (their demand-side net) is supplied it drives the read (`net_them ≥ -_TRADE_REALISTIC_MAX`, 0.05); otherwise a base proxy (`net_val ≤ _TRADE_REALISTIC_MAX`). **`_TRADE_REALISTIC_MAX` (0.05) is a dedicated read band, DISTINCT from `_TRADE_FAIR_BAND` (0.20)** (the latter stays the fair-LANE generation gate). `find_trades` computes `net_them` per deal; **all three callers** (`build_trade_radar`, dashboard `render_trade_radar`, Trade Lab's `build_partner_fit`) pass `net_them=t.get("net_them")`.
    - **GRADUATED star premium (`_star_premium(tval)` → `_deal_star_reach` / `_deal_star_surrender`)** — the endowment premium a manager attaches to a prized player, keyed on **trade VALUE `_tval`, NOT role score** (see NOTES, "star premium moved to `_tval`"). Convex: `max(0, min(_STAR_PREM_CAP=1.10, (tval − _STAR_TVAL_FLOOR=1.10) × _STAR_TVAL_SLOPE=1.4))` → 0.88→0.00, 1.10→0.00, 1.42→0.45, 1.52→0.59, 1.90→1.10. **Why value not score:** role score and `_tval` diverge structurally (ρ≈0.82) — a vulture-win reliever (Aaron Ashby: score 95, `_tval` 0.88) *scores* like a star but *trades* like a role player, and an elite closer scores like every other closer while trading far cheaper. Pricing the premium on the cross-role value currency fixes both and makes relievers comparable to hitters/SPs on ONE axis, so **the old `_star_role` reliever exclusion is GONE** — a mid reliever simply sits below the floor at ~0, an elite closer (Mason Miller, `_tval` 1.42 → 0.45) earns real premium.
      - **`_deal_star_reach(ins, outs, net_val)`** (rival balks): premium is **SUMMED across the package**, not max-per-side — `req = max(0, Σ_star_premium(ins._tval) − Σ_star_premium(outs._tval))` (rival surrenders `ins` = what I acquire, receives `outs` = what I give). Reach (→ **aggressive ask**) when `req > 0 and net_val > -req`. **Summing is what catches "two franchise players for one star + a role player"** — a lone high-value return can't mask that the rival ships two premium assets (the Miller+Witt-for-Ashby+McCarthy bug: max-per-side let McCarthy's high value + Ashby's inflated role score cancel the guard).
      - **`_deal_star_surrender(ins, outs, net_val)`** (MY-side mirror): `req = max(0, Σ_star_premium(outs._tval) − Σ_star_premium(ins._tval))`; fires when `req > 0 and net_val < req`. Wired into `_pending_verdict` (`star_surrender` param) — downgrades an incoming ACCEPT → **COUNTER** ("they're prying your star at par — hold out"); `_counter_suggestion` returns "".
      - **ACCEPTANCE-LAYER only, NOT value:** `_tval` stays a pure category-value currency; the premium is a separate acceptance-layer read on top of it. Star checks run on **base `net_val`** (not the demand-side nets). **Mirrored byte-for-byte into the Trade Lab JS** (`starPremium(tval)`/`dealStarReach`/`dealStarSurrender` — sum, no role filter; constants `starTvalFloor`/`starTvalSlope`/`starPremCap` baked into `DATA.tune`) driving the Lab's "Would they do it?" / "Would you do it?" reads.
    - **DEMAND-SIDE team-need value (`_need_mult` → net_me / net_them)** — rationale → NOTES. `_need_mult(row, need_cats, surplus_cats, need_pos=None, surplus_pos=None)` → a multiplier (start 1.0; `+_NEED_MULT_CAT`=0.14 per need cat covered, `+_NEED_MULT_POS`=0.20 for a hitter at a thin position, `−_NEED_MULT_SURPLUS`=0.10 when he only helps where they're deep; clamped `_NEED_MULT_CLAMP`=(0.70, 1.60)). `effective_value = _tval × _need_mult`. **Acceptance-read layer only** — NOT folded back into `_tval` or the `find_trades` ranking (no double-count). Used two ways: (a) `find_trades` computes **`net_them`** (drives the digest/dashboard tilt); (b) `_grade_pending_trades` computes **`net_me`** (my cat+positional net) and passes it to `_pending_verdict`. The displayed value phrase + counter-gap stay on BASE `_tval`. **Trade Lab shows BOTH POVs** via a three-row Base / My value / Their value block (client `needMult`/`sumEff` mirror `_need_mult`).
      - **`net_them` includes the RIVAL's positional need too, symmetric with the my-side calc (`_pos_need_surplus(pos_data, n)` — shared helper, factored out of the my-side inline block so both sides use one convention).** `find_trades` builds `t_need_pos`/`t_surplus_pos` per rival inside the team loop from `pos_data_by_team.get(team)` if a caller supplied that cache, else computes `positional_breakdown(pitchers, hitters, team, ...)` on demand — then passes them into the `their_get_val`/`their_give_val` `_need_mult` calls exactly like the my-side `my_get_val`/`my_give_val` calls already did. This closes a real divergence from the Trade Lab JS mirror (`netThem`/`partnerMeta.need_pos`), which always included the positional term — the digest was the one that had been skipping it. `find_trades(..., pos_data_by_team=None)` / `find_trades_combined(..., pos_data_by_team=None)`: `trade_lab.py`'s `build_partner_fit` passes its already-built `pos_data_by_team` (trade_lab.py — see below) to avoid a `positional_breakdown` call storm across its every-team-POV loop; the digest/dashboard callers omit it (cheap enough to compute per-rival on demand once per run).
- **Card render:** partner `fantasy_logo` + name; two columns **You give** (RED header, `send_cats` chips MUTED) / **You get** (GREEN header, `get_cats` chips ACCENT + **CYAN position chip** for a thin slot the incoming hitter upgrades). `_trade_player_line(r, hi_cats, hi_color, side, show_pos, best_recent_p, best_recent_h, hit_pctile)` renders the **CANONICAL** buy/sell chip — the same glyph-only `$` (GREEN, `_tbuy`) / `▼` (RED, `_tsell`) used everywhere else. `side` ('give'/'get') only tunes the hover tooltip. **Every player's Score pill is tap-to-expand** (both give AND get sides): because trade players are stacked `<div>`s inside table cells, it uses `_trade_score_reveal` — a div-based analog of `score_reveal` (hidden `<div class="scorebd-div">` revealed by a `div.scorebd-div:target` rule; Gmail strips `<style>`). Breakdown fn role-picked off `_tptype`. Footer "Upgrades your {cats + `{pos} slot`} ; they shore up {send} · {value}" where value from `net_val`, plus a `· sell-high`/`· buy-low` thesis tag. Constants are one-line tunables near the FA-cats constants. **"Build in Trade Lab" link** (`_tradelab_button`, right side of the card's flex header row — no extra row): a plain link (no JS) into the hosted pocket Trade Lab (`TRADE_LAB_URL`) with the deal preloaded via the same `#partner=&give=&get=` hash `trade_lab.py`'s `preloadFromHash()` already parses — `partner=t["team"]`, `give`/`get` from `PlayerName` on `t["outs"]`/`t["ins"]`. `""` when both lists are empty. Styled as a quiet ACCENT text link (no button chrome), matching the digest's other muted inline links (e.g. the `↑ TOP` band link) rather than standing out as a CTA. **Net-value hint** (`_trade_net_summary`, same header row, immediately left of the link): a single quiet line — `base +N.NN · you +N.NN · them +N.NN` — off `net_val`/`my_get_val−my_give_val`/`their_get_val−their_give_val` (all already stored on the trade dict by `find_trades`). Plain numbers only, no verdict language — Trade Radar deals are always something *I'd propose*, so an accept/counter/decline framing doesn't apply (you can't "counter" your own offer); instead a bad-for-me deal never generates in the first place: `find_trades` hard-filters out any package where `_deal_star_surrender(ins, outs, net_val)` fires (would require surrendering my best-role star at even/negative value) via `continue` right after `net_val` is computed, so Trade Radar never suggests it to begin with.

### Pending Trades evaluator (`_grade_pending_trades` + `build_pending_trades_section`) — grades REAL offers
Grades the actual pending trade **offers** involving my team (snapshot `pending_trades`), reusing the Trade Radar machinery so every number matches the digest. Because offers **expire**, the incoming-offer headline surfaces HIGH (Week at a Glance + the Briefing "Act today" list), not just in the dedicated section. Gated on `not override_team`.
- **Grade ONCE, share everywhere:** `_grade_pending_trades(pending, pitchers, hitters, roto, my_team, best_recent_p, best_recent_h, pos_data, hit_pctile, pit_pctile, today_str)` → a list of graded dicts, computed once in `build_email` right after `pos_data`. The dedicated section, the Briefing, and the Week-at-a-Glance headline all read from it (`incoming_pending = [g for g in graded_pending if g["incoming"]]`) so they can't disagree.
- **Reuse (single source):** `_trade_value(r, ptype, hit_pctile, pit_pctile)` and `_enrich_trade_player(r, ptype, best_recent_p, best_recent_h, hit_pctile, pit_pctile)` (sets `_tscore`/`_tcats`/`_tval`/`_tsell`/`_tbuy`/`_tgroups`/`_tptype`) were **promoted from `find_trades` closures to module level** so both share one implementation. `find_trades` calls `_enrich_trade_player` via a thin local `enrich`. `_hitter_fills_need_pos(r, need_pos)` is the shared positional-need test.
- **Player→row resolution:** each traded player's ESPN name is matched to its snapshot YEAR row via a `_badge_name_key` lookup (fall back 30→15→7); role decided by which pool matches. Unmatched names render a plain "(no data)" line and are excluded from the value math (never crash).
- **Grade:** `net_val = Σ_tval(get) − Σ_tval(give)` (BASE value, drives the displayed value tilt + counter-gap, ±0.1 thresholds). A **demand-side `net_me`** = incoming valued by my needs − outgoing (`_need_mult`) drives the **verdict** so it's roster-aware. Category/positional impact = incoming `_tcats ∩ my_needs` + thin hitter positions filled; timing legs `sell_out+buy_in−buy_out−sell_in`.
- **Verdict (`_pending_verdict`, INCOMING only):** keyed off `net_me` (not base `net_val`) — ACCEPT (GREEN) when I win value (or roughly even AND it fills a need) and timing isn't a net trap; COUNTER (YELLOW) when the direction is right but I pay up or the timing is a trap; DECLINE (RED) when no need is addressed and I don't gain value. Rendered as a `_verdict_pill` + one-line rationale. **A `star_surrender` flag (`_deal_star_surrender`) downgrades any otherwise-ACCEPT to COUNTER** when the offer pries my crown-jewel star at par; **a `leaves_me_short` flag (`_leaves_position_short(my_pos_count, outs, ins)`) likewise downgrades ACCEPT→COUNTER** when accepting would strip me below startable depth at a hitter slot ("leaves you thin at SS — get a replacement first"). **OUTGOING** offers (I proposed) show an "Awaiting {partner} · {value} from your side" status instead.
- **Counter suggestion (`_counter_suggestion`, on a COUNTER):** when I'm overpaying by `gap` (= `−net_val`), suggests the single best ADD-ON to request from the partner — a spareable piece (`_tval ≤ gap + _TRADE_MAX_EDGE`) that closes the value gap (`_tval ≥ 0.7·gap`) and ideally helps one of my need cats/positions (preferring buy-low). Rendered as a 💡 line in the card **and** appended to the headline. Empty when no realistic add-on exists.
- **Expiry (`_fmt_trade_expiry`):** "expires in Nd" from the ISO `expires` field, shown RED in the card header, the headline, and the Briefing.
- **High-up headline (`_pending_headline(g, brief=False)`):** one-line 🤝 headline (partner · get/give · verdict pill · expiry · counter) prepended to **Week at a Glance** (`build_week_overview(..., trade_bullets=[...])`, top of the list) and the **Briefing** "⚡ Act today" list (`render_briefing(..., pending_incoming=incoming_pending)`, first items — they expire).
- **Card render (`build_pending_trades_section(graded, best_recent_p, best_recent_h, hit_pctile, team_logos)`):** reuses `_trade_player_line` verbatim, same You give (RED) / You get (GREEN) columns as Trade Radar, at the TOP of the TRANSACTIONS band (section 10b). Returns `""` when nothing is pending. Same **"Build in Trade Lab" button** as Trade Radar (`_tradelab_button(partner, g["give_names"], g["get_names"])`) on every card, incoming or outgoing.
