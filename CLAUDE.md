Guidance for Claude Code when working in this repo. Actionable rules only — background/rationale and completed-work forensics live in `NOTES.md`.

## Commands

```bash
python send_digest.py                          # full run: refresh data (~60s) then send email
python send_digest.py --no-refresh             # use existing snapshot (fast — email-only changes)
python send_digest.py --dry-run                # save HTML to previews/, no email
python send_digest.py --dry-run --no-refresh   # instant preview, no network
python send_digest.py --with-dashboard         # full run: email digest WITH the dashboard attached (one email; used by daily CI)
python send_digest.py --with-dashboard --dry-run --no-refresh  # preview both → previews/digest_preview_*.html + dashboard_*.html
python send_digest.py --dry-run --no-refresh --team "Houck Tuah"  # another team's digest (needs all_matchups)
python fetch_data.py                           # refresh data only → data/snapshot.json
python weekly_recap.py                         # Monday recap: refresh + email full-league recap
python weekly_recap.py --dry-run --no-refresh  # instant recap preview → previews/recap_week_N.html
python bench_leakage.py                         # standalone daily-lineup audit (my team + last week's opponent → console)
python backtest_projections.py                  # walk-forward accuracy of the SP proj line (IP/ER/K) vs actual game logs → console
python backtest_projections.py --limit 30 --no-cache  # quick smoke test (small pool, no cache)
python snapshot_schema.py                       # validate data/snapshot.json against the reader contract (ERROR/WARN report; exit 1 on error)
python dashboard.py                             # single-viewport dashboard → previews/dashboard_{team}.html (uses existing snapshot, no email)
python dashboard.py --refresh                   # refresh data (~60s) first, then write the dashboard
python dashboard.py --team "Houck Tuah"         # another team's dashboard (needs all_matchups)
python dashboard.py --email                     # also email the dashboard to yourself as an attachment (open in a device browser)
python trade_lab.py                              # interactive browser-only Trade Lab → previews/tradelab_{team}.html (uses existing snapshot)
python trade_lab.py --refresh                    # refresh data (~60s) first, then write the Trade Lab
python trade_lab.py --team "Houck Tuah"          # default the LEFT (my) side to another team
pip install -r requirements.txt
```

No linter, no test suite. Verify by opening `previews/digest_preview_{team_slug}.html` or `previews/recap_week_N.html` in a browser. The dashboard (`previews/dashboard_{team_slug}.html`) is designed for a **1440×900 laptop with zero scrolling** — verify by opening maximized (or headless-screenshot at 1440×900) and confirming no scrollbars appear.

## Architecture

Three files, one intermediate artifact. The daily digest and Monday recap are independent scripts that both read the same snapshot:

**`fetch_data.py`** pulls from 5+ sources → `data/snapshot.json`:
1. FantasyPros HTML (`pd.read_html`) — pitcher/hitter stats across 4 ranges (7/15/30/season)
2. ESPN Fantasy API (`espn_api` + one raw endpoint) — rosters, FA list, roto box scores, standings, transactions, and **pending trade proposals** (`get_pending_trades` — raw `mPendingTransactions` view, see gotcha)
3. MLB Stats API — probable starters (batch hydrate) + opponent OPS + today's game schedule & national TV (`fetch_todays_games`)
4. pybaseball — Statcast contact quality, expected stats, sprint speed, recent game logs

**`send_digest.py`** reads the snapshot, computes all derived metrics, builds the email via Gmail SMTP. Two parts: **inline body = "The Briefing"** (a short skimmable plain-English overview — NOT the full digest) + an attached `digest_YYYY-MM-DD.html` (the full render, opened in a browser). All new digest features go in the attachment (`build_email`'s `body_parts`).
- **The Briefing (`render_briefing`)** — the inline email body. Time-sensitive **⚡ Act today** items first (two-start/must-start SPs via `_starts_this_week`, ⚠ low-floor starts via `_is_blowup_risk`, emerging FA closers from `save_role_watch`, injury alerts, and a **📺 Tune in** teaser naming the single highest-overlap MLB game today via the `tune_in` param — the top of `_rank_todays_games`), then a one-line **This matchup** read (current record labelled `now` + projected record from `category_classification`, with ▲▼◆ + light prose naming leading/coin-flip cats — the `lead_in` prose is **projection-framed** ("The projection leans your way…"), never "you're ahead here", so it can't read as a claim about the current record beside it when the two diverge, e.g. a 6-6-0 current tie under a projected 7-5-0 win) and a **Season** line (standing of N + luck read). Alerts-first with a touch of analyst prose. **Inline-styled only** (Gmail strips `<style>`). Built at the END of `build_email` (all locals in scope) inside a try/except; **`build_email` now returns `(full_html, briefing_html)`** — if the briefing fails, `briefing_html=""` and `main` falls back to the full digest as the body. `send_email(body_html, attachment_html, subject, filename, extra_attachments)` takes body and attachment separately. `--dry-run` also writes `previews/briefing_preview_{slug}.html`.
- **Subject:** `⚾ {team} {Digest|Lookahead}[ + Dashboard] — {date}` — "+ Dashboard" appended under `--with-dashboard`.

**`dashboard.py`** reads the same snapshot and builds a **single-viewport, zero-scroll "command dashboard"** — `previews/dashboard_{team_slug}.html`, sized for a **1440×900 laptop with NO scrolling**, even glance-able coverage of every digest topic. Unlike `weekly_recap.py`, it **imports `send_digest` and reuses its scoring/projection functions verbatim** (every number matches the digest). `build_context(snap, my_team)` mirrors `build_email`'s derived-value prelude: it runs the same 4-call scoring setup (`compute_ab_benchmarks` → `compute_pitcher_benchmarks` → `compute_score_calibration` → `compute_league_averages`, **ORDER MATTERS** — the module globals they set drive scoring), then rebuilds `best_recent_p/h`, FA lists, `weekly_avgs/std`, `pit_proj`, `classify_categories`, `positional_breakdown`, `opponent_week_intel`, `save_role_watch`, `_roster_suggestion`, `make_sparkline`, etc.
- **Desktop layout:** top KPI bar (8 chips) + fixed 3-col CSS grid (`#wrap{position:fixed;inset:0;display:flex}` — **load-bearing, see NOTES**; `#grid{display:grid;grid-template-columns:1fr 1fr 1fr;grid-template-rows:minmax(0,1fr)}`, `.col{height:100%}`) + a bottom **legend strip** (`render_legend`/`#legend`). Tiles `overflow:hidden` and clip to the top-N rows that fit (never scroll) — **the ONE exception is the Lineup Watch pane inside the Weakest Spots tile**, which gets its own `overflow-y:auto` region (per user preference — see Column assignment). Per-tile `flex` weights give content-heavy tiles more height — don't equalize blindly. **Verify with a headless screenshot at 1440×768 (not just 900)** — browser chrome shaves usable height (see NOTES).
- **Legend strip (`render_legend` desktop bar / `render_legend_panel` tablet tile):** the shared key items come from **`_legend_items()`** (single source, so the two renderings can't drift). On **desktop** it's a slim one-line **`#legend`** key pinned to the foot of `#wrap` (a `flex:0 0 auto` sibling AFTER both `#grid` and `#gridt`). On **tablet/phone** the `#legend` bar is hidden (media query `#legend{display:none}` ≤1100px) and the same key renders instead as a normal `_tile("Key", …)` **placed last in the right tablet column `colt_r`** (after the Season tile, which now rides 2nd-to-last there) — a panel in the flow, NOT a full-width foot strip (per user preference). Because the panel lives inside `#gridt` (desktop-hidden), it only shows on tablet/phone. Defines every badge/marker: score pill, ▲▼◆ proj W/L/T, ⚡ toss-up, win %, ×2 two-start, QS, 5K+, ⚠, `$`/`▼`, PWR, SB, 🔥/❄. `_legend_chip(glyph, color)` reproduces the tinted `_hit_badge` look; keep it in sync with the real badge colors (QS cyan, two-start blue) when they change.
- **Column assignment:** col 1 = **Today's Games** (`render_tv_games` — compact TV panel at the TOP of col1, above Category Pulse per user preference; favorite team ★-pinned first + the next highest-overlap game, 2 games max; reuses `sd._rank_todays_games(pin_favorite=True)` + `sd._fmt_game_time_et` + the same role-aware badges as the digest — `build_context` exposes `todays_games`, the tile builds its own YEAR→30→15→7 badge-row lookups; `flex=1.05`, `''` when no games) + Category Pulse (`flex=1.6`; 4×3 compact 12-cat grid; my/opp values + `proj X / Y` sub-line + win%/⚡ + ▲▼◆ via `_project`/`_cat_win_prob`) + Weakest Spots/Lineup Watch (`flex=1.0`; fixed Weakest-Spots holes rows on top + a scrollable `overflow-y:auto` Lineup Watch pane below — `build_bench_watch`, empty on `--team`; deliberately short since the watch pane scrolls, so col1's height goes to Today's Games + Pulse); col 2 = My Pitching · Hitting Hot/Cold · **Trade Radar / Trades** (`render_trade_radar` — abbreviated: top **2** swaps, DISTINCT partners preferred, each rendered as a **3-column mini-card** (col 1 = partner + `you win`/`even`/`you pay up` value tilt + a **`realistic`/`aggressive ask` acceptance sub-line** (`sd._trade_tilt`, rival's-POV) + sell-high/buy-low thesis; col 2 = **Give**; col 3 = **Get**) so the card reads horizontally and stays short (both swaps fit un-clipped); players one-per-line with a `_mini_badge` **score pill** (hover `title` = short role-aware `_score_tip` prose via send_digest's narrative helpers — NOT tappable, per user preference) + canonical `$`/`▼` + `(C)` position chips; **replaced the Opponent This Matchup tile per user preference** — opponent scouting still lives in the digest; the dashboard's `render_opponent`/`opp_intel` were removed as dead code. **A REAL incoming trade offer LEADS this tile** (per user preference — a concrete decision outranks speculative ideas): `build_context` runs `sd._grade_pending_trades(... )` → `pending_incoming`; when non-empty the tile retitles to **"Trades"**, renders each incoming offer as a top mini-card (📥 OFFER TO REVIEW · partner · `sd._verdict_pill` ACCEPT/COUNTER/DECLINE · red `expiry_str` · Give/Get with score pills · a 💡 `Counter:` line on a COUNTER), then **ONE speculative radar idea below** (`max_radar = 1`) — chosen from a team I **don't** already have an incoming offer with (`offer_partners` = `{g["partner"]}`, filtered out of the radar loop so the idea is a FRESH partner to shop, not a duplicate of a live negotiation; the loop checks the cap BEFORE appending so the count is exact). Whenever an offer leads the tile the **body scrolls** (`overflow-y:auto` — the only dashboard scroll region besides Lineup Watch, per user preference) so every incoming offer plus the one idea is reachable, none clipped. Empty on `--team`/no offer, where it's plain "Trade Radar" with 2 ideas); col 3 = Recommended Moves + Save-Role · FA Radar (top **2** per role) · Season (standings/roto/`make_sparkline`/weekly-finishes strip). **Recommended Moves** (`render_moves`) shows a `_mini_badge` score pill after every add AND drop player name — `build_context` passes `sd._roster_suggestion(..., pill_fn=_mini_badge)`; the `pill_fn` callback (score→HTML) is off by default so the **digest bullets stay pill-free**. Clean 3-tiles-per-column (col 1 = Today's Games + Pulse + Weakest) — verified zero-scroll at 1440×768 via headless Chrome screenshot. `build_context` also calls `sd.compute_xera_offset` (de-bias the pitcher buy/sell flag) and computes `pit_pctile` + `trades` (`sd.find_trades_combined` — the fair+favor blend, so the tile shows realistic asks too).
- **My Pitching badges:** each upcoming start shows the projected `IP·ER·K` line, a blue `×2` two-start marker BEFORE the matchup date, and cyan **QS** / yellow **5K+** / ⚠ / **`$`/`▼`** (pitcher buy-low/sell-high, `_reg_chip8` — an 8px chip matching the others, off `sd.pitcher_regression_flag`) badges AFTER it — QS/5K+ driven off the SAME `sd._proj_line_vals(r)` as the displayed line so they never contradict it. FA Radar Starters + Relievers render `sd.pitcher_regression_badge` (Starters also `sd.blowup_badge`); Weakest Spots renders it on pitcher-position rows (else `sd.hitter_badges`).
- **Responsive (below 1100px):** media queries relax the fixed pane WITHOUT touching the desktop layout — the pinning inline styles are overridden via **classes + `!important`** (`.tile`, `.tile-body`, `.pulse-grid`, `.topbar`, `.topbar-chips`, `.chip`; inline otherwise wins). **Tablet/phone use a SEPARATE container, not a reflow.** `build_dashboard` renders each of the tiles ONCE and drops the identical markup into two containers: `#grid` (desktop 3-col) + `#gridt` (tablet: two `.colt` independent-packing flex columns), one `display:none` per breakpoint (**why two grids, not `order`/multicol — see NOTES**). Tablet `.colt` split (height-balanced): left = Today's Games · Pulse · Moves · FA; right = Pitching · Hitting · Weakest · Trade Radar · **Season** · Key (Season rides 2nd-to-last, just above the Key panel — per user preference). ≤1100px: `#wrap`→`position:static`+`overflow-y:auto`, `#grid{display:none}`, `#gridt{display:flex;align-items:flex-start}`, tiles content-sized, pulse 3-wide. ≤700px (phone): `#gridt{flex-direction:column}` stacks the two `.colt`, larger font, pulse 2-wide.
- **When adding a tile:** render it once in `build_dashboard`, place it in BOTH `#grid` (a `.col`) and `#gridt` (a `.colt`), and re-check the two `.colt` heights stay balanced.
- **MLB team logos** (`sd.team_logo(r.get("Team"), size)`, `""` on unknown abbrev) precede the player name in every player-listing tile: My Pitching (14px), Hitting Hot/Cold (14px), Weakest Spots (13px), FA Radar (13px), Trade Radar (12px, both give/get), Today's Games (13px, both teams in each game). Recommended Moves + Save-Role skipped (prose/plain-name, no player row).
- **`--email`** sends the dashboard as a STANDALONE ATTACHMENT via `send_dashboard_email` (reuses send_digest Gmail creds); inline body is a pointer only since Gmail strips the `<style>` holding the layout. **Tab title** `Dashboard — {my_team}` (type-first; digest mirrors as `Daily Digest — {my_team}`). No glossary. **The daily dashboard now ships as an ATTACHMENT ON THE DIGEST EMAIL** (`send_digest.py --with-dashboard`, one fetch/one email — see Automation); `daily-dashboard.yml`'s schedule is retired (`workflow_dispatch`-only). Still runnable locally standalone (`python dashboard.py --email`).

**`trade_lab.py`** reads the same snapshot and builds an **interactive, browser-only "Trade Lab"** — `previews/tradelab_{team_slug}.html`, a three-column `My Team | Trade | Trade Partner` builder where you pick any two teams, see each roster grouped by role (Hitters / Starting Pitchers / Relief Pitchers), click players to assemble a hypothetical deal, and watch it graded live (value tilt, verdict, categories/positions gained-lost, timing, partner-fit). **The ONLY tool in the repo that uses JavaScript** — live selection needs it, which also means it is **NOT emailable** (mail clients strip JS; no `--email` flag). Like `dashboard.py` it **imports `send_digest` and reuses its scoring/trade functions verbatim**, so every number matches the digest. A header **freshness badge** (`_freshness`, computed at build time from the snapshot's `refreshed_at`) shows the refresh time in ET with a **green/yellow/red dot** (today / 1-day / 2+-day stale → run `--refresh`).
- **Server side (`build_data(snap, my_team)`):** runs the SAME 5-call scoring prelude as `dashboard.build_context` (`compute_ab_benchmarks` → `compute_pitcher_benchmarks` → `compute_score_calibration` → `compute_league_averages` → `compute_xera_offset`, **ORDER MATTERS**), rebuilds `best_recent_p/h` + the two percentile pools (`hit_pctile`/`pit_pctile`, same qualified YEAR pools as the digest so `_tval` matches). Then for **every** team it enriches each YEAR roster row via **`sd._enrich_trade_player`** (attaches `_tscore`/`_tcats`/`_tval`/`_tsell`/`_tbuy`/`_tgroups`) and `_serialize`s a JSON-safe record per player: name, MLB logo, role (`hit`/`sp`/`rp` via `_is_sp`), compact stat line, score, **pre-rendered badges HTML** (`hitter_badges` / `blowup_badge`+`pitcher_regression_badge` / `pitcher_regression_badge`), **pre-rendered score-breakdown prose** (`_hitter_score_breakdown` / `_pitcher_score_breakdown`), `tval`/`tcats`/`tgroups`/`sell`/`buy`. Per-team meta = `needs`/`surplus` cats (bottom/top third of `team_category_ranks`) + thin-hitter `need_pos` (`positional_breakdown`, `{pos: my_avg_score}`). All → one `json.dumps` blob (`</` escaped for `</script>` safety).
- **Client side (embedded JS):** renders both rosters from the blob (score pills color-coded like `_mini_badge`; clicking a pill toggles its breakdown div; clicking a row toggles the player into that side's package). The center panel recomputes live: `netVal = Σtval(get) − Σtval(give)`, `timing` = sell-out + buy-in − buy-out − sell-in, category gain/lose chips (green = fills a LEFT-team need, red = loses a non-surplus strength), positional upgrades, and the **verdict via the exact `_pending_verdict` thresholds** (±0.1 value slack, `trap = timing<0` → ACCEPT/COUNTER/DECLINE), plus a **partner-fit** read (does what I give hit the RIGHT team's needs → "realistic" vs "tough sell"). All math sums pre-computed per-player numbers, so it can't disagree with the digest. LEFT defaults to my team (or `--team`); either dropdown can pick any of the 12 teams. Verify by opening in a browser (or headless-screenshot) and clicking a lopsided deal — an elite bat for a lesser piece should read DECLINE / "you pay up".
- **Player rows** show a compact **position chip** after the name (hitters only — `_pos_tokens` collapses OF variants; pitchers are already grouped SP/RP), and **partner-side players get a 🎯 target marker + green edge** when they'd help MY (left) team — via `targetReasons(p, meta, poss)` (strong in one of my need cats, or a hitter upgrading one of my thin `need_pos` positions). `targetReasons` is reused from BOTH perspectives (partner players filling my needs; my players filling theirs) — the `poss` arg flips the wording "your"↔"their".
- **Deal Coach (`renderCoach`, center column):** match-up context (You need / They need / Your leverage = my surplus cats ∩ their need cats) + two value-ranked **clickable** suggestion lists (**Add to get** = partner players filling my needs; **Offer them** = my players filling theirs; each click `toggle`s the player into the deal) + a live **balance nudge** off the running `net`. A **`Fair | Favor me | Fleece` strategy toggle** (`strategy` global; `TARGET_NET`/`STUD_CEIL`) reshapes the COACH ONLY (never the objective verdict): moving right protects my studs from the Offer list (value ceiling), leads with cheaper need-fillers (ascending `tval`), and steers the nudge toward a bigger value edge.

**`weekly_recap.py`** reads the same snapshot on Mondays, builds a full-league recap HTML email: Week N Highlights · My Matchup (full 12-cat table) · Lineup Efficiency (`build_lineup_efficiency`) · League Scoreboard (all 6 matchups) · Weekly Roto Rankings ("MATCHUP PERFORMANCE" band, this matchup only) · Top Performers (rostered + hot FAs, hitters/pitchers side-by-side via `_two_col`) · Standings & Luck · Season Trajectory · **Season Roto Rankings** (`build_season_roto_rankings` — heat-mapped 12-cat grid aggregated over EVERY matchup: teams **ranked** by cumulative `Roto_Score`, **displayed** value = the true season-to-date stat from ESPN's cumulative `mTeam` view (`season_cat_totals`), NOT a sum/average of weekly values. Ranking and value are independent by design — for rate cats {OPS,ERA,WHIP} the season stat can't be recovered by averaging weekly values; falls back to the summed/averaged weekly value only when a season total is missing). Does NOT import from `send_digest.py` — copies the ~100 lines it needs. Output: `previews/recap_week_N.html`. GitHub Actions: `.github/workflows/weekly-recap.yml` (Monday 15:30 UTC).

**`build_commissioner_story`** (`weekly_recap.py`) — Week N Highlights at the top of the recap. Two-column: commissioner prose (left, 60%) + stat sidebar (right, 40%). Finds (a) weekly roto winner, (b) hitter of week (rostered, best OPS, min 10 AB from `prev_week_hitting`), (c) pitcher of week (rostered, best ERA, min 8 IP from `prev_week_pitching`), (d) best available FA hitter. `prev_week_hitting`/`prev_week_pitching` are snapshot fields over the exact previous matchup Mon–Sun window — distinct from rolling `recent_hitting`(7d)/`recent_pitching`(15d) (fallback only). **`build_top_performers` also runs on `prev_week_*`** so its timeline matches the recap (subtitle shows the actual week dates; pitcher table shows **K** not G). **Both run names through `_fix_mojibake` before the season-roster lookup** — `prev_week_*` names arrive mojibaked (e.g. `Eury PÃ©rez`), so without the fix the `_name_key` fallback misses the roster row and the player wrongly shows as a free agent. Prose uses named historical benchmarks (Bonds 1.422 OPS, Gibson 1.12 ERA, deGrom 1.70 ERA, Ryan 383 K, etc.). Sidebar cards: fantasy logo + MLB team logo (`_mlb_logo`, ESPN CDN) + slash line / K / QS / categories led. Roto weekly win count = weekly top-roto-score count, NOT H2H W-L. MLB abbrev from season rows' `Team` field; `_MLB_ABBREV_ESPN` maps Baseball Reference outliers (TBR→tb, KCR→kc, SFG→sf, WSN→wsh, etc.).

**`build_league_scoreboard`** (`weekly_recap.py`) — all 6 matchups, one card each, 12-cat grid. Each team's identity (logo + name + W–L–T) lives in its **row label** (first column), NOT a top banner — the row IS the team, columns are categories. Every card is a **separate `<table>`**, so to keep the label→category axis lined up card-to-card the tables use `table-layout:fixed` + a `<colgroup>` (150px label col + 12 equal cat cols); without this each table sizes its first column to its own team-name length and the axis drifts. Long names ellipsis-clip. Category headers are **neutral** (`MUTED`), not green/red — win/loss coloring only makes sense from one team's POV, and this is an even-handed full-league view. The **outlined value** (`outline:1px solid`) marks each category's winner instead. `_row` renders both teams (top = `my_team`/`my_val`, bottom = `opp_team`/`opp_val`).

**`get_lineup_efficiency(league, my_team_name, mode)`** (`fetch_data.py`) — reconstructs MY team's **daily** lineup to quantify start/sit opportunity cost (mechanism + rationale in NOTES). **Both modes span the FULL matchup period, not just the current calendar week.** `mode="prev"` audits the last completed matchup (recap post-mortem); `mode="current"` audits the in-progress matchup **first Monday → yesterday** (today excluded as incomplete; digest, so misses are still fixable). Returns `{}` on a Monday with no completed days. The span comes from ESPN's schedule — a matchup entry's **`home.pointsByScoringPeriod` keys ARE the exact days that count** (robust for the 14-day All-Star matchup; falls back to this-Monday±7 on fetch failure). Computed in `main()`, stored as snapshot fields `lineup_efficiency` (prev) + `lineup_efficiency_current` (current). Per-day slot comes from `mRoster?scoringPeriodId=<day>` (categories `box_scores()` has team totals only). **Three findings:** (a) **batter bench leakage** — R/HR/RBI/SB in a `BE` slot, reported **NET of the weakest startable bat you'd have benched** (opportunity-cost correction; feasibility of a no-bench slot-in is a max-bipartite-matching check `_full_match` on `eligibleSlots` — see NOTES); net uses only *notable* days (HR/SB/2+R/2+RBI). (b) **pitcher blowups** — an active-slot start with 5+ ER (or 4+ ER in <3 IP) that counted, cross-refed against `recent_activity(size=150)` drops → "imploded then dropped". (c) **idle active hitters ("wasting space")** — 0-AB active-slot hitter, **counting only games his MLB team actually played** (schedule-gated via `_ESPN_PROID_TO_MLBID`; fetch-fail → count all); fires only on a pattern: **3+ consecutive idle games** OR an AB in **< 50%** of active games (min 4). **OPS is never netted** (rate stat — benching an 0-4 day helps OPS). Pitcher/hitter split via `eligibleSlots`. Broad try/except → `{}` on any failure. Standalone opponent + head-to-head version: **`bench_leakage.py`**.

**`fetch_todays_games(league)`** (`fetch_data.py`) → snapshot field `todays_games` — today's real MLB games enriched with which **rostered** players (any fantasy team) are in each, so the digest can surface the games that most overlap the current matchup. One `statsapi /schedule?date=today&hydrate=probablePitcher,broadcasts(all)` call. **Roster→game join is on MLB team id** (robust vs FantasyPros/StatsAPI abbrev quirks): espn_api exposes a player's club as a `proTeam` **abbrev string** (`'Sea'`, NOT a numeric `proTeamId`), so it's mapped abbrev → ESPN id via **`espn_api.baseball.constant.PRO_TEAM_MAP`** (inverted) → MLB id via `_ESPN_PROID_TO_MLBID`; players bucketed by MLB id, then matched to each game's `teams.home/away.team.id`. Each involved player carries `is_p` (pitcher?), `is_sp` (the game's **confirmed probable starter**?), and `is_rp` (**reliever by USAGE ROLE**, not position — via `_espn_is_reliever(pl)`: ESPN season GS/GP ≤ 0.20 with ≥ 5 apps → RP, ≥ 0.80 → SP, ambiguous/thin falls back to `eligibleSlots` RP-only). `is_rp` is what lets the renderer COUNT relievers (a save/hold chance moves the matchup) while still skipping a starter on his off-day. Per game: `gamePk`, `game_time_utc`, `home_name`/`away_name` (full names — renderer maps via `_FULLNAME_TO_ABBREV`), `home_prob`/`away_prob` (probable-starter full names for the pitching-matchup line, `""` when TBD), broadcasts split into `national_tv` (list, `isNational`), `home_tv`/`away_tv` (each side's local RSN — so every game has a channel, not just the rare national game), and `involved` (`[{name, FantasyTeam, is_p, is_rp, is_sp}]`). Games with zero rostered involvement are dropped. Broad try/except → `[]` so a StatsAPI hiccup never blocks the snapshot write. Consumed by `send_digest.build_todays_games_section` + the Briefing "📺 Tune in" teaser.

**`build_lineup_efficiency`** (`weekly_recap.py`) renders `lineup_efficiency` (prev matchup) as a **LINEUP EFFICIENCY** band after My Matchup: red "LEFT ON THE BENCH" net headline, per-player bench cards (each notable day + swap target `[vs Name H-AB]`/`[open slot]`), active-slot blowups with the drop flag, then an **"IDLE — WASTED ACTIVE SLOTS"** group. **`build_bench_watch`** (`send_digest.py`) renders `lineup_efficiency_current` (matchup-to-date) as the compact **Lineup Watch** callout (section 1b, after `alert_section`) — net-recoverable batter line + implode-then-drop pitchers + a "Wasting active space" idle-hitter line (top 3); silent on a clean matchup; header "matchup to date". Gated on `not override_team` (snapshot stores only MY daily lineup).

**`data/snapshot.json`** is the schema contract between the two files. ~1.2 MB, not committed. Numeric missing values are stored as `-1` (not `NaN`) after the merge pipelines run.

**`snapshot_schema.py`** guards that contract (stdlib-only, no deps). `validate_snapshot(snap)` → `(errors, warns)` (never raises); `assert_valid(snap)` raises `SnapshotValidationError` on any error; `report(errors, warns)` prints an ASCII summary. **ERROR** (fetch fundamentally failed → a reader crashes/reads garbage): missing spine key (`refreshed_at`/`my_team`/`standings`/`pitchers`/`hitters`/`roto`), an empty core list, wrong-typed spine key, unparseable `refreshed_at`. **WARN** (degraded but survivable — readers `.get()`-guard these): a missing/mistyped non-spine key, row-field coverage below 90% (`_ROW_FIELDS`; **`FantasyTeam` is intentionally excluded** — empty for the FA pool), or a NaN leak where the `-1` sentinel belongs. **Wired at both ends:** `fetch_data.main()` calls `assert_valid(...)` right before `json.dump` — a raise exits nonzero, so send_digest/dashboard fall back to the last good snapshot instead of writing a broken one; the three readers (`send_digest`/`dashboard`/`weekly_recap`) call `validate_snapshot` right after `json.load` and `report` only on a hit (quiet on a clean snapshot, never blocks — catches drift under `--no-refresh` where the write-time guard didn't run). All wiring is `try/except ImportError` so the validator is optional. CLI: `python snapshot_schema.py [path]` (defaults to `data/snapshot.json`). **When you add/rename a top-level snapshot key or a required row field, update `_SPINE`/`_EXPECTED`/`_ROW_FIELDS` to match.**

## Critical gotchas

### Data sources
- **Pending trades aren't in `espn_api`.** `get_pending_trades(league, my_team)` hits the raw ESPN endpoint `.../leagues/{id}?view=mPendingTransactions` with the same `espn_s2`/`SWID` cookies. **Trades live under the top-level `pendingTransactions` key, NOT `transactions`** (that key is empty in this view — the initial dead end). Each `TRADE_PROPOSAL` item has `fromTeamId`/`toTeamId` (direction) + `type=="TRADE"`; `memberId == my SWID` ⇒ I proposed it (`incoming=False`). Broad try/except → `[]`.
- FanGraphs returns 403 — never use directly. pybaseball works (handles headers).
- `pitching_stats()` (FanGraphs leaderboard) → 403. Use `pitching_stats_range()` (Baseball Reference) — but it has no `HLD` column.
- SVHD comes from ESPN via `get_pitcher_espn_svhd()`, reading `pl.stats[0]['breakdown']`, which uses **string keys** (`'SV'`, `'HLD'`, `'SVHD'`, `'K'`, `'W'`, `'OUTS'`, `'ERA'`, `'WHIP'`, `'GP'`, `'GS'`) — not numeric IDs. Called at fetch time for all rostered + FA pitchers.
- ESPN season stats (`ESPN_SV`/`ESPN_K`/`ESPN_W`/`ESPN_IP`/`ESPN_GS`/`ESPN_GP`/`ESPN_SVHD`) are stored on **all** dataset rows so send_digest can use season counts for players who only appear in short-range FantasyPros datasets. `ESPN_SVHD`/`ESPN_SV`/`ESPN_HLD` override `SVHD`/`SV`/`HLD` on `Dataset==YEAR` rows; `ESPN_HLD` is then dropped but `ESPN_SV` stays on all rows (the only way `save_role_watch` distinguishes a real closer from a holds-only reliever for players outside the FP top-300). Use `_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))` (not `if >= 0`) — `_n` floors negatives to 0 so `>= 0` is always true.
- xFIP / CSW are unavailable from FantasyPros (and FanGraphs — 403). But Baseball Savant (via pybaseball) supplies predictive pitcher stats, merged on `PlayerName`: `xERA` + `xwOBA_against` (`get_savant_pitcher_expected`, absolute values), `WhiffPctile` (a league whiff **percentile** 0–100, not a rate, from `get_savant_pitcher_skill`), and `BarrelPctAllowed`/`HardHitPctAllowed`/`AvgEVAllowed` (`get_savant_pitcher_contact`). Coverage ≈ 90–99% of the FP top-300; missing rows fall back to raw ERA/K% cleanly.
- **`WhiffPct` (raw overall whiff %, DISPLAY-ONLY)** — via `get_savant_pitcher_whiff` (pybaseball `statcast_pitcher_arsenal_stats`), pitch-type rows aggregated by pitches-weighting per `player_id`. A raw 0–100 **rate**, DISTINCT from `WhiffPctile` (a 0–100 **percentile**). Merged on `PlayerName` via `merge_on_name`. **Never feed it into `pitcher_score`/`rp_score`** — `WhiffPctile` already drives the K component, so raw whiff% would double-count. Displayed only: a muted `whiff NN%` subline under the K% cell in the compacted My Upcoming Starts + FA SP tables (`_whiff_sub`, keeps them at 8 cols), and a real **Whiff%** column (green ≥ 30) in Pitcher Hot/Cold (7 cols; `score_reveal` colspan 7).
- ESPN injury statuses are `TEN_DAY_DL`/`FIFTEEN_DAY_DL`/`SIXTY_DAY_DL` — not `IL`/`OUT`. Constant `_DL_STATUSES` covers all. FA views and positional breakdown exclude all DL-status players.

### Names & merges
- **Team name double-space:** `MY_TEAM_NAME = "Guerrero  Warfare"` (double space) in fetch_data.py matches ESPN exactly. `MY_TEAM = "Guerrero Warfare"` (single space) in send_digest is for display. Never normalize these to match each other.
- **Merge direction:** pitcher/hitter merges start from FantasyPros (left side). Players outside the FP top 300 drop from short-range (7-day) views but appear in longer ranges. FantasyPros↔FantasyPros merges (`fp7`, season→all-rows enrich) stay exact.
- **`merge_on_name` / `_name_key`:** ESPN and FantasyPros names differ by accents and generational suffixes (`Luis García Jr.` vs `Luis Garcia`); an exact-string merge drops the roster link so a rostered player wrongly shows as a free agent. `merge_on_name` does the exact merge first, then fills still-unmatched rows via `_name_key` (accent-stripped, lowercased, trailing Jr./Sr./II–V + punctuation removed). Invariants: (a) exact matches always win — the fallback only fills NaNs; (b) a key is trusted only when it maps to a single player on **both** sides (the several MLB "Luis Garcia" pitchers stay ambiguous, never guessed). Wired into both roster+FA merges AND the Statcast/pybaseball merges. Older per-player `HITTER_NAME_PATCHES`/`PITCHER_NAME_PATCHES` still applied.
- **Index-alignment invariant:** build the fill-loop keys `fkeys` from `merged["PlayerName"]`, NOT `fp["PlayerName"]` — `fp.merge()` resets to a clean RangeIndex, so keys from `fp` can be unalignable with the `missing` mask from `merged` → `IndexingError: Unalignable boolean Series`.
- **Statcast name matching:** `lf_to_name()` converts Baseball Savant "Last, First" → "First Last" AND strips accents (`Ramírez, José` → `Jose Ramirez`) so the merge against ASCII FantasyPros names succeeds. Without it, accented-name players silently lose all Statcast data.

### Fields & sentinels
- **PSP sentinel:** `PSP_Date = "1999-01-01"` = no upcoming start. `PSP_Projected = True` = start projected via the +6-day rotation rule, not confirmed by the MLB API.
- **Two-start pitchers (`PSP_Dates`):** fetch_data preserves a list of ALL upcoming start dates per pitcher (`PSP_Dates` + parallel `PSP_HomeVAways`) via `_attach_start_lists` before the one-row-per-pitcher dedup. Scalar `PSP_Date`/`PSP_HomeVAway`/`PSP_Projected` remain the earliest start. `_starts_this_week(r, today, week_end)` counts entries within the matchup week. ≥ 2 starts Mon–Sun → bold **blue** (`ACCENT`) chip labelled just `2` (`two_start_badge()`) in FA SP + My Upcoming Starts, and preferred (secondary sort key, NOT a score change) in the Week-at-a-Glance best-FA-SP bullet ("×2 starts this week"). Never fold two-start into the 0–100 score.
- **B_SO is lower-is-better BUT a COUNTING stat (not a rate):** batter strikeouts sit in `_LOWER_BETTER` alongside ERA/WHIP. Affects Category Pulse bar direction and projection flip logic — fewer B_SO than the opponent is a win. **Do NOT add it to `_RATE_CATS`** — it accumulates over the week (values ~70), so `_project` must use the counting branch (`current + remaining·avg`, monotonically increasing), not the rate blend (which pulls a high current value *down* toward the weekly avg → a projection below the current count, the "proj 69 when already at 76" bug). Its `_CLOSE_THRESH` is on the counting scale (`8`, matching R/RBI/K), not the rate scale. `_RATE_CATS`/`_SEASON_RATE_CATS` are `{OPS, ERA, WHIP}` only — B_SO is the one lower-better cat that is NOT a rate.
- **`ESPN_OnIL`** (native python bool = `pl.lineupSlot == "IL" or pl.injured`) is captured in `get_pitcher_roster`/`get_hitter_roster` and broadcast to all rows. `lineupSlot` is the primary check (slot 17 in ESPN's POSITION_MAP); `pl.injured` is a fallback because ESPN's API sometimes omits `lineupSlotId` from the roster entry (e.g. Will Smith on the 60-day IL), leaving `lineupSlot = ''`. Keep it a **native bool**, NOT `.astype(bool)` → numpy `bool_`, which `json.dump(default=str)` stringifies to the truthy `"False"`.

### FA logic
- **FA exclusion:** players claimed today are found via today's `transactions` list; the *most recent* transaction per player wins. FA views + positional breakdown exclude all `_DL_STATUSES`.
- **FA RP requires SVHD ≥ 1:** `fa_relievers` gates on `(_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))) >= 1`. Zero saves + zero holds all season = no role.
- **FA SP score floor (`_FA_SP_MIN_SCORE = 30`):** `fa_starters` drops any starter whose `_score_p` is below 30 (streamer-tier arms not worth the risk) BEFORE the top-12 slice. Applied inside `fa_starters` so all three consumers agree — the FA SP section, the Week-at-a-Glance best-FA-SP bullet, and `_roster_suggestion`'s pitch-stabilizer pool. Tune the one constant to change the bar; the section subtitle + empty-state message interpolate it.
- **FA "Cats" column (`_cats_cell` / `player_cat_strengths` / `build_cat_percentiles`):** FA Hitters + FA Relievers show a **Cats** column (before Score) listing up to 3 roto cats the player is strong in (percentile ≥ 0.70 within a qualified YEAR pool; `_LOWER_BETTER` inverted). Cats in `need_cats` (my currently-losing ∪ tossup) render `ACCENT`; others `MUTED`. `_FA_HIT_CATS=[R,HR,RBI,SB,OPS]`, `_FA_RP_CATS=[SVHD,K,W,ERA,WHIP]`.

### Roster/drop rules
- **Never drop an IL-slot player (`_on_il`):** the league has **2 dedicated IL slots** that don't consume active/bench room, so dropping a player parked there frees nothing. `_on_il(r)` (tolerant of a stringified `"true"`) gates two drop paths: (a) `_can_drop` returns False (Week-at-a-Glance bullet 4); (b) `positional_breakdown`'s `worst_player` uses `drop_pool = [r for r in my_p if not _on_il(r)] or my_p`. It is lineup-SLOT-specific, NOT injury status — a DL player on the BENCH is `False` and stays droppable (cutting him frees a real bench spot). Populated for all teams; only my rows affect my suggestions.
- **`_can_drop`** still guards that every position keeps ≥ 1 healthy player.

### SP/RP role detection (`_is_sp(r)`)
Never use `"SP" in pos` or `gs > 3` alone. Priority chain: ESPN season GS/GP ratio (≥ 5 appearances) → dataset GS/G ratio (≥ 4 appearances) → IP/G → Position field. Thresholds: GS/G ≥ 0.80 → SP, ≤ 0.20 → RP; IP/G ≥ 4.5 → SP, < 2.5 → RP. Used by `pitcher_score`, `_score_p`, `fa_starters`, `fa_relievers`, My RP filter, `positional_breakdown`.

### Pitcher hot/cold uses 15-day ERA
`build_pitcher_hot_cold_section`, My Upcoming Starts, and FA Starting Pitchers compare season ERA vs 15-day ERA (`p15` = Dataset==15 rows). The 7-day window is too short for infrequent SPs. When a player is absent from the FP 15-day top 300 (fringe starters), fall back to `rec_p` (pybaseball Baseball Reference 15-day scrape in `recent_pitching`). Column header "L15 ERA". `fetch_recent_pitcher_stats` fetches 15 days to match. Both `fetch_recent_pitcher_stats` and `fetch_recent_hitter_stats` accept optional `start_dt`/`end_dt` string params for an exact date window (used to populate `prev_week_hitting`/`prev_week_pitching`).

### Save-Role Watch (`save_role_watch`)
**Recent holds are unavailable anywhere in the pipeline** (per-window `SVHD` captures recent SAVES only; ESPN exposes only season totals), so recency is save-only. Flags (a) **emerging FA closers** (FA RP with ≥ 3 saves in last 15 days) and (b) **fading rostered closers** (my RP with **season saves `ESPN_SV` ≥ 5**, 0 recent saves despite ≥ 3 recent appearances). Gated on season *saves*, not SV+H, so a holds-based reliever (JoJo Romero 0 SV / 20 HLD) is never falsely flagged. Callout on the FA Relief Pitchers section.

### Category classification (`classify_categories`)
Returns `{cat: (proj_res, tier)}` reusing Category Pulse's projection math (`_project` + `pit_proj` for K/QS/W). Tier is only `tossup` (margin ≤ `_CLOSE_THRESH`) or `leaning` — **no `locked` tier / no 🔒 badge**. Used to detect a THIN ERA/WHIP lead for the ratio-stat pickup warning. Computed once in `build_email` as `category_classification`, passed to `_roster_suggestion`; the pickup steering targets all losing cats (no dead-cat pruning).

### Matchup W-L uses ESPN's `result` field, NOT raw comparison
`get_all_matchups`/`get_all_prev_matchups` read `box.home_stats[cat]["result"]` (`WIN`/`LOSS`/`TIE`) as source of truth (→ `W`/`L`/`T`). Critical because ESPN applies a ratio-stat **innings-pitched minimum** (~25 IP) before ERA/WHIP count — the better WHIP under the IP floor **loses** (rationale + the 10-2 vs 11-1 bug → NOTES). **Intended consequence:** the current result can differ from the projected result (Category Pulse compares raw projected values without the floor) — don't unify, the divergence is informative. **Falls back to a raw value comparison when `result` is `None`** — essential because ESPN leaves per-category `result` unset for the *live* matchup until it closes.

### `roto_score_week` ranks by VALUE, not `result` (so the live current week populates)
Ranks teams on each category's `value` (coerced numeric), NOT ESPN's per-category `result`. Deliberate: ESPN leaves `result = None` on the live matchup until it closes, so a result-gated filter would drop the whole current week → `_all_roto_tied` True → **silently suppresses digest sections 4 + 4b** mid-week. Ranking by value is identical for completed weeks and additive for the live week. `get_all_roto` uses `continue` (not `break`) on a failed week so one bad week can't truncate history.

### Category Pulse
- Tied cats use `TEXT` (white) for border/value/status — not `YELLOW`. Win=green, loss=red, tie=white. Card value (`my score`/`vs opp`) stacked on two lines so decimal-heavy stats don't cause width/height inconsistency. Card div is `position:relative`.
- **Corner badge** (`position:absolute` top-right, not inline) carries two things: the **⚡ toss-up flag or the WIN %** (mutually exclusive — ⚡ replaces the number on a toss-up since exact odds don't matter at a coin-flip; otherwise the % shows), then the **projected-outcome marker ▲▼◆** after it.
  - **⚡ = win-% toss-up, NOT a current-margin close:** fires when `win_pct` ∈ 45–55 **or `proj_res == "T"`**. Collected into `close_flags` → summary `⚡N close`. (`_CLOSE_THRESH` is still used for the `_cat_win_prob` sigma fallback + `classify_categories` tossup tier — just not the card ⚡.)
  - **▲▼◆ = the PROJECTED OUTCOME, not a flip:** renders on **every** card with `proj_res is not None` (▲ green win / ▼ red loss / ◆ white tie); a flip is visible by *contrast* with the card's current status. `proj_res` uses `round(pm/po, dec)` so it can't disagree with the point-estimate tie test.
- **INTENTIONAL DIVERGENCE — do NOT unify:** `build_matchup_section` (~line 1720) shows ▲▼◆ **only on a flip** (`flip = proj_res != res`). Only Category Pulse always-shows projected outcome; leave the Matchup table flip-only.
- **Opponent This Week — always wrap text in an explicit color.** The panel sits on `SURFACE2` (dark); without an explicit `color:` span text inherits the client default (often black) and disappears. Every name/value in `opp_preview_section` must carry `color:{TEXT}` (or `MUTED`) — same rule for any new dark-panel content.
- Summary line: current then projected record, each as full **W · L · T** (T always shown, even `0T`): `10W · 2L · 0T · ⚡N close → proj 11W · 1L · 0T`. The `⚡N close` segment appears only when ≥ 1 cat is close.
- **`days_elapsed`** = days since matchup start (0 on Monday). Derived from `matchup_start_date` — NOT `datetime.now().weekday()` — so it counts across a 2-week matchup. Guard: `day_clause = f' through Day {days_elapsed}' if days_elapsed > 0 else ' (week starting)'`.
- **Pitcher projections (K, QS, W)** use actual remaining starts × per-start rate (`pit_proj`, passed as `build_category_pulse(remaining_proj=pit_proj)`), not weekly averages. Rate/hitter cats still use historical averages via `compute_weekly_avgs`.
- **Win-probability (`_cat_win_prob` + `compute_weekly_std`):** each card's `WIN %` chip is colored to `proj_res`; ⚡ replaces it on toss-ups. `compute_weekly_std(roto, week)` → per-team/per-cat stddev (needs ≥ 2 completed weeks), threaded into `build_category_pulse(weekly_std=…)`. `_cat_win_prob(pm, po, cat, sigma, remaining_frac)` → `(p_win, p_tie)` via normal-CDF (`math.erf`): `edge` direction-adjusted for `_LOWER_BETTER`; `sigma = sqrt(my_std² + opp_std²)` (falls back to `_CLOSE_THRESH[cat]`); counting-cat uncertainty × `remaining_frac`; tie band `0.5·10^-dec`. Display-only — `classify_categories` untouched, summary record stays point-estimate `→ proj`.

### Terminology: user-facing labels say "Matchup N", NOT "Week N"
The ESPN unit is a **matchup period** (usually 7 days, **14 for the All-Star break/playoffs**; the number is a matchup *index*, not a calendar week — rationale in NOTES). Every **display label** for that unit uses **"Matchup {n}"** (Roto Rankings, "Matchup at a Glance", "Starts/Opponent This Matchup", recap Highlights, Season Trajectory "by matchup"). Only **genuine 7-day rolling windows keep "week"**: "last 7 days", "7-day OPS", "L15 ERA". (Lineup Watch now spans the full matchup — header "matchup to date" — NOT a week exception.) **Internal names are NOT renamed** (snapshot `Week`, `weekly_results`, `week_end_str`, `current_week_num`, `prev_week`, `weekly_avgs`, `recap_week_N.html`, `weekly-recap.yml`) — display strings only.

### Weekly matchup is Monday–Sunday (or Monday–Sunday×2 for All-Star break)
`week_end_str` comes from `matchup_end_date` in the snapshot (not a hardcoded `today + 6 - weekday`). `fetch_data.get_matchup_dates(league)` derives the window in two parts. **START** = the earliest DAILY scoring-period id actually scored in the current matchup, from `pointsByScoringPeriod` (`_current_matchup_start_sp`, raw `mMatchupScore` view — the same authoritative day-span source `get_lineup_efficiency` trusts), converted to a date via `today - (today_sp - start_sp)` (daily SP ids are consecutive days). **Do NOT anchor START at "this Monday" (`today - weekday`)** — that lands a full week late for the **2nd week of a multi-week (All-Star) matchup**, zeroing `days_elapsed`/`game_days_elapsed` so every counting-cat projection stacks a full period onto already-banked stats (the "proj 145 B/SO when at 76" bug). Falls back to the Monday anchor only when `pointsByScoringPeriod` is unavailable. **LENGTH** uses the `finalScoringPeriod` surplus arithmetic (`remaining_daily = final_sp - true_start_sp + 1`; `period_days = 7 + min(max(0, remaining_daily - remaining_regular*7), 7)`) — NOT `len(matchup_periods[id]) * 7`, because ESPN's `matchup_periods` dict is **degenerate in this league** (period 15 → `[15]`, all length-1, so it never reveals the 14-day span). Anchoring at the true start is what makes the surplus resolve to 14 mid-break (the extra first week is no longer excluded). Stored in snapshot as `matchup_start_date`, `matchup_end_date`, `matchup_period_days`, `next_matchup_end_date`, plus **`matchup_game_days`/`matchup_game_days_elapsed`** (real MLB game days in the window via a `statsapi /schedule?gameType=R` query, excluding the All-Star break; drive `elapsed_frac` in Category Pulse/`classify_categories`). Dates past `week_end_str` get a `NEXT WK` badge. **Bullet 2 scopes both `confirmed` and `thin_days` to `PSP_Date <= week_end`** so its count matches the KPI.

### End-of-matchup mode (`is_sunday`)
When `today >= matchup_end_date` (last day of the matchup period — not always a calendar Sunday for multi-week periods): header subtitle → "Weekly Lookahead"; subject → "Lookahead"; KPI → "Starts Next Week" (counts starts after `week_end_str`); Category Pulse subtitle → "Final stretch — week ends today"; Week at a Glance → "Next Week Preview"; bullet 1 appends "— final"; bullet 2 shows next-period confirmed starts; bullet 3 shows best FA SP for next period. `classify_categories` and `build_category_pulse` accept `matchup_days=matchup_period_days` so `elapsed_frac` divides by 14 (not 7) for a 2-week period.

### Probable starters
Primary: two MLB API calls (range schedule → batch hydrate) give **confirmed** starters (`PSP_Projected=False`), published only ~1–2 days out (live-feed fallback if the batch returns nothing). **Unannounced slots are filled by a rotation projection that advances each team's rotation ORDER through its actual upcoming games** (`get_probable_starters` Step 3, NOT the old +6-day guess — rationale in NOTES). The walk: `_get_team_recent_starts(days_back=14)` → build a rotation **queue** of members who started within `_ROT_RECENCY` (11 days — excludes IL/optioned arms), ordered by last-start ascending (longest-rested = most "due" = front); walk upcoming games in date order, assigning each **open** game to the front-of-queue pitcher and rotating them to the back (one game = one pitcher). Confirmed MLB entries **anchor**: their slot is skipped and the pitcher rotated to the back to re-sync; a pitcher with a confirmed *future* start is skipped for earlier open games (`confirmed_date` guard). A pitcher can legitimately appear once confirmed + once projected — `_attach_start_lists` folds both into `PSP_Dates`; scalar `PSP_Date` stays the earliest.

### HR% (`_hrp_cell`)
`HR_Probability` (`fetch_data.compute_hr_probability` from barrel%, hard-hit%, launch angle, HR/AB, xwOBA, ISO, recent HR streak; ≈ 0.05–0.31 modeled per-game HR prob) → color-coded `HR%` column in Roster Hot/Cold + FA Hitters via `_hrp_cell(row)`, hover `title` = drivers. Green ≥ 20%, yellow ≥ 14%. Takes the full row (Roster Hot/Cold stashes the season row as `srow`). **Measures power SKILL, not availability** — must NOT gate on `ESPN_Status` (an earlier gate zeroed out Judge/Trout/Buxton). Returns 0.0 → "—" only when no usable signal. `ISO = SLG − AVG` (FP omits it).

### Week at a Glance pickups (`_roster_suggestion`) are roster-context aware — returns a LIST of bullets
Returns a **list** of up to two bullets (build_week_overview `.extend`s them; still tolerates a bare string). Rationale/history → NOTES.
- **BAT bullet** — upgrades my **weakest hitter position** where a real FA upgrade exists. Uses `positional_breakdown` (`pos_data`) for per-position league `rank` + my `worst_player` + best `top_fa` (both carry `_pscore`). A position is a **NEED** if `rank` is bottom-third (`rank ≥ n − round(n/3) + 1`) AND `top_fa._pscore − worst._pscore ≥ _UPGRADE_MARGIN` (**3.0**). Among needs, picks the **weakest** (highest rank), tie-broken by upgrade size. **SURPLUS positions are excluded** (top-third rank OR `lineup_efficiency_current` shows bench leakage there via `leak_groups`). Fallback with no positional need: best FA hitter for a losing hitter cat, still not at a surplus position.
- **PITCH bullet** — fires when `lineup_eff.blowups` is non-empty **OR** I'm losing ERA/WHIP by a **non-tossup** margin (`classification[cat][1] != "tossup"`). Recommends a high-**floor** stabilizer from `fa_sp + fa_rp`, gated on a real sample (`_pit_viable_min`) + good ratios (ERA ≤ 4.00, WHIP ≤ 1.25) + `not _is_blowup_risk`, ranked by `(xERA or ERA, WHIP)` ascending.

Drops prefer **surplus** (deep position / bench-leaker), surplus-first then score-ascending; the two bullets take **distinct** drops (`_take_drop`/`_used_drops`); a `[surplus]` tag marks a drop-from-strength. `_move_tail` shows a green "roster spot open" badge while `slots_left > 0`, else appends the drop. `_can_drop` guards ≥ 1 healthy player per position and never drops an IL-slot player. Wired: `_roster_suggestion(..., pos_data=pos_data, lineup_eff=(lineup_efficiency_current if not override_team else {}))` — `--team` passes `{}`, so its pitch bullet fires only on ratio loss.

### Trade Radar (`find_trades` + `build_trade_radar`) — cross-team, FAVOR-ME
Surfaces trade ideas that **fix a rival's category need** (their reason to accept). Two lanes (see TWO LANES below): a **FAIR** lane (realistic, near-even/pay-up deals a rival would actually accept — the ones that land a scarce C/SS upgrade) and a **FAVOR-ME** lane (good-not-great pieces, my elite bats protected, value tilted to me, buy-low/sell-high timing); `find_trades_combined` blends them, leading with fair. Perspective-driven via `my_team`, so it works under `--team` for free (all inputs are all-team) — **not gated** on `not override_team`. Renders in the TRANSACTIONS band (the renamed FREE AGENTS band — it now holds FA pickups + trades; anchor id stays `band-fa`) after FA Hitters (`build_email` body_parts, section 13b); returns `""` (section omitted) when no candidates exist.
- **Category matrix:** `team_category_ranks(roto)` generalizes `category_ranks` to `{team_key: {cat: rank}}` (rank 1 = best; same `{cat}_Points` aggregation, so `_LOWER_BETTER` is baked in). A team's **need cats** = bottom-third (`rank ≥ n − third + 1`), **surplus cats** = top-third (`rank ≤ third`), `third = round(n/3)` — same convention as `_roster_suggestion`.
- **Acceptance gate:** for rival T, `send_cats = my_surplus ∩ T_needs` (cats I help them in) and `get_cats = T_surplus ∩ my_needs` (cats they help me in). **`send_cats` must be non-empty** (their reason to say yes) or the pair is skipped; the get side can be a category need OR a positional need (below).
- **CROSS-ROLE TRADE VALUE (`_trade_value` → `_tval`) — the parity currency, NOT the role-score badge.** A 95 closer and a 98 hitter both top their role but are NOT equal chips: an everyday bat moves 5 cats daily while a closer touches one PUNT cat (SV+H) + a sliver of K/ERA/WHIP in ~60 IP. `_trade_value` = `Σ max(0, pctile(c) − 0.5) × w` over the player's role cats, `w = _TRADE_SVHD_W`(0.35) for SV+H else 1.0. Percentiles are vs the WHOLE qualified pool (hitters `hit_pctile`; pitchers `pit_pctile` = `build_cat_percentiles(all YEAR pitchers, _FA_RP_CATS)`, built in the prelude next to `hit_pctile`), so **a reliever's low K/W volume ranks low** and an everyday smasher outweighs a one-category closer. **The role-score badge (`_tscore`) is still DISPLAYED; parity/labels use `_tval`.** (Rationale + the "James Wood for a top closer" bug → NOTES.)
  - **POSITIONAL SCARCITY (`compute_position_scarcity` → `_POS_SCARCITY`, HITTERS ONLY):** the raw percentile sum is position-*blind* — an everyday OF and an everyday SS graded equal, so the favor-me radar handed you "give abundant OF depth, get a scarce-position bat" deals that read fair on paper but **no rival would accept** (a fleece the currency couldn't see). The fix scales a hitter's `_tval` by a **positional-scarcity multiplier**: `mult[P] = baseline / starter_avg_tval[P]`, where `starter_avg_tval[P]` = mean raw `_tval` of the league's top-`POS_STARTERS[P]·n_teams` eligible bats at position P (the "starters" there — same top-K convention as `positional_breakdown`), and `baseline` = the mean of those per-position averages. A position whose starter talent is weak (**low** mean value, e.g. **C**) is SCARCE → `mult > 1` (a competent bat there clears replacement by more); a deep position (e.g. **1B**) → `mult < 1`. Clamped to `_POS_SCARCITY_CLAMP`(0.75–1.50) so a thin pool can't over-swing. A **multi-eligible** bat takes his **scarcest** position's multiplier (best-slot value — a C-eligible hitter carries the C premium). Set by `compute_position_scarcity(hitters, hit_pctile)` in the prelude of all three trade surfaces (`build_email`, `dashboard.build_context`, `trade_lab.build_data`) **right after `hit_pctile`/`pit_pctile`**, so Trade Radar, Trade Lab, and the Pending-Trade grader all inherit accurate values at once. **Reset to `{}` at the top of the compute fn** so the internal `_trade_value` calls (which read the global) return RAW value while the scale is being derived — the empty global is a no-op passthrough, which is also the cold-start behavior when uncalled. **Pitchers are NOT scaled** — pitcher value is already punt-saves-shaped (`SV+H` discounted), a separate axis from scarcity, and uniform positional scarcity would fight the punt-saves discount (boosting relievers). The scale is mild in practice (a typical snapshot: C ~×1.28, 1B ~×0.78, 2B/3B/SS/OF ~×1.0 — so it mainly says "1B value is cheap, catcher value is a premium," and confirms SS is a *roster* weakness, not market scarcity).
- **Outgoing pool (GOOD-not-GREAT, my studs protected)** = my players strong in a `send_cats` cat, **at a surplus POSITION** (`pos_data` rank top-third, no hole opened), **AND `_tval ≤ _TRADE_MAX_VAL`(1.35) UNLESS `_tsell`** — my elite keepers are never offered, except a **sell-high** regression candidate (whom I *want* to move). **Incoming pool** = T's players that either help a `get_cats` cat OR **`_fills_need_pos`** — a hitter at one of my THIN hitter positions (`need_pos` = `pos_data` bottom-third rank) whose `_tscore` clears my positional avg there (a genuine upgrade). Positional need is hitter-only (the hitting game is filling C/1B/2B/3B/SS/OF holes; pitching need stays category-shaped). "Strong in" reuses `player_cat_strengths`. **QS/B_SO aren't per-player stats**, so a team category-need in those finds no player — intentional (same as FA "Cats").
- **Buy-low / sell-high timing (`_tsell`/`_tbuy`):** hitters via `_regression_flag` (xBA/xSLG-vs-actual), pitchers via `pitcher_regression_flag` (ERA-vs-xERA, de-biased — see the pitcher-badge gotcha). **Directional:** selling-high on the GIVE side and buying-low on the GET side are the arbitrage (my sell-high guy's actual stats OVERstate his value, their buy-low guy's UNDERstate his — I win even when `_tval` looks even); the reverse (dealing a riser, acquiring a regressor) is a TRAP, penalized in ranking and chip-warned ORANGE. Timing chip tooltips are role-neutral ("results running ahead/behind his Statcast expected") so they read right for a pitcher too.
- **TWO LANES (`find_trades(..., mode)`) — both grade on the SAME scarcity-weighted `_tval`.** `mode="favor"` (default) tilts value to me; `mode="fair"` finds realistic, near-even deals a rival would ACCEPT — the ones that actually land a scarce C/SS upgrade (favor-me by construction can't surface a fair-priced scarce-position deal, so it never showed them). Each trade is tagged `"lane"`. **`find_trades_combined`** runs both lanes and **interleaves them leading with FAIR** (user priority — the realistic ask first, then a value play), deduping identical packages and keeping the per-team (`_TRADE_PER_TEAM_CAP`=2) + total (`_TRADE_MAX_CARDS`=6) caps joint across lanes. The digest `build_trade_radar` and dashboard `build_context` both call `find_trades_combined` (NOT `find_trades` directly), so both surfaces show the blend.
  - **FAVOR lane gate (`_favor_me`):** I don't overpay (`give − get ≤ _TRADE_GIVE_SLACK`(0.15)) and may extract up to a steal ceiling (`get − give ≤ _TRADE_MAX_EDGE`(0.45)); give-side capped at `_TRADE_MAX_VAL`(1.35) (studs protected, sell-high exempt). Ranking `score = Σ my-need-depth(gcov+gpos) + 0.3·Σ their-need-depth(scov) + 5.0·net_val + 4.0·timing − 0.5·(pkg−1)` — their benefit DOWN-weighted (0.3), reward my value edge.
  - **FAIR lane gate (`_fair`):** near-even — I keep at most `_TRADE_FAIR_BAND`(0.20) edge and may PAY UP at most `_TRADE_FAIR_PAYUP`(0.35) to land a scarce need-fill. Give-side ceiling RAISED to `_TRADE_FAIR_MAX_VAL`(1.90) (deal a genuinely good chip for fair return, but still protect a franchise anchor — else the value gate would just block every fair deal; dropping the ceiling entirely auto-offered my single best bat, so a higher ceiling not NO ceiling). Ranking `score = Σ my-need-depth + 0.5·Σ their-need-depth + 4.0·timing − 4.0·max(0,−net_val) − 0.5·(pkg−1)` — their benefit UP-weighted (0.5, acceptance likelihood matters) and instead of rewarding an edge it PENALIZES paying up, so the closest-to-even need-fills rank first. Both gates: 2-for-2 scales the value slack by ×1.5; shapes 1-for-1 + 2-for-2 (top-`_TRADE_POOL_WIDTH`=4 each side; incomers cover ≥ 2 distinct needs — cats OR positions). `net_val` = my `_tval` edge (`get − give`), `timing` = `sell_out + buy_in − buy_out − sell_in`.
  - **Acceptance read (`_trade_tilt(net_val)` → `(value_phrase, accept_phrase, color)`):** shared by the digest card header + the dashboard tile. `value_phrase` is my-POV (*you win the value / even value / you pay up*); `accept_phrase` is the RIVAL's-POV — **realistic** (GREEN, `net_val ≤ _TRADE_FAIR_BAND`) vs **aggressive ask** (YELLOW, I clearly win). Net-val-driven, NOT lane-driven (so a fair-lane 2-for-2 that happens to tilt >0.20 to me still reads "aggressive ask" — honest, no user-visible lane contradiction since the lane tag is internal-only). The digest renders it as a bordered chip in the card header next to the partner; the dashboard as a colored sub-line under the value line.
- **Card render:** partner `fantasy_logo` + name; two columns **You give** (RED header, `send_cats` chips MUTED) / **You get** (GREEN header, `get_cats` chips ACCENT + **CYAN position chip** for a thin slot the incoming hitter upgrades). `_trade_player_line(r, hi_cats, hi_color, side, show_pos, best_recent_p, best_recent_h, hit_pctile)` renders the **CANONICAL** buy/sell chip — the same glyph-only `$` (GREEN, `_tbuy`) / `▼` (RED, `_tsell`) used everywhere else, NOT a recolored/relabeled variant (keep the visual language consistent — an earlier cut used green ▼ + word labels + ORANGE trap warnings and read inconsistently). `side` ('give'/'get') only tunes the hover tooltip; trap-direction trades are penalized out of the list anyway. **Every player's Score pill is tap-to-expand** (role-aware prose breakdown, both give AND get sides): because trade players are stacked `<div>`s inside table cells (not per-player `<tr>` rows), it uses `_trade_score_reveal` — a **div-based analog** of `score_reveal` that appends a hidden `<div class="scorebd-div">` right after the player line, revealed by a `div.scorebd-div:target` rule (mirrors the `tr.scorebd-row:target` rule; browser-attachment only, Gmail strips `<style>`). Breakdown fn is role-picked off `_tptype` (`_hitter_score_breakdown(r, best_recent_h, hit_pctile)` / `_pitcher_score_breakdown(r, best_recent_p)`); `build_trade_radar` threads `best_recent_p`/`best_recent_h`/`hit_pctile` in. Footer "Upgrades your {cats + `{pos} slot`} ; they shore up {send} · {value}" where value = *you win the value / even value / you pay up* from **`net_val`**, plus a `· sell-high`/`· buy-low` thesis tag that carries the whole-trade framing. Constants are one-line tunables near the FA-cats constants.

### Pending Trades evaluator (`_grade_pending_trades` + `build_pending_trades_section`) — grades REAL offers
Grades the actual pending trade **offers** involving my team (snapshot `pending_trades`), reusing the Trade Radar machinery so every number matches the digest. Because offers **expire**, the incoming-offer headline surfaces HIGH (Week at a Glance + the Briefing "Act today" list), not just in the dedicated section. Gated on `not override_team` (the snapshot stores only my pending trades).
- **Grade ONCE, share everywhere:** `_grade_pending_trades(pending, pitchers, hitters, roto, my_team, best_recent_p, best_recent_h, pos_data, hit_pctile, pit_pctile, today_str)` → a list of graded dicts, computed once in `build_email` right after `pos_data`. The dedicated section, the Briefing, and the Week-at-a-Glance headline all read from it (`incoming_pending = [g for g in graded_pending if g["incoming"]]`) so they can't disagree.
- **Reuse (single source):** `_trade_value(r, ptype, hit_pctile, pit_pctile)` (the `_tval` currency) and `_enrich_trade_player(r, ptype, best_recent_p, best_recent_h, hit_pctile, pit_pctile)` (sets `_tscore`/`_tcats`/`_tval`/`_tsell`/`_tbuy`/`_tgroups`/`_tptype`) were **promoted from `find_trades` closures to module level** so both `find_trades` and this evaluator share one implementation. `find_trades` now calls `_enrich_trade_player` via a thin local `enrich`. `_hitter_fills_need_pos(r, need_pos)` is the shared positional-need test.
- **Player→row resolution:** each traded player's ESPN name is matched to its snapshot YEAR row via a `_badge_name_key` lookup (fall back 30→15→7), exactly like the Today's Games block; role (hit/pit) is decided by which pool matches. Unmatched names render a plain "(no data)" line and are excluded from the value math (never crash).
- **Grade:** `net_val = Σ_tval(get) − Σ_tval(give)`; value tilt reuses Trade Radar's ±0.1 thresholds (*you win the value / even value / you pay up*). Category/positional impact = incoming `_tcats ∩ my_needs` + thin hitter positions filled (`need_pos` from `pos_data`); timing legs `sell_out+buy_in−buy_out−sell_in`.
- **Verdict (`_pending_verdict`, INCOMING only):** ACCEPT (GREEN) when I win value (or roughly even AND it fills a need) and timing isn't a net trap; COUNTER (YELLOW) when the direction is right but I pay up or the timing is a trap; DECLINE (RED) when no need is addressed and I don't gain value. Rendered as a `_verdict_pill` + one-line rationale. **OUTGOING** offers (I proposed) show an "Awaiting {partner} · {value} from your side" status instead (their call, no verdict).
- **Counter suggestion (`_counter_suggestion`, on a COUNTER):** when I'm overpaying by `gap` (= `−net_val`), suggests the single best ADD-ON to request from the partner — a spareable piece (`_tval ≤ gap + _TRADE_MAX_EDGE`) that closes the value gap (`_tval ≥ 0.7·gap`) and ideally helps one of my need cats/positions (preferring buy-low). Rendered as a 💡 line in the card **and** appended to the headline (`→ counter: ask them to add X`). Empty when no realistic add-on exists.
- **Expiry (`_fmt_trade_expiry`):** "expires in Nd" from the ISO `expires` field, shown RED in the card header, the headline, and the Briefing.
- **High-up headline (`_pending_headline(g, brief=False)`):** one-line 🤝 headline (partner · get/give · verdict pill · expiry · counter) prepended to **Week at a Glance** (`build_week_overview(..., trade_bullets=[...])`, top of the bullet list) and the **Briefing** "⚡ Act today" list (`render_briefing(..., pending_incoming=incoming_pending)`, first items — they expire, so most time-sensitive). The old roster-alerts-box nudge was removed (redundant with these higher placements).
- **Card render (`build_pending_trades_section(graded, best_recent_p, best_recent_h, hit_pctile, team_logos)`):** reuses `_trade_player_line` verbatim (MLB logo + tap-to-expand score badge + cat chips + canonical `$`/`▼` buy/sell chip), same You give (RED) / You get (GREEN) columns as Trade Radar, at the TOP of the TRANSACTIONS band (section 10b). Returns `""` when nothing is pending.

### Tap-to-expand score breakdown v2 (v1→v2 rationale in NOTES)
Tapping a Score badge reveals a **full-width row below the player's row** narrating the 2–3 most decisive drivers in prose. The recent-form clause **names the actual window** (`30-day`/`15-day`/`7-day`) from the recent row's `Dataset` (30 > 15 > 7 > pybaseball; hitter → 7-day, pitcher → 15-day) — intentionally differs from the Hot/Cold Δ column.
- **Mechanism (no JS, email-safe):** `score_reveal(score, breakdown_html, uid, colspan)` returns a tuple `(cell_html, row_html)`. `cell_html` = badge in `<a href="#{uid}">` (▾ caret); `row_html` = a hidden `<tr id="{uid}" class="scorebd-row" style="display:none;">` spanning `colspan`, appended immediately after the player's `</tr>`. Head-`<style>` rule `tr.scorebd-row:target { display:table-row !important; }` reveals it; `✕` (`href="#{uid}x"` → dead anchor) closes it. (Gmail strips `<style>` → rows stay hidden there.)
- **Scroll positioning:** the `:target` rule also sets `scroll-margin-top:40vh` (send_digest.py ~line 4015). Browser-attachment only (Gmail strips `<style>`).
- **`_bd_uid(prefix, name)`** mints a globally-unique anchor id (`bd-{prefix}-{slug}-{counter}` via a running `_BD_SEQ`). Prefixes: `rhc`/`phc`/`mus`/`myrp`/`fasp`/`farp`/`fahit`/`posw`/`posfa`.
- **Narrative (`_score_narrative` + `_hit_clauses`/`_sp_clauses`/`_rp_clauses`):** each `_*_clauses` returns `(fill, strength_phrase, weakness_phrase)` per component (`fill = comp_points / max`); `_score_narrative` names ≤ 2 strongest (fill ≥ .60) + ≤ 2 weakest (≤ .35): `Carried by … ; held back by …`. Punt-saves-consistent: low SVHD / low HR% NOT surfaced as weaknesses; SP `Role` omitted. **HR/ISO power dedupe (`_hit_clauses`):** when ISO is strong and HR weak (same "power" concept), the HR weakness clause is dropped (symmetric for the reverse); the strength always survives.
- **Wired into all Score badges:** the 7 tables (Roster Hot/Cold, Pitcher Hot/Cold, My Upcoming Starts, My Relief Pitchers, FA SP, FA RP, FA Hitters) plus Positional Breakdown (weakest-my-player `posw` + best-FA `posfa`, role-aware via `p["ptype"]`, `colspan=4`). **HR% drivers are in the expanded hitter panel** (`_hitter_score_breakdown`) as a trailing muted `<div>` via `_hrp_driver_str(row)` — so touch users (no hover) see them.
- **Badge context (why each chip fired):** the panel appends a muted block explaining each tactical badge the row earns (exact chip + triggering stat). Hitters — `_hit_badge_context(row, hit_pctile, cap=None)` (SAME predicates/order/cap as `hitter_badges`, so it explains exactly the chips shown), appended inside `_hitter_score_breakdown` (takes a `hit_pctile` param, threaded from all 3 call sites). SP — `_sp_badge_context(row, qs_fires, k_fires, two_start_n, recent_era=None)` fed the fire flags + L15 ERA (`p15r.get("ERA")`) **at the My Upcoming Starts + FA SP sites only, NOT in the shared `_pitcher_score_breakdown`** (else QS/5K+/⚠ context would show in Pitcher Hot/Cold where no chip renders). Explains QS / 5K+ / 2 + the **⚠** chip (fires on `_is_blowup_risk(row, recent_era)`, renders `blowup_badge` + `_risk_drivers` — same `recent_era` as the inline chip). Reuses `_hit_badge`/`two_start_badge()`/`blowup_badge()` so chips are byte-identical to the table.

### Unified role scores — a player shows the SAME score in every section
Three canonical role scores, all calibrated to p50→50, p90→80: SP → `_score_p` (blended `pitcher_score`), RP → `rp_score` (never blended — ESPN season counting stats, identical across My RP / FA RP / Positional Breakdown), Hitter → `_blend(r, hitter_score, best_recent_h)`. Never score a section with a different function than others use for the same role.

### Hot/Cold columns & KPI
- Both `build_pitcher_hot_cold_section` and `build_hot_cold_section` take a `best_recent_*` index and render a role-aware Score badge (pitcher → `_score_p`, hitter → `_blend(hitter_score)`).
- **Roster KPI hot/cold counter (`hc_str`):** the "Roster" KPI tile counts my ENTIRE roster — hitters AND pitchers. Hitters use 7-day OPS vs season (±0.015); pitchers use 15-day ERA vs season (±0.40, ≥ 3 recent IP, `rec_p` fallback). The two thresholds differ by design (OPS vs ERA scale) — keep the KPI in sync with each section's threshold. Tile label is "Roster" (whole team).

### Score cascade (`best_recent_p` / `best_recent_h`)
Built in `build_email` by merging `{**rec_p_fp, **p7, **p15, **p30}` (pitchers) and `{**rec_h, **h7, **h15, **h30}` (hitters) — later dicts win, so 30d FP > 15d FP > 7d FP > Baseball Ref. Passed to `_blend` and `positional_breakdown`.

### positional_breakdown viable filter
FA pool per position excludes benchies. SP: `GS >= _pit_viable_min("SP","GS")`. RP: `ESPN_GP >= _pit_viable_min("RP","GP") or IP >= _pit_viable_min("RP","IP")`. Hitters: `OPS > 0.200 or R+RBI > 5`. FA quality (`fa_quality`) = avg blended score of top-3 viable FAs. Scarcity: `< 50` scarce (RED), `< 60` moderate (YELLOW), `>= 60` deep (MUTED).

### positional_breakdown ranks on TOP-K STARTERS, not the mean of all eligible (no phantom needs)
`my_avg`/`rank` per position = the average of a team's **top-K players** at that spot, K = `POS_STARTERS` (`{C:1,1B:1,2B:1,3B:1,SS:1,OF:3,SP:4,RP:3}` ≈ the active-lineup count). It is **NOT the mean of every eligible player** — that let bench/utility depth create phantom needs: a backup catcher carrying 1B eligibility (or a cold bat parked behind a starter) dragged a position's average into the bottom third even when the actual starter was strong (the "1B need with Olson at 1B" bug — Caratini's C/1B eligibility + a cold post-injury Murakami sank a position anchored by an 83-score Olson). `_starter_avg(scores, k)` slices the top-K of each team's eligible scores (my team AND every rival, so the league rank stays fair) before averaging. Feeds the digest Positional Breakdown, `_roster_suggestion`'s BAT bullet need detection, and Trade Radar / Trade Lab `need_pos` (all read `rank`/`my_avg`) — so the fix flows everywhere at once. `worst_player` (the implicit drop target) is still the genuine weakest non-IL body, unaffected.

### Dynamic volume benchmarks (no hard-coded IP/AB/GS minimums)
"Full-time" thresholds are derived from the live snapshot each run so they scale with the season. Two builders, both called once at the top of `build_email`:
- `compute_ab_benchmarks(hitters)` → `_AB_BENCH[window]` = `_AB_LEADER_FRAC` (0.62) × the window's p95 (leader) AB. Consumed by `_ab_opportunity_mult` in `hitter_score`. `_FULLTIME_AB` is a cold-start fallback only.
- `compute_pitcher_benchmarks(pitchers)` → `_PIT_BENCH[(window, role)]` = leader IP/GS/GP (p95) per role, `_is_sp`-split. `_ip_reliability_mult` uses `_IP_RELY_FRAC` (0.20) × leader IP for the row's window+role. `_pit_viable_min(role, stat)` uses `_GS_VIABLE_FRAC`/`_GP_VIABLE_FRAC`/`_IP_VIABLE_FRAC` (0.17/0.30/0.38) × the season leader. `_PIT_FALLBACK` holds cold-start constants.

### Data-derived league averages (`_LG` / `compute_league_averages`)
Called once in `build_email` next to the benchmark builders; writes `_LG` with `ops` (full-time regulars), `team_ops` (mean opponent OPS faced), `team_k`, and starter `era`/`whip`/`k_pct`/`ip_per_start`/`barrel_allowed` from qualified YEAR rows. Consumers read `_LG.get(key) or <old literal>`. `qs_probability` stays calibrated because the intercept `38` and multipliers are fixed. fetch_data derives its own `LG_OPS` for wRC+. ONLY genuine league averages live in `_LG`; calibration/scaling constants (score spans/floors, park factor, `IP*4.3`, `compute_hr_probability` weights) do NOT.

### Dry-run preview filenames
Always `previews/digest_preview_{team_slug}.html` (e.g. `digest_preview_Guerrero_Warfare.html`). No `digest_preview.html` fallback — always slug-based.

### `--team` flag
`python send_digest.py --team "Team Name"` shows a full digest from another team's perspective (all sections render, incl. Category Pulse + Matchup banner). Requires `all_matchups` in the snapshot. `build_matchup_section` accepts `my_team` (default `MY_TEAM`). **Monday recap is per-team:** `get_all_prev_matchups(league)` builds prior-week recap for ALL teams into `all_prev_matchups`; `prev_matchup` resolves to `all_prev_matchups[my_team]`.

## Scoring functions (send_digest.py)

- `_is_sp(r)` → bool. Usage-based SP/RP detection (see gotcha).
- `_blend(r, score_fn, idx_recent, w=None)` → blended score. `_BLEND_W = 0.35` (35% recent + 65% season) — single source for math + tooltip. `idx_recent` is `best_recent_p`/`best_recent_h`. Applies to hitters + SPs; RP `rp_score` never blended.
- `hitter_score(r, _parts=…)` / `pitcher_score(r, _raw=…, _parts=…)` / `rp_score(r, _raw=…, _parts=…)` → `_parts=True` returns `(components_dict, multiplier)`. Component insertion order == display order (single source for tap-to-expand).
- `_score_p(r, idx_recent=None)` → canonical role-aware pitcher score. SP → `_blend(r, pitcher_score, idx_recent)`; RP → `rp_score(r)` unblended. Used by every pitcher Score display/sort.
- `_starts_this_week(r, today, week_end)` → int. Upcoming starts within the matchup week (from `PSP_Dates`; falls back to scalar `PSP_Date`). Drives the two-start `2` badge and best-FA-SP preference.
- `save_role_watch(pitchers, my_team, claimed)` → `(emerging, fading)` (see gotcha).
- `classify_categories(matchup, weekly_avgs, days_elapsed, remaining_proj, matchup_days=7)` → `{cat: (proj_res, tier)}` (see gotcha). Pass `matchup_days=matchup_period_days` for 2-week periods.
- `compute_weekly_std(roto, current_week)` → per-team/per-cat stddev; `_cat_win_prob(pm, po, cat, sigma, remaining_frac)` → `(p_win, p_tie)` (see Category Pulse gotcha).
- `opponent_week_intel(pitchers, hitters, opp_team, best_recent_h, today, week_end)` → dict (starts, two-start pitchers, hot hitters) for the Opponent This Week block. None when `opp_team` empty.
- `pitcher_score(r, _raw=False)` → 0–100. Components: K/WhiffPctile (28), ERA/xERA (28), WHIP (20), contact-quality/BarrelPct+xwOBA (0–12), SP role bonus 9–12 by GS. Small-sample penalty `s *= min(1.0, ip/20)`. Calibrated `s * A + C` with `(A,C) = _SCORE_CALIB["sp"]` (re-anchored live — see below). `_raw=True` returns pre-calibration.
- `rp_score(r, _raw=False)` → 0–100. SVHD de-emphasized to ~15% (punt-saves). Components from ESPN season counts: SVHD (15), K (26), W (15), ERA/xERA (16), WHIP (12), IP/G (8), contact-quality (0–8). Calibrated `s * A + C` with `(A,C) = _SCORE_CALIB["rp"]`. My Relief Pitchers picks best dataset per player (YEAR → 30 → 15 → 7).
- **Live SP/RP recalibration — `compute_score_calibration(pitchers)`** (rationale → NOTES): SP/RP calibration constants are **re-derived every run** from the snapshot's raw-score distribution (same p50→50/p90→80 solve + qualified pool as `recalibrate_scores.py`: SP = `_is_sp` + IP past `_IP_RELY_FRAC` of the leader; RP = `_pit_viable_min` on GP or IP; both YEAR rows), written to the module global `_SCORE_CALIB = {"sp":(A,C), "rp":(A,C)}` that `pitcher_score`/`rp_score` read. **Called in `build_email` right AFTER `compute_pitcher_benchmarks`** (`_raw` scores read `_PIT_BENCH` — order matters). **SMALL-POOL FALLBACK:** a role whose pool is below `_MIN_CALIB_POOL` (30) or degenerate (`p90 <= p50`) keeps the hand-tuned defaults (SP 1.5070/-44.3346, RP 1.6543/-28.0645). `recalibrate_scores.py` is now a manual inspection tool + the source of those fallback literals — update the defaults from it if the component mix changes materially. Hitter score is NOT live-recalibrated (`s * 1.587 - 5.2`).
- `hitter_score(r)` → 0–100. Prefers wRC+ over OPS. Uses xwOBA, sprint speed, Barrel%, ISO, HR_Probability. Opportunity multiplier (`_ab_opportunity_mult`): raw score scaled by AB vs a full-time benchmark (floored `_AB_FLOOR = 0.40`, capped 1.0) — a full-time hitter lands at 1.0 (no penalty). Calibrated `s * 1.587 - 5.2`. Displayed everywhere as `_blend(r, hitter_score, best_recent_h)`.
- `qs_probability(r)` → 1–99. Calibrated league-avg ~38%, ace ~75%. Uses IP/G (not IP/GS).
- `_fmt_ip(ip_decimal)` → baseball IP notation. `whole = int(d); outs = round((d-whole)*3); if outs>=3: whole+=1, outs=0`.
- `_proj_line_html(r)` / `_proj_line_vals(r)` → `IP · ER · K`. ER = `raw_er * opp_factor * park_factor`, `raw_er = era_reg * IP_per_G / 9`; **`era_reg` = row ERA regressed toward `xERA` (falls back to `_LG["era"]` when missing), IP-weighted: `(ERA*IP + target*_ERA_REG_PRIOR_IP)/(IP + _ERA_REG_PRIOR_IP)`, `_ERA_REG_PRIOR_IP = 40`** (backtest-confirmed ER win; residual −0.33 bias is structural blowup-skew, not ERA noise — see NOTES/todo). `opp_factor = clamp(opp_ops / LG_team_ops, 0.80, 1.20)`; `park_factor` = 0.97 home / 1.03 away. K opponent-adjusted via `Team_K_Value`. IP = `IP_per_G` (backtest-confirmed best predictor at r=0.27; IP_per_GS and pitch-count predictors rejected).
- `hot_cold_cell(season_val, recent_val, …, no_data_title=None, td_style=TDC)` → `<td>` with colored recent stat + 🔥/↑/❄/↓ icon vs season baseline. When recent is missing/zero and `no_data_title` is set, renders `—` with a dotted underline + hover tooltip. Optional `td_style` so the compacted pitcher tables' L15 ERA cell matches.
- `band_divider(label, color, anchor=…)` → full-width band boundary `<div>`.

### QS / 5K+ / 2 (two-start) badges (My Upcoming Starts + FA SP)
- `2` (blue/`ACCENT`, `_starts_this_week ≥ 2`), QS (cyan), 5K+ (yellow) render next to the pitcher name via `two_start_badge(title)` / `qs_badge(ip_g, er, row=None)` / `k5_badge(k, row=None)` — the QS/5K+ helpers wrap `_hit_badge` so the chip is byte-identical to the hitter badges + carries a hover `title` naming the projected line that earned it. The dashboard My Pitching tile mirrors the tooltips on its own 8px chips. **QS and 5K+ purely annotate the projected line** — driven ONLY by `_proj_line_vals(r)`, NOT season rates: QS = `_proj_is_qs` (6+ displayed IP & ≤ 3 ER via `_fmt_ip` rounding), 5K+ = projected `K ≥ 5`. The **QS% column** shows season QS probability separately. **The QS tooltip + its `_sp_badge_context` dropdown line both name the run-prevention analytic behind the projection** via the shared `_qs_stat_clause(row)` (`xERA`, falling back to raw `ERA`; empty when neither present; deliberately does NOT restate IP/start, which would just repeat the projected-line IP) — pass `qs_badge` the row at every site (parallel to `k5_badge`). The QS tooltip drops the "6+ IP, ≤ 3 ER" definition (it's in the glossary). **The 5K+ tooltip + its `_sp_badge_context` dropdown line both name the K-skill behind the projection** via the shared `_k5_stat_clause(row)` (raw `WhiffPct` → `WhiffPctile` → `Kpct_P`, first available; empty when none) — pass `k5_badge` the row at every site so the inline chip's tooltip carries it too. Display-only — never fed into `pitcher_score` (`WhiffPctile` already drives K; raw whiff% would double-count). (See NOTES.)
- **FA SP badges are unconditional:** they fire on the projected line wherever the pitcher appears (the old thin-rotation-day gate was removed). `thin_days`/`my_starts_by_day` still drive the ⚑ per-day "N my starts" banner and Week-at-a-Glance bullet 2.

### Blowup-risk (⚠ / low-floor) flag — starters only
A burnt-orange (`ORANGE`) glyph-only **⚠** chip next to a startable arm's name flags a **low floor** (disaster-start prone). **`blowup_risk(r, recent_era=None)` → 0–100** (higher = lower floor); **`_is_blowup_risk(r, recent_era)`** = `_is_sp(r) and blowup_risk ≥ _RISK_MIN` (55 → ~10–12% of startable arms); **`blowup_badge(r, recent_era)`** renders the chip (empty when not flagged, hover title = worst 2–3 drivers via `_risk_drivers`). **DISPLAY-ONLY — never folded into `pitcher_score`/`_score_p`** (an independent floor warning, not a quality knock; a high-ceiling arm can still be risky). Four skill drivers league-anchored via `_LG`, tunable via `_RISK_*`: WHIP, K%/`WhiffPctile` (escape hatch), **`_effective_era(r)`** (ERA regressed toward xERA, SAME `_ERA_REG_PRIOR_IP=40` shrinkage as `_proj_line_vals`), `HardHitPctAllowed`. **Recent form is an ADDITIVE-ONLY escalator** (`_RISK_RECENT_W=0.25`): a cold `recent_era` (L15, from the `p15`/`rec_p` row at each render site) RAISES risk, a hot stretch does NOT lower it. (Why skill-only + validation numbers → NOTES.) **Wired:** inline chip in My Upcoming Starts + FA SP (`blowup_badge(r, p15r.get("ERA"))`), dashboard My Pitching (8px) + FA Radar Starters (9px); the tap-to-expand dropdown explains it via `_sp_badge_context(..., recent_era=p15r.get("ERA"))` (see Badge context). **Steer:** best-FA-SP pick deprioritizes flagged arms (`not _is_blowup_risk` primary `max` key, "⚠ low floor" note if the top pick is still flagged); `_roster_suggestion` pitch-stabilizer pool excludes them (`and not _is_blowup_risk(r)`).

### Hitter tactical badges (PWR / SB / BUY / SELL)
Glance flags next to a **hitter's** name, mirroring the SP badges — **display-only, never folded into any score** (rationale → NOTES). Shared source `hitter_badges(row, hit_pctile=None, cap=None)` in send_digest.py (dashboard imports `sd.hitter_badges`), chips via `_hit_badge(text, color, title)`. Each carries a hover `title` naming the justifying stat, so a badge stays anchored even in a table with no matching column. Priority **PWR → SB → BUY/SELL**; `cap=None` → all applicable badges show (`_hit_badge_context` matches with the same order + `cap=None`). Four flags (tunable constants, ~10–25% of the qualified YEAR pool):
- **PWR** (`PURPLE`) — `HR_Probability ≥ _PWR_HRP_MIN` (0.23, ~7% of pool). Title = `_hrp_driver_str(row)`.
- **SB** (`SILVER`, "Quicksilver") — SB percentile `≥ _SB_PCTILE_MIN` (0.80) via the `build_cat_percentiles` SB pool AND `SprintSpeed ≥ _SB_SPEED_MIN` (27.0, skipped when missing). Needs the `hit_pctile` pool — silently doesn't fire when `None`.
- **$** (buy-low, `GREEN`) / **▼** (sell-high, `RED`) — Statcast expected-vs-actual regression, mutually exclusive; `SLG = ISO + AVG`; buy = `xBA−AVG ≥ _XREG_BA` (0.020) AND `xSLG−SLG ≥ _XREG_SLG` (0.030); sell = both inverted. Requires `xBA/xSLG/AVG/ISO > 0`.

**Callers pass the `hit_pctile` pool** (`build_cat_percentiles(_hit_pool, _FA_HIT_CATS)` on the qualified YEAR pool — `build_email`, and dashboard `build_context` as `ctx["hit_pctile"]`). **Digest sites:** Roster Hot/Cold (`build_hot_cold_section` takes a `hit_pctile` param), FA Hitters, Positional Breakdown (hitter branch, gate on `p["ptype"]`), **Today's MLB Games** (each involved player, role-picked in `_badges`). **Dashboard sites:** Hitting Hot/Cold, FA Radar hitters, Weakest Spots (hitter positions). **When adding a hitter surface**, drop `hitter_badges(row, hit_pctile)` after the name with a `hit_pctile` pool in scope.

### Pitcher buy/sell badge (`$` / `▼`) — the pitcher analog of the hitter regression badge
`pitcher_regression_badge(row)` → glyph-only `$` (buy-low, `GREEN`) / `▼` (sell-high, `RED`) chip, `""` when neither. `pitcher_regression_flag(row)` → `'buy'`/`'sell'`/`None`. Signal = **ERA vs Baseball Savant xERA**, gated on `IP ≥ _XREG_ERA_IP` (20). **DISPLAY-ONLY** (never folded into any score), same as the hitter version.
- **DE-BIASED (critical):** xERA runs systematically ABOVE ERA in this data (~+0.33 median), so a raw symmetric threshold over-fires "sell" (37% vs 17% observed). `compute_xera_offset(pitchers)` sets module global `_XERA_OFFSET` = league median `(xERA − ERA)` over the qualified YEAR pool; the flag tests `(xERA − ERA) − _XERA_OFFSET` against `_XREG_ERA` (1.00) → luckier/unluckier **than typical** (~12% sell / 15% buy). **Called in `build_email` right after `compute_league_averages`** (same calibration block as `_LG`/`_SCORE_CALIB`); 0.0 cold-start default if uncalled.
- **DISTINCT from `⚠` (blowup-risk):** `▼` sell-high = MEAN regression / luck (ERA better than deserved, due to rise); `⚠` = single-start TAIL risk / low floor. They share the ERA/xERA input so they can co-fire (a strong "move him" signal) but are not redundant — a soft-contact arm can be sell-high without being blowup-prone, and a hard-contact arm can be blowup-prone with an honest (non-lucky) ERA.
- **Digest sites:** Pitcher Hot/Cold (`r["srow"]`), My Upcoming Starts + FA SP (appended to the `start_badges`/`pickup_badges` list beside QS/5K+/⚠), My Relief Pitchers + FA RP (name cell), Positional Breakdown (pitcher branch — replaces the empty `_is_pit_pos` else). Tap-to-expand explained in `_sp_badge_context` (SP sites). **Dashboard sites:** My Pitching (`_reg_chip8`, 8px), FA Radar SP+RP, Weakest Spots (pitcher rows) — `build_context` calls `compute_xera_offset` so `_XERA_OFFSET` is set there too. **Feeds Trade Radar** (`_tsell`/`_tbuy` for pitchers). **When adding a pitcher surface**, drop `pitcher_regression_badge(row)` after the name (best on a YEAR/season row — the flag needs season ERA vs xERA).

  - **`--with-dashboard` (`send_digest.main`):** after building the digest, imports `dashboard` (deferred — dashboard imports send_digest at module load, so the import must be inside `main()` at call time to avoid a circular import), calls `dashboard.build_dashboard(snap, my_team)`, and passes `(dash_html, filename)` as `extra_attachments` to `send_email` (which now takes an optional `extra_attachments` list of `(html, filename)` tuples appended after the digest attachment). **The dashboard build is wrapped in try/except** — a dashboard failure only drops its attachment; the digest still sends (a bad-data day can never sink the whole email). Under `--dry-run` it also writes `previews/dashboard_{slug}.html`.
- **Fetch-time freshness badge:** the record and category standings are a point-in-time snapshot of ESPN's live box scores at fetch time. The header badge shows the fetch **time in ET** (`_fmt_refresh_time`): green ✓ when data is from today, yellow ⚠ when stale. `fetch_data.py` writes `refreshed_at` as tz-aware UTC (`datetime.now(timezone.utc)`) so the ET conversion is correct on CI (UTC) or local; `_fmt_refresh_time` converts to `America/New_York`, shows naive stamps as-is. `_data_fresh` keys off the date only.
- **Local runner:** `scripts/run_digest.bat` for manual runs. Captures console output (incl. tracebacks) to `logs/run_console.log`; the structured one-line-per-send record goes to `logs/digest.log` (written by send_digest.py). Kept in **separate files** — the old shared handle collided (`PermissionError` on Windows). The Python write is wrapped so a locked log can't crash a run that already sent.
