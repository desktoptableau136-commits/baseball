# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Full run: refresh data (~60s) then send email
python send_digest.py

# Skip data refresh, use existing snapshot (fast ‚ÄĒ for email-only changes)
python send_digest.py --no-refresh

# Save HTML to previews/digest_preview.html without sending email
python send_digest.py --dry-run

# Instant preview with no network calls
python send_digest.py --dry-run --no-refresh

# View any team's full digest (requires a fresh snapshot with all_matchups)
python send_digest.py --dry-run --no-refresh --team "Houck Tuah"

# Refresh data only (writes data/snapshot.json)
python fetch_data.py

# Install dependencies
pip install -r requirements.txt
```

No linter, no test suite. Verify changes by opening `previews/digest_preview.html` in a browser.

## Setup

Copy `.env.example` ‚Üí `.env` and add a Gmail App Password (not your regular password ‚ÄĒ create one at myaccount.google.com/security ‚Üí App Passwords).

## Architecture

Two files; one intermediate artifact:

**`fetch_data.py`** pulls from 5+ sources and writes `data/snapshot.json`:
1. FantasyPros HTML (`pd.read_html`) ‚ÄĒ pitcher and hitter stats across 4 ranges (7/15/30/season)
2. ESPN Fantasy API (`espn_api`) ‚ÄĒ rosters, FA list, roto box scores, standings, transactions
3. MLB Stats API ‚ÄĒ probable starters (batch hydrate method) + opponent OPS
4. pybaseball ‚ÄĒ Statcast contact quality, expected stats, sprint speed, recent game logs

**`send_digest.py`** reads the snapshot, computes all derived metrics, and builds a single self-contained HTML email sent via Gmail SMTP (`smtplib`). The email has two parts: inline HTML body (may be clipped by Gmail at 102 KB) and an attached `digest_YYYY-MM-DD.html` for full render. All new features go here.

**`data/snapshot.json`** is the schema contract between the two files. It is ~1.2MB and not committed.

## Critical gotchas

**Data sources:**
- FanGraphs returns 403 ‚ÄĒ never use it directly. pybaseball functions work because they handle headers.
- `pitching_stats()` (FanGraphs leaderboard) returns 403. Use `pitching_stats_range()` instead, which scrapes Baseball Reference ‚ÄĒ but it has no `HLD` column.
- SVHD (saves+holds) is pulled from ESPN player stats via `get_pitcher_espn_svhd()` in fetch_data.py, which reads `pl.stats[0]['breakdown']`. The breakdown uses **string keys** (`'SV'`, `'HLD'`, `'SVHD'`, `'K'`, `'W'`, `'OUTS'`, `'ERA'`, `'WHIP'`, `'GP'`, `'GS'`) ‚ÄĒ not numeric stat IDs. This is called at fetch time for all rostered and FA pitchers.
- ESPN season stats (`ESPN_SV`, `ESPN_K`, `ESPN_W`, `ESPN_IP`, `ESPN_GS`, `ESPN_GP`, `ESPN_SVHD`) are stored on **all dataset rows** in the snapshot so send_digest.py can use season counts for players who only appear in short-range FantasyPros datasets. `ESPN_SVHD`/`ESPN_SV`/`ESPN_HLD` override `SVHD`/`SV`/`HLD` on `Dataset==YEAR` rows; `ESPN_HLD` is then dropped but `ESPN_SV` is kept on all rows (it's the only way `save_role_watch` can tell a real closer from a holds-only reliever for players outside the FP top-300, who have no YEAR row). Use `_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))` (not `if >= 0`) for the fallback ‚ÄĒ `_n` floors negatives to 0 so `>= 0` is always true.
- xFIP and raw WhiffPct/CSW are not available from FantasyPros pitcher tables (nor FanGraphs ‚ÄĒ 403). But Baseball Savant (via pybaseball) *does* supply predictive pitcher stats, pulled in `fetch_data.py` and merged on `PlayerName` (broadcasts to all dataset rows): `xERA` + `xwOBA_against` from `get_savant_pitcher_expected()` (absolute values), and `WhiffPctile` (a league whiff **percentile** 0‚Äď100, not a rate) from `get_savant_pitcher_skill()`. Plus `BarrelPctAllowed`/`HardHitPctAllowed`/`AvgEVAllowed` from `get_savant_pitcher_contact()`. `pitcher_score`/`rp_score` blend these with the results stats ‚ÄĒ see the scoring section. Coverage ‚Čą 90‚Äď99% of the FP top-300; missing rows fall back to raw ERA/K% cleanly.
- ESPN injury statuses are `TEN_DAY_DL`, `FIFTEEN_DAY_DL`, `SIXTY_DAY_DL` ‚ÄĒ not `IL` or `OUT`. The constant `_DL_STATUSES` in send_digest.py covers all of these. FA views and positional breakdown exclude all DL-status players.

**Team name double-space:** `MY_TEAM_NAME = "Guerrero  Warfare"` in fetch_data.py has a double space to match ESPN exactly. `MY_TEAM = "Guerrero Warfare"` in send_digest.py uses a single space for display. Never normalize these to match each other.

**Merge direction:** Pitcher and hitter merges start from FantasyPros as the left side. Players outside the FP top 300 are dropped from short-range (7-day) views but appear in longer ranges.

**PSP sentinel:** `PSP_Date = "1999-01-01"` means no upcoming start. `PSP_Projected = True` means the start was projected via the +6-day rotation rule, not confirmed by the MLB API.

**FA exclusion logic:** Players claimed today are identified by reading today's `transactions` list from the snapshot. The *most recent* transaction per player wins ‚ÄĒ so add-then-drop-same-day is handled correctly (dropped players re-appear as FA). FA views and positional breakdown replacement options exclude all `_DL_STATUSES` players.

**B_SO is lower-is-better:** `B_SO` (batter strikeouts) is in `_LOWER_BETTER` alongside ERA and WHIP. This affects the Category Pulse bar direction and projection flip logic ‚ÄĒ having fewer B_SO than the opponent is a win.

**Matchup W-L uses ESPN's own `result` field, NOT a raw value comparison:** `get_all_matchups` / `get_prev_matchup` in fetch_data.py read `box.home_stats[cat]["result"]` (`"WIN"`/`"LOSS"`/`"TIE"`) as the source of truth for each category, mapping it to `W`/`L`/`T`. This is critical because ESPN applies a **ratio-stat innings-pitched minimum** (e.g. 25 IP) before ERA/WHIP count: a team with the *better* WHIP that is still under the IP floor **loses** that category. A naive `h_val < a_val` comparison misses this and reported the wrong record (e.g. 10-2 when ESPN showed 11-1 ‚ÄĒ my WHIP 1.55 "lost" to the opponent's 1.37 on raw value, but the opponent had only ~22.7 IP < 25 so ESPN scored it a WIN for me). The raw-value comparison remains only as a fallback when ESPN supplies no `result` (old snapshots / edge cases). **Note the intended consequence:** the *current* result (from ESPN's `result`, which honors the live IP floor) can legitimately differ from the *projected* result (Category Pulse / `classify_categories` still compare raw projected values, since both teams usually clear the IP min by Sunday) ‚ÄĒ so a category you're currently "winning" only because the opponent is under the IP min will correctly show a ‚Ėľ flip arrow projecting the loss once they qualify. Don't try to make the projection honor the IP floor; the divergence is informative.

**Category Pulse card design:**
- Tied categories use `TEXT` (#e2e8f0, white) for border/value/status ‚ÄĒ not `YELLOW`. Win = green, loss = red, tie = white.
- ‚ö° (close) and flip indicators (‚Ė≤‚Ėľ‚óÜ) live in an `position:absolute` top-right corner badge, not inline with the status or projection text. The card div is `position:relative`.
- Flip uses `round(pm, dec)` / `round(po, dec)` (display precision) for outcome comparison ‚ÄĒ raw floats cause false flips when both round to the same displayed value.
- Flip arrow: ‚Ė≤ green = projecting to flip to a win; ‚Ėľ red = projecting to flip to a loss; ‚óÜ white = projecting to flip to a tie.
- Summary line shows the current record then the projected record, each as full **W ¬∑ L ¬∑ T** (the T is always shown, even at `0T`, on both sides ‚ÄĒ the user wants to see the zero, not have it omitted): `10W ¬∑ 2L ¬∑ 0T ¬∑ ‚ö°N close ‚Üí proj 11W ¬∑ 1L ¬∑ 0T`. The `‚ö°N close` segment still only appears when at least one category is close.
- Card value (`my score` / `vs opp`) is stacked on two lines (score block + "vs X" below) so decimal-heavy stats (OPS/ERA/WHIP) don't cause width or height inconsistency across the row.

**My Season Category Rankings** (section 14) subtitle shows a pseudo-single-week roto score: `sum(n - rank + 1 for rank in cats.values())` ‚ÄĒ same scale as a weekly roto score, max = n √ó 12. Directly comparable to the "This Week's Category Rankings" subtitle score.

**My Upcoming Starts subheader** format: `X starts across Y days | N this wk[, N next wk]`. The "this wk" count is red when 0. The ", N next wk" segment is omitted entirely when next-week count is 0.

**Probable starters:** The primary method uses two MLB API calls (range schedule ‚Üí batch hydrate). The +6-day rotation projection fills unannounced slots. A live-feed fallback exists if the batch returns nothing.

**Pitcher hot/cold uses 15-day ERA:** `build_pitcher_hot_cold_section`, My Upcoming Starts, and FA Starting Pitchers all compare season ERA vs 15-day ERA (from `p15` index ‚ÄĒ Dataset==15 rows). The 7-day window is too short for SPs who start infrequently. When a player is absent from the FP 15-day top 300 (fringe starters like Davis Martin), the code falls back to `rec_p` ‚ÄĒ the pybaseball Baseball Reference 15-day scrape stored in `recent_pitching`. `p15` and `rec_p` are both built in the main build function. Column header is "L15 ERA". `fetch_recent_pitcher_stats` fetches 15 days (not 7) to match this window.

**Statcast name matching:** `lf_to_name()` in fetch_data.py converts Baseball Savant "Last, First" names to "First Last" AND strips accents (e.g. `Ram√≠rez, Jos√©` ‚Üí `Jose Ramirez`) so the merge against FantasyPros ASCII names succeeds. Without accent stripping, accented-name players (Jose Ramirez, etc.) silently lose all Statcast data (xwOBA, Barrel%, SprintSpeed, HR_Probability).

**Roster merge name matching (`merge_on_name` / `_name_key`):** ESPN roster/FA names and FantasyPros names often differ by accents and generational suffixes ‚ÄĒ ESPN `Luis Garc√≠a Jr.` vs FantasyPros `Luis Garcia` ‚ÄĒ and an exact-string merge drops the roster link, so a rostered player wrongly shows as a **free agent**. `merge_on_name` (used for both hitter and pitcher roster+FA merges) does the exact merge first, then fills any still-unmatched rows via `_name_key` (accent-stripped, lowercased, trailing Jr./Sr./II/III/IV/V + punctuation removed). Two invariants: (a) exact matches always win ‚ÄĒ the fallback only fills NaNs, so it can add a match but never change or remove one; (b) a key is trusted only when it maps to a single player on **both** sides, so the several MLB "Luis Garcia" pitchers stay ambiguous and are never guessed. This generalizes the older per-player `HITTER_NAME_PATCHES`/`PITCHER_NAME_PATCHES` (still applied, still fine for odd cases). Takes effect only on a real data refresh, not `--no-refresh` previews. **`merge_on_name` is also wired into the Statcast/pybaseball merges** (probable starters, Savant pitcher contact/expected/skill, hitter contact/expected-stats/sprint, and the ESPN-status merge) ‚ÄĒ those previously stripped accents via `lf_to_name` but NOT suffixes, silently nulling a suffixed player's xwOBA/Barrel%/xERA/WhiffPctile. The fill loop uses scalar `.at` assignment so it also handles list-valued columns (`PSP_Dates`). The FantasyPros‚ÜĒFantasyPros merges (`fp7`, the season‚Üíall-rows enrich re-broadcast) stay exact ‚ÄĒ no suffix gap there. **Index-alignment invariant:** `fkeys` (the fp-side name keys) is built from `merged["PlayerName"]`, NOT `fp["PlayerName"]` ‚ÄĒ `fp.merge()` resets to a clean `RangeIndex`, so when the incoming `fp`/`season_df` carries a non-default index the boolean `missing` mask (from `merged`) and `fkeys` (from `fp`) become unalignable and pandas raises `IndexingError: Unalignable boolean Series`. Always derive the fill-loop keys from `merged` so their index matches the mask.

**Weekly matchup is Monday‚ÄďSunday:** `week_end_str` is computed as the Sunday of the current week (`today + timedelta(days=6 - today.weekday())`). FA SP sections and My Upcoming Starts show all starts including next week, but dates past Sunday get a `NEXT WK` badge. The KPI "Starts This Week" and Week at a Glance bullets 2 and 3 count/recommend only within the current matchup week ‚ÄĒ except on Sundays (see below). **Bullet 2 (rotation coverage) scopes both `confirmed` and `thin_days` to `PSP_Date <= week_end`** so its start count matches the "Starts This Week" KPI exactly ‚ÄĒ `my_starts_by_day` spans this week AND next (two-start rotations push starts into next week), and without the filter bullet 2 showed e.g. 10 while the KPI showed 8.

**Two-start pitchers (`PSP_Dates`):** `fetch_data.py` now preserves a list of ALL upcoming start dates per pitcher (`PSP_Dates`, plus parallel `PSP_HomeVAways`) via `_attach_start_lists` before the one-row-per-pitcher dedup. The scalar `PSP_Date`/`PSP_HomeVAway`/`PSP_Projected` remain the earliest start (unchanged for existing consumers). `_starts_this_week(r, today, week_end)` in send_digest counts entries within the matchup week (falls back to the scalar `PSP_Date` for old snapshots). A pitcher with ‚Č• 2 starts Mon‚ÄďSun gets a bold **purple** `2-START` chip (`two_start_badge()`, filled `PURPLE` bg / white text ‚ÄĒ purple so it's distinct from the green QS and yellow 5K+ chips, since 2-START + QS often render side by side) in FA SP + My Upcoming Starts, and is preferred (secondary sort key, NOT a score change) in the Week-at-a-Glance best-FA-SP bullet, which appends "√ó2 starts this week". Note: two-start weeks are only visible when both starts fall in the window ‚ÄĒ mid-week runs usually show 0 because the +6 rotation pushes the 2nd start into next week; the signal lights up on Mon/Tue runs. Never fold two-start into the 0‚Äď100 score (keeps scores normalized).

**Save-Role Watch (`save_role_watch`):** SVHD is the most volatile category. **Recent holds are NOT available anywhere in the pipeline** ‚ÄĒ per-window (`Dataset` 7/15/30) `SVHD` captures recent SAVES only (FantasyPros windows have no HLD), and ESPN exposes only season totals (no rolling 15-day split ‚ÄĒ verified: `pl.stats` has keys 0/98/99, none a usable window). So recency is save-only. The function flags (a) **emerging FA closers** ‚ÄĒ FA RP with ‚Č• 3 saves in the last 15 days ‚ÄĒ and (b) **fading rostered closers** ‚ÄĒ my RP gated on **season saves `ESPN_SV` ‚Č• 5** (a real closer) with 0 recent saves despite pitching (‚Č• 3 recent appearances). The fading side is gated on season *saves*, not SV+H, specifically so a holds-based reliever (e.g. JoJo Romero: 0 SV / 20 HLD ‚ÄĒ whose recent hold production we can't see) is never falsely flagged as fading. Rendered as a callout appended to the FA Relief Pitchers section.

**Category classification (`classify_categories`):** Returns `{cat: (proj_res, tier)}` reusing Category Pulse's projection math (`_project` + `pit_proj` for K/QS/W). Tier is only `tossup` (margin ‚Č§ `_CLOSE_THRESH` ‚ÄĒ a thin lead/deficit) or `leaning`. **There is no `locked` tier / no ūüĒí badge** ‚ÄĒ lock detection was removed because it mislabeled mid-week margins as clinched (a category could show "locked" and a flip arrow simultaneously). Now used solely to detect a THIN ERA/WHIP lead for the ratio-stat pickup warning. Computed once in `build_email` as `category_classification` and passed to `_roster_suggestion`; `build_category_pulse` no longer takes it, and the pickup steering no longer prunes "dead" categories (targets all losing cats).

**HR% column (`_hrp_cell`):** `HR_Probability` (computed in `fetch_data.compute_hr_probability` from barrel%, hard-hit%, launch angle, HR/AB, xwOBA, ISO, recent HR streak; range ‚Čą 0.05‚Äď0.31, a modeled per-game HR probability) is surfaced as a color-coded `HR%` column in Roster Hot/Cold and FA Hitters via `_hrp_cell(row)`, which also renders a hover `title` tooltip of the underlying drivers (Barrel% ¬∑ HardHit% ¬∑ EV ¬∑ xwOBA ¬∑ ISO). Green ‚Č• 20%, yellow ‚Č• 14%. `_hrp_cell` takes the full player row (needs the Statcast fields), so Roster Hot/Cold stashes the season row as `srow`. **`compute_hr_probability` measures power SKILL, not availability** ‚ÄĒ it must NOT gate on `ESPN_Status` (an earlier status gate returned 0.0 for any injured/recently-injured or "Unknown"-status player, which zeroed out Judge/Trout/Buxton despite intact power). It returns 0.0 only when there's no usable signal at all (no HR rate and no Statcast), so those show "‚ÄĒ". `ISO` is derived as `SLG ‚ąí AVG` in `fetch_data` (the FP feed omits it); `HR_per_AB`/`ISO`/`Barrel_Pct`/`HardHit_Pct`/`MaxEV`/`xwOBA` are in `enrich_cols` so all rows carry them for the tooltip.

**Hot/Cold Score columns:** Both `build_pitcher_hot_cold_section` and `build_hot_cold_section` take a `best_recent_*` index and render a role-aware Score badge (pitcher ‚Üí `_score_p`, hitter ‚Üí `_blend(hitter_score)`) ‚ÄĒ same normalized number shown everywhere else for that player.

**Roster KPI hot/cold counter (`hc_str`):** The "Roster" KPI tile counts my ENTIRE roster's hot/cold players ‚ÄĒ hitters AND pitchers ‚ÄĒ not just hitters. Hitters use 7-day OPS vs season (¬Ī0.015, same as `build_hot_cold_section`); pitchers use 15-day ERA vs season (¬Ī0.40, requires ‚Č•3 recent IP, same as `build_pitcher_hot_cold_section`, with `rec_p` fallback for fringe SPs). The two per-section thresholds differ by design (OPS scale vs ERA scale) ‚ÄĒ keep the KPI in sync with each section's threshold, not a single shared number. The tile label is "Roster" (whole team), not "Hitters".

**Ratio-stat risk guardrail:** In `_roster_suggestion`, when the chosen add is an SP (`_is_sp`) and ERA or WHIP is a currently-won **tossup** (thin lead, per `classify_categories`), and the candidate's ERA > 4.20 / WHIP > 1.30, the pickup bullet appends a yellow `‚ö† boosts K/W/QS but his {ERA} {cat} over ~{IP} IP risks your thin {cat} lead.` IP is `IP_per_G √ó _starts_this_week`, formatted via `_fmt_ip`. A good-ERA streamer, or a comfortable (non-tossup) lead, correctly produces no warning.

**Sunday mode (`is_sunday`):** When `datetime.now().weekday() == 6`, the digest shifts to a next-week preview. Changes: header subtitle ‚Üí "Weekly Lookahead"; email subject ‚Üí "Lookahead"; KPI tile ‚Üí "Starts Next Week" (counts starts after `week_end_str`); Category Pulse subtitle ‚Üí "Final stretch ‚ÄĒ week ends today"; Week at a Glance box ‚Üí "Next Week Preview" label; bullet 1 appends "‚ÄĒ final" instead of "through Day N"; bullet 2 shows next-week confirmed starts; bullet 3 shows best FA SP for next week. `next_week_end_str` is `today + timedelta(days=13 - today.weekday())` and is available in `build_email` scope.

**SP/RP role detection uses `_is_sp(r)`:** Never use `"SP" in pos` or `gs > 3` alone. The helper uses a priority chain: ESPN season GS/GP ratio (‚Č• 5 appearances) ‚Üí dataset GS/G ratio (‚Č• 4 appearances) ‚Üí IP/G ‚Üí Position field. Thresholds: GS/G ‚Č• 0.80 ‚Üí SP, ‚Č§ 0.20 ‚Üí RP; IP/G ‚Č• 4.5 ‚Üí SP, < 2.5 ‚Üí RP. All SP/RP-sensitive functions use it: `pitcher_score`, `_score_p`, `fa_starters`, `fa_relievers`, My RP filter, `positional_breakdown`.

**Tap-to-expand score breakdown v2 (`score_reveal` / `_hitter_score_breakdown` / `_pitcher_score_breakdown` / `_score_narrative`):** Tapping a Score badge reveals a **full-width row rendered directly below the player's row** (not a panel inside the narrow Score cell ‚ÄĒ that's the v1 look the user found cramped). The panel narrates the score's **2‚Äď3 most decisive drivers in prose** ‚ÄĒ e.g. `SP score 63. Carried by swing-and-miss (24% K) and limits baserunners (1.23 WHIP); no glaring holes. 15-day form 58 (cold) ‚Üí shown blends 65% season / 35% recent.` ‚ÄĒ not a `points/max` list. **The recent-form clause names the actual window** (`30-day`/`15-day`/`7-day`) from the recent row's `Dataset`, not a bare "Recent form" ‚ÄĒ the window is whichever slice `best_recent_*` resolved to (30 > 15 > 7 > pybaseball; hitter no-`Dataset` fallback ‚Üí 7-day, pitcher ‚Üí 15-day). This is deliberate so it doesn't read as contradicting the Hot/Cold őĒ column right beside it, which is a **different** window/metric (hitters: L7 OPS vs season; pitchers: L15 ERA vs season) ‚ÄĒ a bat can be ūüĒ• on L7 OPS yet "30-day form (cold)" on the broader composite.
- **Mechanism (no JS, email-safe):** `score_reveal(score, breakdown_html, uid, colspan)` returns a **tuple `(cell_html, row_html)`**. `cell_html` is the badge wrapped in an `<a href="#{uid}">` (with a ‚Ėĺ caret); `row_html` is a `<tr id="{uid}" class="scorebd-row" style="display:none;">` spanning `colspan` columns. The caller inserts `cell_html` into the Score `<td>` **and appends `row_html` immediately after the player's `</tr>`**. A single head-`<style>` rule `tr.scorebd-row:target { display:table-row !important; }` reveals the row when its badge is tapped (CSS `:target`). The `!important` beats the inline `display:none`. A `‚úē` link (`href="#{uid}x"` ‚Üí a non-existent anchor) un-targets ‚Üí closes. Only one row is open at a time (accordion) ‚ÄĒ tapping another badge closes the prior.
- **Scroll positioning (`scroll-margin-top`):** A fragment jump scrolls the target element to the **top** of the viewport. Because the breakdown `<tr>` sits *below* the player's row, an un-mitigated jump pushes the tapped player row (and the table header) off the top of the screen ‚ÄĒ the user has to scroll back up to see who they tapped. The `:target` rule therefore also sets `scroll-margin-top:40vh`, which tells the browser to leave a 40%-viewport gap *above* the anchor when it scrolls to it, so the tapped player row lands in the upper-middle of the screen and stays visible with the breakdown right below it. Tune the one `40vh` value on the `tr.scorebd-row:target` rule (higher = player sits lower / more context above; lower = closer to the top). Browser-attachment only (Gmail strips the `<style>`).
- **Why `:target` rows, not `<details>` (v1):** a `<details>` lives inside one `<td>` and can only expand within that narrow cell; there is no way to make its panel a full-width row below. A `:target`-toggled `<tr colspan>` is the only no-JS, email-safe way to get the full-width-below look. Trade-off: Gmail's inline body strips `<style>`, so there the rows stay **hidden** (badge still shows; the link is a harmless no-op) rather than always-visible as v1 degraded. The user reads the **browser-rendered attachment** (mobile Safari/Chrome), where `:target` works ‚ÄĒ so hidden-inline is the better degrade (no clutter, smaller inline body).
- **`_bd_uid(prefix, name)`** mints a globally-unique anchor id (`bd-{prefix}-{slug}-{counter}`) ‚ÄĒ a running `_BD_SEQ` counter guarantees uniqueness since a player can appear in several tables. Each table has its own prefix (`rhc`/`phc`/`mus`/`myrp`/`fasp`/`farp`/`fahit`/`posw`/`posfa`).
- **Narrative (`_score_narrative` + `_hit_clauses`/`_sp_clauses`/`_rp_clauses`):** each `_*_clauses` returns `(fill, strength_phrase, weakness_phrase)` per component, where `fill = comp_points / max` and the phrases embed the raw driver stat (e.g. `quality contact (xwOBA .360)` / `empty contact (xwOBA .272)`). `_score_narrative` names the ‚Č§2 strongest drivers (fill ‚Č• .60) and ‚Č§2 weakest (fill ‚Č§ .35): `Carried by ‚Ä¶ ; held back by ‚Ä¶`. Punt-saves-consistent: low SVHD / low HR% are **not** surfaced as weaknesses (`weakness_phrase=None`); SP `Role` (start volume) is omitted from the narrative entirely (a marker, not a skill). Clauses read raw stats off the row via `_n`, but the fills come from the **real** `_parts=True` component dict, so the prose can't disagree with the score.
- **Wired into all Score badges:** the 7 tables (Roster Hot/Cold, Pitcher Hot/Cold, My Upcoming Starts, My Relief Pitchers, FA SP, FA RP, FA Hitters) **plus Positional Breakdown** (both the weakest-my-player badge `posw` and the best-FA badge `posfa`, role-aware via `p["ptype"]`, `colspan=4`). No bare `badge()` renders a displayed score anymore ‚ÄĒ only `two_start_badge()` and `score_reveal`'s own fallback remain. **HR% drivers are surfaced in the expanded hitter panel** (`_hitter_score_breakdown`) as a trailing muted `<div>` line ‚ÄĒ `HR% NN% modeled per-game HR probability (Barrel ‚Ä¶ ¬∑ HardHit ‚Ä¶ ¬∑ EV ‚Ä¶ ¬∑ xwOBA ‚Ä¶ ¬∑ ISO ‚Ä¶)` ‚ÄĒ so touch users (no hover) see them; `_hrp_cell`'s `title=` hover keeps the same string via the shared `_hrp_driver_str(row)` helper (single source for both). The glossary section (`build_glossary_section`) keeps its own `<details>` groups (unrelated).

**Unified role scores ‚ÄĒ a player shows the SAME score in every section.** Three canonical role scores, all calibrated to the shared 0‚Äď100 scale (p50‚Üí50, p90‚Üí80): SP ‚Üí `_score_p` (blended `pitcher_score`), RP ‚Üí `rp_score` (never blended ‚ÄĒ built on ESPN season counting stats, and skipping the blend keeps the number identical across My RP, FA RP, and Positional Breakdown), Hitter ‚Üí `_blend(r, hitter_score, best_recent_h)`. Never score a section with a different function than the others use for the same role ‚ÄĒ that's how Ashby showed 72 in one table and 58 in another. `sp_fa_score` (pitcher_score + hidden start bonus) was removed for this reason; the FA SP Score column now equals the My Upcoming Starts badge for the same pitcher.

**Dynamic volume benchmarks (no hard-coded IP/AB/GS minimums):** "Full-time" / "big enough sample" thresholds are **derived from the live snapshot each run** so they scale as the season progresses instead of a fixed number that goes stale (a 225-AB "regular" line is right in July, absurd in September). Two builders, both called once at the top of `build_email`:
- `compute_ab_benchmarks(hitters)` ‚Üí module global `_AB_BENCH[window]` = `_AB_LEADER_FRAC` (0.62) √ó the window's p95 (leader) AB. Consumed by `_ab_opportunity_mult` in `hitter_score`. `_FULLTIME_AB` is a cold-start fallback only (early season / a window with < 20 hitters).
- `compute_pitcher_benchmarks(pitchers)` ‚Üí module global `_PIT_BENCH[(window, role)]` = leader IP/GS/GP (p95) per role, `_is_sp`-split. `_ip_reliability_mult` uses `_IP_RELY_FRAC` (0.20) √ó leader IP for the row's window+role as the small-sample floor inside `pitcher_score` (replaces the old flat `ip/20` ‚ÄĒ which wrongly penalized *recent-window* rows too, since nobody has 20 IP in a 15-day window; now window-relative so recent form is trusted). `_pit_viable_min(role, stat)` uses `_GS_VIABLE_FRAC`/`_GP_VIABLE_FRAC`/`_IP_VIABLE_FRAC` (0.17/0.30/0.38) √ó the season leader for the `positional_breakdown` viable-FA filter (replaces `GS‚Č•3` / `GP‚Č•12` / `IP‚Č•20`) and the recalibration population. `_PIT_FALLBACK` holds the old constants for cold start.
- All fractions were chosen so the derived floors ‚Čą the old hard-codes **today** (SP rely ‚Čą 20.4 IP, SP viable ‚Čą 3.06 GS, RP viable ‚Čą 12 GP / 19.8 IP), so the change is calibration-neutral now (`recalibrate_scores.py` prints the same constants) and only diverges as the season grows. p95 (not max) is the "leader" for outlier robustness.

**Data-derived league averages (`_LG` / `compute_league_averages`):** The "league-average OPS" concept was hard-coded three times with three different values (`_LEAGUE_AVG_OPS=0.717`, fetch_data `LG_OPS=0.720`, inline `0.730` in `qs_probability`), plus `qs_probability`'s ERA/WHIP/K%/IP-per-start/barrel anchors. `compute_league_averages(hitters, pitchers)` (called once in `build_email` next to the benchmark builders) writes the module global `_LG` with `ops` (full-time regulars), `team_ops` (mean opponent OPS faced), and starter `era`/`whip`/`k_pct`/`ip_per_start`/`barrel_allowed` from qualified YEAR rows; consumers read `_LG.get(key) or <old literal>`. **`qs_probability` stays calibrated because the intercept `38` and the multipliers are fixed** ‚ÄĒ a league-average starter scores ~38 regardless of the derived anchors, so only the reference point tracks the season (verified: qualified-SP mean QS ‚Čą 38). `_proj_line_html` opp_factor uses `_LG["team_ops"]`. fetch_data derives its own `LG_OPS` for wRC+ from the snapshot's full-time regulars (it's a fetch-time write). ONLY genuine league averages live in `_LG`; calibration/scaling constants (score-component spans/floors, park factor, recalibration constants, `IP*4.3`, `compute_hr_probability` weights) deliberately do NOT.

**FA "Cats" column (`_cats_cell` / `player_cat_strengths` / `build_cat_percentiles`):** FA Hitters and FA Relievers show a **Cats** column (before Score) listing up to 3 roto categories the player is strong in (percentile ‚Č• 0.70 within a qualified YEAR pool of that type; `_LOWER_BETTER` cats inverted). Categories in `need_cats` ‚ÄĒ my currently-losing (`result=="L"`) ‚ą™ tossup (`classify_categories`) cats ‚ÄĒ render in `ACCENT` (highlighted); others `MUTED`. `_FA_HIT_CATS = [R,HR,RBI,SB,OPS]`, `_FA_RP_CATS = [SVHD,K,W,ERA,WHIP]` (B_SO/QS omitted ‚ÄĒ not surfaced on FA rows). `_cat_value` prefers ESPN season counts for RP (SVHD/K/W). **`category_classification` is now computed early in `build_email` (right after `pit_proj`, before the FA tables) instead of in Final Assembly**, so `need_cats` is available to the FA tables; the old late computation was removed (single source).

**Week at a Glance pickup bullet (bullet 4) is hitter-only:** The add is **always** the best available FA **hitter** ‚ÄĒ `focus_pit` is hard-set to `False` in `_roster_suggestion` (was `len(losing_pit) >= len(losing_hit)`). Pitcher streaming is already covered by My Upcoming Starts / FA Starting Pitchers, so this bullet is dedicated to bats. `add_reason` targets our losing hitter cats when any, else reads `"bat depth"` (we may be losing only pitching cats). Consequence: the SP ratio-stat `ratio_warn` (gated on `_is_sp(add_candidate)`) now never fires here, and `add_type` is always `"hit"` so the drop prefers a hitter. `losing_pit` is left computed but unused (pairs with `losing_hit` for clarity). Shows positions for both the add and the drop (via `_pos_disp`, which hides the generic `P` tag). Drop selection is position-aware: weakest droppable player sharing a `POS_GROUPS` group with the add first (adding an OF drops the worst OF, not an infielder), then same player type (pit/hit), then any droppable. `_can_drop` still guards that every position keeps ‚Č• 1 healthy player.

**Never suggest dropping an IL-slot player (`ESPN_OnIL` / `_on_il`):** The league has **2 dedicated IL roster slots** that do NOT consume active/bench room, so dropping a player parked there frees nothing usable. `fetch_data.py`'s `get_pitcher_roster`/`get_hitter_roster` capture `ESPN_OnIL = (pl.lineupSlot == "IL")` and broadcast it to all dataset rows (pitchers: added to the `merge_on_name(fp, roster_df, [...])` column list; hitters: added to the `merge_on_name(fp, roster_df, [...])` at the top of `build_hitter_data`). Kept as a **native python bool** (NOT `.astype(bool)` ‚Üí numpy `bool_`, which `json.dump(..., default=str)` would stringify to the *truthy* string `"False"`); unmatched FP-only/FA rows fill `False`. In send_digest, `_on_il(r)` reads it (tolerant of a stringified `"true"`). Two drop paths exclude IL-slot players: (a) `_can_drop` returns `False` for them (Week-at-a-Glance bullet 4); (b) `positional_breakdown`'s `worst_player` skips them (`drop_pool = [r for r in my_p if not _on_il(r)] or my_p`, falling back only if *every* player at the position is on IL). **Note `ESPN_OnIL` is lineup-slot-specific, NOT injury status** ‚ÄĒ a DL player sitting on the BENCH (e.g. Will Smith when both IL slots are full) is `False` and stays droppable, because cutting him *would* free a bench spot. The flag is populated for all teams (each team's own IL occupants), but only my rows affect my drop suggestions.

**FA RP requires SVHD ‚Č• 1:** `fa_relievers` gates on `(_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))) >= 1`. A pitcher with zero saves and zero holds all season has no role and should not be recommended.

**Score cascade (`best_recent_p` / `best_recent_h`):** Built in `build_email` by merging `{**rec_p_fp, **p7, **p15, **p30}` (pitchers) and `{**rec_h, **h7, **h15, **h30}` (hitters) ‚ÄĒ later dicts win, so 30d FP > 15d FP > 7d FP > Baseball Ref. `rec_p_fp` is `recent_pitching` with computed `K/IP = K/IP` and `IP_per_G = IP/G` added so `pitcher_score` can use it. These dicts are passed to `_blend` and `positional_breakdown`. Coverage: ~500 pitchers / ~460 hitters vs 300 from 30d alone.

**`positional_breakdown` viable filter:** FA pool for each position excludes benchies. SP: `GS >= _pit_viable_min("SP","GS")`. RP: `ESPN_GP >= _pit_viable_min("RP","GP") or IP >= _pit_viable_min("RP","IP")` (dynamic, ‚Čą `GS‚Č•3` / `GP‚Č•12` / `IP‚Č•20` today ‚ÄĒ see "Dynamic volume benchmarks"). Hitters: `OPS > 0.200 or R+RBI > 5`. FA quality (`fa_quality`) = avg blended score of top-3 viable FAs. Scarcity thresholds: `< 50` ‚Üí scarce (RED), `< 60` ‚Üí moderate (YELLOW), `>= 60` ‚Üí deep (MUTED).

**Category Pulse `days_elapsed`:** `days_elapsed = datetime.now().weekday()` (Mon=0 ‚Ä¶ Sun=6). ESPN stats are through *yesterday*, so today is always remaining ‚ÄĒ do not add 1. Guard: `day_clause = f' through Day {days_elapsed}' if days_elapsed > 0 else ' (week starting)'`.

**Category Pulse pitcher projections (K, QS, W):** These three use actual remaining starts √ó per-start rate instead of historical weekly averages. Computed in `build_email` as `pit_proj = {"QS": {"my": ..., "opp": ...}, "K": ..., "W": ...}` from pitchers with `PSP_Date >= today and <= week_end_str`. Passed to `build_category_pulse(remaining_proj=pit_proj)`. Rate/hitter cats (ERA, WHIP, OPS, R, HR, RBI, SB, B_SO, SVHD) still use historical averages via `compute_weekly_avgs`.

**Dry-run preview filenames:** Always `previews/digest_preview_{team_slug}.html` (e.g. `digest_preview_Guerrero_Warfare.html`, `digest_preview_Giga_Vlad.html`). The old `digest_preview.html` fallback is gone ‚ÄĒ always slug-based regardless of whether `--team` is passed.

## Scoring functions (send_digest.py)

- `_is_sp(r)` ‚Üí bool. Usage-based SP/RP detection. Priority: ESPN season GS/GP ‚Üí dataset GS/G ‚Üí IP/G ‚Üí Position field. See gotcha above.
- `_blend(r, score_fn, idx_recent, w=None)` ‚Üí blended score. Default weight is the module constant **`_BLEND_W = 0.35`** (35% recent best-available window + 65% season) ‚ÄĒ a single source shared by the math and the tooltip text. Was 0.40 (60/40); moved 2026-07-02 to lean on the stable season signal since hot/cold streaks are surfaced separately in the Hot/Cold sections. `idx_recent` is `best_recent_p` or `best_recent_h` (see below). Falls back to `score_fn(r)` if player has no recent row. Applies to hitters and SPs; RP `rp_score` is never blended. **No recalibration needed when changing `_BLEND_W`** ‚ÄĒ the blend is a post-calibration weighted average of two already-calibrated 0‚Äď100 scores.
- `hitter_score(r, _parts=‚Ä¶)` / `pitcher_score(r, _raw=‚Ä¶, _parts=‚Ä¶)` / `rp_score(r, _raw=‚Ä¶, _parts=‚Ä¶)` ‚Üí all three take **`_parts=True`** to return `(components_dict, multiplier)` ‚ÄĒ the raw pre-multiplier component contributions (`{label: points}`) and the playing-time / IP-reliability multiplier ‚ÄĒ instead of the final 0‚Äď100 score. This is the single source of truth for the tap-to-expand Score breakdown (see "Tap-to-expand score breakdown"): the tooltip re-derives nothing, it reads the same components the score is summed from. Component insertion order == display order; the refactor is byte-identical to the old inline `s += ‚Ä¶` (verified ‚ÄĒ the six-hitter regression 45/54/50/61/74/74 is unchanged).
- `_score_p(r, idx_recent=None)` ‚Üí canonical role-aware pitcher score. SP ‚Üí `_blend(r, pitcher_score, idx_recent)`; RP ‚Üí `rp_score(r)` unblended. Used by every pitcher Score display/sort: FA SP (`fa_starters`), My Upcoming Starts badge, `positional_breakdown`, and Week at a Glance add/drop pools. See "Unified role scores" gotcha.
- `_starts_this_week(r, today, week_end)` ‚Üí int. Count of the pitcher's upcoming starts within the matchup week (from `PSP_Dates`; falls back to scalar `PSP_Date`). Drives the `2-START` badge and best-FA-SP preference. See "Two-start pitchers" gotcha.
- `save_role_watch(pitchers, my_team, claimed)` ‚Üí `(emerging, fading)` lists for the Save-Role Watch callout. See gotcha.
- `classify_categories(matchup, weekly_avgs, days_elapsed, remaining_proj)` ‚Üí `{cat: (proj_res, tier)}`. Powers ūüĒí lock badges + pickup steering. See gotcha.
- `opponent_week_intel(pitchers, hitters, opp_team, best_recent_h, today, week_end)` ‚Üí dict (starts, two-start pitchers, hot hitters) for the Opponent This Week block. Returns None when `opp_team` is empty.
- `pitcher_score(r, _raw=False)` ‚Üí 0‚Äď100. Role-aware via `_is_sp(r)`. **Blended advanced/results scoring** (added 2026-07-02). K component (28): results-based K% (`Kpct_P`, else K/IP) **blended 60/40** with `WhiffPctile` (a Baseball Savant league whiff PERCENTILE 0‚Äď100, not a rate) when present. Run-prevention (28): actual `ERA` **blended 55/45** with `xERA` (Savant deserved-ERA, absolute) when both present. WHIP (20): results only. **Contact-quality-allowed (0‚Äď12, NEW)**: `BarrelPctAllowed` (0‚Äď5, lower better) + `xwOBA_against` (0‚Äď7, ~.360‚Üí0 scale). **SP path**: role bonus 9‚Äď12 based on GS volume; SVHD ignored. **RP path**: role bonus 5‚Äď12 scaled by SVHD + W + IP/G (note: `_score_p` routes all RPs to `rp_score`, so this RP branch is effectively never displayed ‚ÄĒ `pitcher_score` is calibrated on the SP distribution only). All advanced blends **fall back to the raw stat** when the Savant field is missing (`_n` floors the `-1` sentinel to 0). **Small-sample penalty**: `s *= min(1.0, ip / 20)` before calibration. Calibrated to p50=50, p90=80: `s * 1.4341 - 39.957`. `_raw=True` returns the pre-calibration score (used by `recalibrate_scores.py`).
- `rp_score(r, _raw=False)` ‚Üí 0‚Äď100 composite for RP ranking. **Punt-saves weighting (2026-07-02):** SVHD is deliberately DE-EMPHASIZED to ~15% of the raw score (below an equal 5-cat share) because saves are the most volatile category and one we're willing to sacrifice ‚ÄĒ skill/ratio cats carry the weight. Raw maxes: SVHD (15) ¬∑ K (26) ¬∑ W (15) ¬∑ IP/G (8), from `ESPN_SVHD`/`ESPN_K`/`ESPN_W` with FantasyPros fallback; run-prevention (16): `ERA` **blended 50/50** with `xERA`; WHIP (12); **contact-quality-allowed (0‚Äď8)**: `BarrelPctAllowed` (0‚Äď4) + `WhiffPctile` (0‚Äď4). Advanced blends fall back to raw when missing. Calibrated: `s * 1.9619 - 43.0286` (p50‚Üí50, p90‚Üí80). Used by FA RP, My Relief Pitchers, and (via `_score_p`) every RP-scoring section. My Relief Pitchers picks the best available dataset per player (YEAR ‚Üí 30 ‚Üí 15 ‚Üí 7). `_raw=True` returns the pre-calibration score. **When the SVHD-vs-skill balance changes, rerun `recalibrate_scores.py`.**
- **Recalibration:** when the raw component mix of `pitcher_score`/`rp_score` changes, rerun `python recalibrate_scores.py` (reads the snapshot, computes raw distributions via `_raw=True`, prints new `p50‚Üí50 / p90‚Üí80` constants) and paste the constants back. Qualified populations are now **dynamic/role-relative** (see "Dynamic volume benchmarks" below), not fixed IP/GP ‚ÄĒ recalibrate imports `compute_pitcher_benchmarks` + `_pit_viable_min` so its population matches send_digest's. Because the benchmarks scale with the season, the qualified population drifts over time; rerun periodically (not just on a component-mix change) to keep p50/p90 honest.
- `hitter_score(r)` ‚Üí 0‚Äď100. Prefers wRC+ over OPS. Uses xwOBA, sprint speed, Barrel%, ISO, HR_Probability. **Opportunity multiplier** (`_ab_opportunity_mult`): the rate components would score a part-time masher like a regular, but over a week a bench bat who gets ~1 AB every few games can't accumulate counting stats ‚ÄĒ so the raw score is scaled by AB vs a full-time benchmark (floored at `_AB_FLOOR = 0.40`, capped at 1.0). A full-time hitter lands at 1.0 (no penalty, calibration anchors untouched). Calibrated: `s * 1.587 - 5.2`. Displayed everywhere as `_blend(r, hitter_score, best_recent_h)` ‚ÄĒ `fa_hitters` takes `idx_recent` for this.
- `qs_probability(r)` ‚Üí 1‚Äď99. Calibrated to league-average ~38%, ace ~75%. Uses IP/G (not IP/GS).
- `_fmt_ip(ip_decimal)` ‚Üí baseball IP string. Converts true decimal (5.333) to notation (5.1). Formula: `whole = int(d); outs = round((d-whole)*3); if outs>=3: whole+=1, outs=0`. Used in Proj. Line display for both FA SP and My Upcoming Starts.
- `_proj_line_html(r)` ‚Üí `IP ¬∑ ER ¬∑ K` span. ER is adjusted for opponent strength and park: `raw_er * opp_factor * park_factor`. `opp_factor = clamp(opp_ops / (_LG["team_ops"] or 0.717), 0.80, 1.20)`. `park_factor = 0.97` if `PSP_HomeVAway` starts with `"vs "` (home), `1.03` if `"@ "` (away), else `1.0`. **K IS now opponent-adjusted**: `k = round(kip * ip_g * k_factor)` where `k_factor = clamp(opp_k / (_LG["team_k"] or 0.22), 0.85, 1.15)` ‚ÄĒ a whiff-prone lineup (high team K%) inflates a starter's Ks, a contact lineup suppresses them. Clamped tighter than ER (¬Ī15% vs ¬Ī20%) since team K% varies less than team OPS; falls back to 1.0 when `Team_K_Value` is missing. `Team_K_Value` (opponent team K per PA) is fetched in `get_opponent_ops` from the **same** MLB Stats API call as `Team_OPS_Value` (`strikeOuts / plateAppearances`) and merged onto pitcher rows the same way. IP is the pitcher's per-start average (`IP_per_G`). All fields are already on the pitcher row.
- `hot_cold_cell(season_val, recent_val, ..., no_data_title=None)` ‚Üí `<td>` with colored recent stat + ūüĒ•/‚ÜĎ/‚ĚĄ/‚Üď icon vs season baseline. When recent_val is missing/zero and `no_data_title` is set, renders `‚ÄĒ` with a dotted underline and hover tooltip explaining the absence.
- `band_divider(label, color)` ‚Üí full-width `<div>` with centered label between `BORDER` lines. Used at band boundaries in final assembly.

Falls back to `current_matchup` (Guerrero Warfare only) for old snapshots. `build_matchup_section` accepts `my_team` param (default `MY_TEAM` constant) so it renders the correct team name and logo.

**My Upcoming Starts + FA SP badges (`2-START` / QS / 5K+):** `2-START` (purple, when `_starts_this_week ‚Č• 2`), QS (green) and 5K+ (yellow) badges next to the pitcher name. **QS and 5K+ purely annotate the projected game line the reader sees** ‚ÄĒ they are driven ONLY by `_proj_line_vals(r)` (the same numeric `(ip, er, k)` the `Proj. Line` cell renders), NOT by season rates: QS fires when the projected line is a quality start (`_proj_is_qs` ‚Üí 6+ displayed IP & ‚Č§3 ER); 5K+ fires when projected `K ‚Č• 5`. This is deliberate ‚ÄĒ an earlier version also fired on season rate (`K/IP ‚Č• 0.90 or K% ‚Č• 24%` / `qs_probability ‚Č• 51%`), which produced confusing contradictions like a `5K+` badge next to a `4 K` proj line (a pitcher with a 0.90 K/IP but low IP/G and a contact-heavy opponent). Now the badge can never disagree with the line. The **QS% column** still shows the season quality-start probability separately for nuance. `_proj_is_qs` applies the same third-of-an-inning `_fmt_ip` rounding as the display (5.84 IP/G ‚Üí "6.0" counts; 5.5 ‚Üí "5.2" doesn't), so the QS badge matches the shown IP exactly. Both tables use the identical rule, so the same pitcher shows the same badges in both.

**FA Starting Pitchers badges are unconditional** (as of 2026-07-03). The QS/5K+ badges ‚ÄĒ and the green/yellow name-border stripe ‚ÄĒ used to render only on **thin rotation days** (`thin_day = my_count < 2 and date_str <= week_end_str`) or next week, so a strong streamer showed a great Proj. Line but no badge on a day your rotation was already full. That gate is removed: FA SP badges now fire on the pitcher's projected line wherever he appears. (`thin_day` local was deleted; `thin_days`/`my_starts_by_day` still drive the ‚öĎ per-day "N my starts" banner and Week-at-a-Glance bullet 2.)

## Automation

- **GitHub Actions:** `.github/workflows/daily-digest.yml` triggers at 06:00 and 15:00 UTC (2 AM / 11 AM EDT). GitHub's scheduler is unreliable ‚ÄĒ actual delays vary 1‚Äď4 hours, so expected delivery is roughly 4‚Äď6 AM / 1‚Äď3 PM EDT. **Cron is always UTC** ‚ÄĒ no GitHub account or org timezone setting affects it. ESPN credentials are stored as repo secrets (`ESPN_SWID`, `ESPN_S2`).
- **Fetch-time freshness badge:** The record and category standings are a **point-in-time snapshot of ESPN's live box scores at fetch time**, not a cached number ‚ÄĒ ESPN keeps settling stats (late West-Coast finals, corrections) for hours after a fetch, so a morning digest can legitimately read e.g. 10-2 while ESPN later shows 11-1. The header badge therefore shows the fetch **time in ET** (`_fmt_refresh_time`), not just the date, so the lag is legible: "‚úď data as of today at 6:32 AM ET" (fresh) or "‚ö† data from Jul 2 at 8:01 PM ET ‚ÄĒ run a refresh‚Ä¶" (stale). `fetch_data.py` writes `refreshed_at` as a **tz-aware UTC** timestamp (`datetime.now(timezone.utc)`) so the ET conversion is correct whether the run was on CI (UTC) or a manual local box; `_fmt_refresh_time` converts tz-aware stamps to `America/New_York` and shows naive (pre-change) stamps as-is. `_data_fresh` still keys off the date only (`refreshed[:10] == today`), so "as of today" can still be hours behind ‚ÄĒ that's the whole point of surfacing the clock.
- **Local runner:** `scripts/run_digest.bat` can be used for manual local runs. It captures full console output (incl. tracebacks) to `logs/run_console.log`; the structured one-line-per-send record is written separately by `send_digest.py` to `logs/digest.log`. They are deliberately kept in **separate files** ‚ÄĒ the old setup redirected the `.bat`'s stdout into `digest.log` while Python also appended to it, and the two handles collided (`PermissionError` on Windows). The Python write is now wrapped so a locked log can never crash a run that already sent.
