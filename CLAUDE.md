## Commands

```bash

python send_digest.py --no-refresh             # use existing snapshot (fast вЂ” email-only changes)
python send_digest.py --dry-run                # save HTML to previews/, no email

python fetch_data.py                           # refresh data only в†’ data/snapshot.json

python dashboard.py --email                     # also email the dashboard to yourself as an attachment (open in a device browser)
```

## Architecture

Three files, one intermediate artifact. The daily digest and Monday recap are independent scripts that both read the same snapshot:

**`fetch_data.py`** pulls from 5+ sources в†’ `data/snapshot.json`:
1. FantasyPros HTML (`pd.read_html`) вЂ” pitcher/hitter stats across 4 ranges (7/15/30/season)
2. ESPN Fantasy API (`espn_api`) вЂ” rosters, FA list, roto box scores, standings, transactions
3. MLB Stats API вЂ” probable starters (batch hydrate) + opponent OPS
4. pybaseball вЂ” Statcast contact quality, expected stats, sprint speed, recent game logs

**`send_digest.py`** reads the snapshot, computes all derived metrics, builds one self-contained HTML email via Gmail SMTP. Two parts: inline HTML body (Gmail may clip at 102 KB) + an attached `digest_YYYY-MM-DD.html` for full render. All new features go here.

**`dashboard.py`** reads the same snapshot and builds a **single-viewport, zero-scroll "command dashboard"** вЂ” `previews/dashboard_{team_slug}.html`, sized for a **1440Г—900 laptop with NO scrolling** and even, glance-able coverage of every topic the digest covers. Unlike `weekly_recap.py`, it **imports `send_digest` and reuses its scoring/projection/aggregation functions verbatim** (so every number matches the digest вЂ” honors the "same score in every section" rule). `build_context(snap, my_team)` mirrors the derived-value prelude of `build_email` (`send_digest.py` ~3572вЂ“3777): it runs the same 4-call scoring setup (`compute_ab_benchmarks` в†’ `compute_pitcher_benchmarks` в†’ `compute_score_calibration` в†’ `compute_league_averages`, ORDER MATTERS вЂ” the module globals they set drive scoring), then rebuilds `best_recent_p/h`, FA lists, `weekly_avgs/std`, `pit_proj`, `classify_categories`, `positional_breakdown`, `opponent_week_intel`, `save_role_watch`, `_roster_suggestion`, `make_sparkline`, etc. Layout: a top KPI bar (8 chips) + a fixed 3-column CSS grid (`#wrap{position:fixed;inset:0;display:flex}`, `#grid{display:grid;grid-template-columns:1fr 1fr 1fr;grid-template-rows:minmax(0,1fr)}`, `.col{height:100%}`) вЂ” tiles `overflow:hidden` and clip to the top-N rows that fit (never scroll). **The `position:fixed;inset:0` on `#wrap` is load-bearing** вЂ” plain `height:100vh` + `flex:1` on the grid collapsed the grid to ~half height. **Column layout (content-balanced):** col 1 = Category Pulse (4Г—3 compact 12-cat grid; re-stacks if the tile ever gets too narrow); col 3 = Recommended Moves + Save-Role В· FA Radar (top **2** per role) В· Season (standings/roto/`make_sparkline`/weekly-finishes strip). **Per-tile `flex` weights are tuned so content-heavy tiles get more height** вЂ” don't equalize them blindly. **Sizing gotcha вЂ” design for a REAL viewport, not 900px:** browser chrome (tabs/URL bar) shaves usable height to ~768вЂ“820, so a tile tuned to exactly fill 900 clips its bottom rows there. Keep the vertical rhythm compact (small wrap/grid gaps + tile padding + `line-height` в‰€ 1.4) so everything survives ~768 usable; **verify with a headless screenshot at 1440Г—768 (not just 900)**. Two alignment fixes worth keeping: `make_sparkline`'s fixed-width SVG is stretched to full tile width via an injected `viewBox` + `preserveAspectRatio=none` (`_stretch_spark`) so it lines up with the Weekly Finishes pills below; and the Weekly Finishes strip plots the SAME completed-week set as the sparkline (weeks `< current_week`), NOT a last-N slice, so the two counts match. **Responsive (small screens):** the fixed no-scroll pane is a wide-screen idea, so media queries relax it below 1100px WITHOUT touching the desktop layout. The inline styles that pin the desktop grid (`position:fixed`, tile `flex`/`overflow:hidden`, the pulse grid columns) are overridden via **classes + `!important`** (`.tile`, `.tile-body`, `.pulse-grid`, `.topbar`, `.topbar-chips`, `.chip`) вЂ” inline styles otherwise win. **Tablet/phone use a SEPARATE layout container, not a reflow of the desktop grid.** `build_dashboard` renders each of the 8 tiles ONCE into a variable (safe to reuse вЂ” tiles carry no ids/anchors, `make_sparkline` has no SVG ids) and drops the identical markup into two containers inside `#wrap`: `#grid` (desktop 3-col: Pulse+Weakest / PitchingВ·HittingВ·Opponent / MovesВ·FAВ·Season) and `#gridt` (tablet 2-col: two `.colt` **independent-packing flex columns**). Recommended Moves + Save-Role Watch are skipped (prose bullets / plain name strings, no player row to source `Team`). **Browser tab title** is `Dashboard вЂ” {my_team}` (type first, so the tab is identifiable at a glance; the digest mirrors this with `Daily Digest вЂ” {my_team}`). No glossary. **Wired into GitHub Actions** (`daily-dashboard.yml`, 06:00 UTC / 2 AM EDT fire в†’ arrives ~8 AM ET, same cron as the digest в†’ `python dashboard.py --refresh --email`) вЂ” no longer a local-only artifact; still also runnable locally.

**`weekly_recap.py`** reads the same snapshot on Mondays, builds a full-league recap HTML email: Week N Highlights В· My Matchup (full 12-cat table) В· Lineup Efficiency (my start/sit opportunity cost вЂ” see `build_lineup_efficiency`) В· League Scoreboard (all 6 matchups) В· Weekly Roto Rankings ("MATCHUP PERFORMANCE" band, this matchup only) В· Top Performers (rostered + hot FAs, hitters/pitchers rendered side-by-side via `_two_col`) В· Standings & Luck В· Season Trajectory В· **Season Roto Rankings** (`build_season_roto_rankings` вЂ” same heat-mapped 12-cat grid as the per-matchup version but aggregated over EVERY matchup: teams **ranked** by cumulative `Roto_Score` (summed weekly roto points), while the **displayed** category value is the true season-to-date stat from ESPN's cumulative `mTeam` view (snapshot `season_cat_totals`), NOT a sum/average of weekly values. Ranking and value are independent by design вЂ” points reflect who won each category week by week; the value is ESPN's innings/AB-weighted season stat, which for rate cats {OPS,ERA,WHIP} can't be recovered by averaging weekly ERAs. Does NOT import from `send_digest.py` вЂ” copies the ~100 lines of constants/helpers it needs. Output: `previews/recap_week_N.html`. **`build_commissioner_story`** (`weekly_recap.py`) вЂ” Week N Highlights section at the top of the recap. Two-column layout: commissioner-style prose (left, 60%) + compact stat sidebar (right, 40%). Finds (a) weekly roto winner (top roto score that week), (b) hitter of week (rostered, best OPS, min 10 AB from `prev_week_hitting`), (c) pitcher of week (rostered, best ERA, min 8 IP from `prev_week_pitching`), (d) best available FA hitter. `prev_week_hitting`/`prev_week_pitching` are snapshot fields fetched over the exact previous matchup MonвЂ“Sun window вЂ” distinct from the rolling `recent_hitting` (7-day) and `recent_pitching` (15-day) fields (kept only as a fallback). **`build_top_performers` also runs on `prev_week_hitting`/`prev_week_pitching`** so its timeline matches the rest of the recap (MonвЂ“Sun matchup window), NOT the rolling windows; its subtitle shows the actual week dates, and its rostered/FA pitcher table shows **K** (not G). Both `build_commissioner_story` and `build_top_performers` run names through `_fix_mojibake` before the season-roster lookup вЂ” `prev_week_*` names arrive mojibaked (e.g. `Eury PГѓВ©rez`), so without the fix the accent-stripped `_name_key` fallback misses the roster row and the player is wrongly shown as a free agent. Prose uses named historical benchmarks (Bonds 1.422 OPS 2004, Gibson 1.12 ERA 1968, deGrom 1.70 ERA 2018, Ryan 383 K 1973, etc.). Sidebar cards show fantasy logo + MLB team logo (`_mlb_logo`, ESPN CDN) + slash line / K / QS / categories led. Roto weekly win count uses weekly top-roto-score count, NOT H2H W-L. **`build_league_scoreboard`** (`weekly_recap.py`) вЂ” all 6 matchups, one card each, 12-cat grid. Each team's identity (logo + name + WвЂ“LвЂ“T) lives in its **row label** (first column), NOT a top banner вЂ” the row IS the team, columns are categories. Every card is a **separate `<table>`**, so to keep the labelв†’category axis lined up card-to-card the tables use `table-layout:fixed` + a `<colgroup>` (150px label col + 12 equal cat cols); without this each table sizes its first column to its own team-name length and the axis drifts. Long names ellipsis-clip. Category headers are **neutral** (`MUTED`), not green/red вЂ” win/loss coloring only makes sense from one team's POV, and this is an even-handed full-league view. The **outlined value** (`outline:1px solid`) marks each category's winner instead. `_row` renders both teams (top = `my_team`/`my_val`, bottom = `opp_team`/`opp_val`).

**`get_lineup_efficiency(league, my_team_name, mode)`** (`fetch_data.py`) вЂ” reconstructs MY team's **daily** lineup to quantify start/sit opportunity cost. `mode="prev"` audits the last completed matchup week (MonвЂ“Sun, for the recap post-mortem); `mode="current"` audits the in-progress week Monв†’**yesterday** (today is incomplete, excluded; for the daily digest so misses are still fixable). Returns `{}` for the current week on a Monday (no completed days). Both are computed in `main()` and stored as snapshot fields `lineup_efficiency` (prev) + `lineup_efficiency_current` (current). **Mechanism:** categories-league `box_scores()` exposes team totals only вЂ” no per-player lineup. The per-day slot comes from `league.espn_request.league_get(params={"view":"mRoster","scoringPeriodId":<day>})`, which returns each roster entry's `lineupSlotId` **as set that day** (ESPN preserves historical daily lineups вЂ” verified: a player's slot genuinely varies day to day) + that day's actual stat split (`statSourceId==0`, matching `scoringPeriodId`). ~7 `mRoster` calls/week + one `mSettings`. **Two findings:** (a) **batter bench leakage** вЂ” a hitter's R/HR/RBI/SB put up in a `BE` slot (slot id 16), reported **NET of the weakest startable bat you'd have benched to play him** вЂ” this is the opportunity-cost correction: gross bench production overstates the miss because a full lineup means starting him displaces someone. Net uses only his *notable* days (HR/SB/2+R/2+RBI); quiet bench days have nothing to recover. Whether he *could* have been slotted without benching anyone (open slot **or** legal reshuffle) is a **max-bipartite-matching feasibility** check (`_full_match`) over the day's active hitters + candidate against `lineupSlotCounts`-expanded hitter-slot instances, using each player's `eligibleSlots`. If feasible в†’ displaced = nobody (free). Else в†’ displaced = min-value (`TB+BB+SB`) active hitter sharing an eligible slot. (b) **pitcher blowups** вЂ” an active-slot start with 5+ ER (or 4+ ER in <3 IP) that counted toward ERA/WHIP, cross-referenced against `recent_activity(size=150)` drops to flag "imploded then dropped" (damage banked). **OPS is never netted** (rate stat вЂ” benching a 0-4 day *helps* OPS, so it's ambiguous). Pitcher/hitter split via `eligibleSlots` (pitcher = has {13,14,15} and no hitter slot). Wrapped in a broad try/except в†’ `{}` on any failure (never breaks the fetch). Standalone/opponent + head-to-head version: **`bench_leakage.py`** (`python bench_leakage.py`) вЂ” imports `fetch_data`, audits my team + last week's opponent, prints to console.

**`build_lineup_efficiency`** (`weekly_recap.py`) renders `lineup_efficiency` (prev week) as a **LINEUP EFFICIENCY** band right after My Matchup: red "LEFT ON THE BENCH" net headline, per-player bench cards with each notable day + its swap target (`[vs Name H-AB]` or `[open slot]`), then active-slot blowups with the drop flag. **`build_bench_watch`** (`send_digest.py`) renders `lineup_efficiency_current` (week-to-date) as a compact **Lineup Watch** callout in the digest's MY ROSTER band (section 1b, right after `alert_section`) вЂ” net-recoverable batter line + implode-then-drop pitchers only; silent on a clean week. Gated on `not override_team` (snapshot stores only MY daily lineup, so the `--team` view omits it). Glossary entry added under "Projections & matchup".

**`data/snapshot.json`** is the schema contract between the two files. ~1.2 MB, not committed. Numeric missing values are stored as `-1` (not `NaN`) after the merge pipelines run.

## Critical gotchas

### Data sources
- FanGraphs returns 403 вЂ” never use directly. pybaseball works (handles headers).
- `pitching_stats()` (FanGraphs leaderboard) в†’ 403. Use `pitching_stats_range()` (Baseball Reference) вЂ” but it has no `HLD` column.
- SVHD comes from ESPN via `get_pitcher_espn_svhd()`, reading `pl.stats[0]['breakdown']`, which uses **string keys** (`'SV'`, `'HLD'`, `'SVHD'`, `'K'`, `'W'`, `'OUTS'`, `'ERA'`, `'WHIP'`, `'GP'`, `'GS'`) вЂ” not numeric IDs. Called at fetch time for all rostered + FA pitchers.
- ESPN season stats (`ESPN_SV`/`ESPN_K`/`ESPN_W`/`ESPN_IP`/`ESPN_GS`/`ESPN_GP`/`ESPN_SVHD`) are stored on **all** dataset rows so send_digest can use season counts for players who only appear in short-range FantasyPros datasets. `ESPN_SVHD`/`ESPN_SV`/`ESPN_HLD` override `SVHD`/`SV`/`HLD` on `Dataset==YEAR` rows; `ESPN_HLD` is then dropped but `ESPN_SV` stays on all rows (the only way `save_role_watch` distinguishes a real closer from a holds-only reliever for players outside the FP top-300). Use `_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))` (not `if >= 0`) вЂ” `_n` floors negatives to 0 so `>= 0` is always true.
- xFIP / CSW are unavailable from FantasyPros (and FanGraphs вЂ” 403). But Baseball Savant (via pybaseball) supplies predictive pitcher stats, merged on `PlayerName`: `xERA` + `xwOBA_against` (`get_savant_pitcher_expected`, absolute values), `WhiffPctile` (a league whiff **percentile** 0вЂ“100, not a rate, from `get_savant_pitcher_skill`), and `BarrelPctAllowed`/`HardHitPctAllowed`/`AvgEVAllowed` (`get_savant_pitcher_contact`). Coverage в‰€ 90вЂ“99% of the FP top-300; missing rows fall back to raw ERA/K% cleanly.
- **`WhiffPct` (raw overall whiff %, DISPLAY-ONLY)** вЂ” via `get_savant_pitcher_whiff` (pybaseball `statcast_pitcher_arsenal_stats`), pitch-type rows aggregated by pitches-weighting per `player_id`. A raw 0вЂ“100 **rate**, DISTINCT from `WhiffPctile` (a 0вЂ“100 **percentile**). Merged on `PlayerName` via `merge_on_name`. **Never feed it into `pitcher_score`/`rp_score`** вЂ” `WhiffPctile` already drives the K component, so raw whiff% would double-count. Displayed only: a muted `whiff NN%` subline under the K% cell in the compacted My Upcoming Starts + FA SP tables (`_whiff_sub`, keeps them at 8 cols), and a real **Whiff%** column (green в‰Ґ 30) in Pitcher Hot/Cold (7 cols; `score_reveal` colspan 7).
- ESPN injury statuses are `TEN_DAY_DL`/`FIFTEEN_DAY_DL`/`SIXTY_DAY_DL` вЂ” not `IL`/`OUT`. Constant `_DL_STATUSES` covers all. FA views and positional breakdown exclude all DL-status players.

### Names & merges
- **Team name double-space:** `MY_TEAM_NAME = "Guerrero  Warfare"` (double space) in fetch_data.py matches ESPN exactly. `MY_TEAM = "Guerrero Warfare"` (single space) in send_digest is for display. Never normalize these to match each other.
- **Merge direction:** pitcher/hitter merges start from FantasyPros (left side). Players outside the FP top 300 drop from short-range (7-day) views but appear in longer ranges. FantasyProsв†”FantasyPros merges (`fp7`, seasonв†’all-rows enrich) stay exact.
- **`merge_on_name` / `_name_key`:** ESPN and FantasyPros names differ by accents and generational suffixes (`Luis GarcГ­a Jr.` vs `Luis Garcia`); an exact-string merge drops the roster link so a rostered player wrongly shows as a free agent. `merge_on_name` does the exact merge first, then fills still-unmatched rows via `_name_key` (accent-stripped, lowercased, trailing Jr./Sr./IIвЂ“V + punctuation removed). Invariants: (a) exact matches always win вЂ” the fallback only fills NaNs; (b) a key is trusted only when it maps to a single player on **both** sides (the several MLB "Luis Garcia" pitchers stay ambiguous, never guessed). Wired into both roster+FA merges AND the Statcast/pybaseball merges. Older per-player `HITTER_NAME_PATCHES`/`PITCHER_NAME_PATCHES` still applied.
- **Index-alignment invariant:** build the fill-loop keys `fkeys` from `merged["PlayerName"]`, NOT `fp["PlayerName"]` вЂ” `fp.merge()` resets to a clean RangeIndex, so keys from `fp` can be unalignable with the `missing` mask from `merged` в†’ `IndexingError: Unalignable boolean Series`.
- **Statcast name matching:** `lf_to_name()` converts Baseball Savant "Last, First" в†’ "First Last" AND strips accents (`RamГ­rez, JosГ©` в†’ `Jose Ramirez`) so the merge against ASCII FantasyPros names succeeds. Without it, accented-name players silently lose all Statcast data.

### Fields & sentinels
- **PSP sentinel:** `PSP_Date = "1999-01-01"` = no upcoming start. `PSP_Projected = True` = start projected via the +6-day rotation rule, not confirmed by the MLB API.
- **Two-start pitchers (`PSP_Dates`):** fetch_data preserves a list of ALL upcoming start dates per pitcher (`PSP_Dates` + parallel `PSP_HomeVAways`) via `_attach_start_lists` before the one-row-per-pitcher dedup. Scalar `PSP_Date`/`PSP_HomeVAway`/`PSP_Projected` remain the earliest start. `_starts_this_week(r, today, week_end)` counts entries within the matchup week. в‰Ґ 2 starts MonвЂ“Sun в†’ bold **purple** `2-START` chip (`two_start_badge()`) in FA SP + My Upcoming Starts, and preferred (secondary sort key, NOT a score change) in the Week-at-a-Glance best-FA-SP bullet ("Г—2 starts this week"). Never fold two-start into the 0вЂ“100 score.
- **B_SO is lower-is-better:** batter strikeouts sit in `_LOWER_BETTER` alongside ERA/WHIP. Affects Category Pulse bar direction and projection flip logic вЂ” fewer B_SO than the opponent is a win.
- **`ESPN_OnIL`** (native python bool = `pl.lineupSlot == "IL" or pl.injured`) is captured in `get_pitcher_roster`/`get_hitter_roster` and broadcast to all rows. `lineupSlot` is the primary check (slot 17 in ESPN's POSITION_MAP); `pl.injured` is a fallback because ESPN's API sometimes omits `lineupSlotId` from the roster entry (e.g. Will Smith on the 60-day IL), leaving `lineupSlot = ''`. Keep it a **native bool**, NOT `.astype(bool)` в†’ numpy `bool_`, which `json.dump(default=str)` stringifies to the truthy `"False"`.

### FA logic
- **FA exclusion:** players claimed today are found via today's `transactions` list; the *most recent* transaction per player wins. FA views + positional breakdown exclude all `_DL_STATUSES`.
- **FA RP requires SVHD в‰Ґ 1:** `fa_relievers` gates on `(_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))) >= 1`. Zero saves + zero holds all season = no role.
- **FA SP score floor (`_FA_SP_MIN_SCORE = 30`):** `fa_starters` drops any starter whose `_score_p` is below 30 (streamer-tier arms not worth the risk) BEFORE the top-12 slice. Applied inside `fa_starters` so all three consumers agree вЂ” the FA SP section, the Week-at-a-Glance best-FA-SP bullet, and `_roster_suggestion`'s pitch-stabilizer pool. Tune the one constant to change the bar; the section subtitle + empty-state message interpolate it.
- **FA "Cats" column (`_cats_cell` / `player_cat_strengths` / `build_cat_percentiles`):** FA Hitters + FA Relievers show a **Cats** column (before Score) listing up to 3 roto cats the player is strong in (percentile в‰Ґ 0.70 within a qualified YEAR pool; `_LOWER_BETTER` inverted). Cats in `need_cats` (my currently-losing в€Є tossup) render `ACCENT`; others `MUTED`. `_FA_HIT_CATS=[R,HR,RBI,SB,OPS]`, `_FA_RP_CATS=[SVHD,K,W,ERA,WHIP]`.

### Roster/drop rules
- **Never drop an IL-slot player (`_on_il`):** the league has **2 dedicated IL slots** that don't consume active/bench room, so dropping a player parked there frees nothing. `_on_il(r)` (tolerant of a stringified `"true"`) gates two drop paths: (a) `_can_drop` returns False (Week-at-a-Glance bullet 4); (b) `positional_breakdown`'s `worst_player` uses `drop_pool = [r for r in my_p if not _on_il(r)] or my_p`. It is lineup-SLOT-specific, NOT injury status вЂ” a DL player on the BENCH is `False` and stays droppable (cutting him frees a real bench spot). Populated for all teams; only my rows affect my suggestions.
- **`_can_drop`** still guards that every position keeps в‰Ґ 1 healthy player.

### SP/RP role detection (`_is_sp(r)`)
Never use `"SP" in pos` or `gs > 3` alone. Priority chain: ESPN season GS/GP ratio (в‰Ґ 5 appearances) в†’ dataset GS/G ratio (в‰Ґ 4 appearances) в†’ IP/G в†’ Position field. Thresholds: GS/G в‰Ґ 0.80 в†’ SP, в‰¤ 0.20 в†’ RP; IP/G в‰Ґ 4.5 в†’ SP, < 2.5 в†’ RP. Used by `pitcher_score`, `_score_p`, `fa_starters`, `fa_relievers`, My RP filter, `positional_breakdown`.

### Pitcher hot/cold uses 15-day ERA
`build_pitcher_hot_cold_section`, My Upcoming Starts, and FA Starting Pitchers compare season ERA vs 15-day ERA (`p15` = Dataset==15 rows). The 7-day window is too short for infrequent SPs. When a player is absent from the FP 15-day top 300 (fringe starters), fall back to `rec_p` (pybaseball Baseball Reference 15-day scrape in `recent_pitching`). Column header "L15 ERA". `fetch_recent_pitcher_stats` fetches 15 days to match. Both `fetch_recent_pitcher_stats` and `fetch_recent_hitter_stats` accept optional `start_dt`/`end_dt` string params for an exact date window (used to populate `prev_week_hitting`/`prev_week_pitching`).

### Save-Role Watch (`save_role_watch`)
SVHD is the most volatile category, and **recent holds are unavailable anywhere in the pipeline** вЂ” per-window `SVHD` captures recent SAVES only (FP windows have no HLD), ESPN exposes only season totals. So recency is save-only. Flags (a) **emerging FA closers** вЂ” FA RP with в‰Ґ 3 saves in last 15 days вЂ” and (b) **fading rostered closers** вЂ” my RP gated on **season saves `ESPN_SV` в‰Ґ 5** (a real closer) with 0 recent saves despite в‰Ґ 3 recent appearances. The fading side is gated on season *saves*, not SV+H, so a holds-based reliever (e.g. JoJo Romero 0 SV / 20 HLD) is never falsely flagged. Rendered as a callout on the FA Relief Pitchers section.

### Category classification (`classify_categories`)
Returns `{cat: (proj_res, tier)}` reusing Category Pulse's projection math (`_project` + `pit_proj` for K/QS/W). Tier is only `tossup` (margin в‰¤ `_CLOSE_THRESH`) or `leaning` вЂ” **no `locked` tier / no рџ”’ badge**. Used to detect a THIN ERA/WHIP lead for the ratio-stat pickup warning. Computed once in `build_email` as `category_classification`, passed to `_roster_suggestion`; the pickup steering targets all losing cats (no dead-cat pruning).

### Matchup W-L uses ESPN's `result` field, NOT raw comparison
`get_all_matchups`/`get_prev_matchup` read `box.home_stats[cat]["result"]` (`WIN`/`LOSS`/`TIE`) as source of truth, mapped to `W`/`L`/`T`. Critical because ESPN applies a ratio-stat **innings-pitched minimum** (~25 IP) before ERA/WHIP count вЂ” a team with the better WHIP but under the IP floor **loses** that category. **Intended consequence:** the current result can differ from the projected result (Category Pulse / `classify_categories` compare raw projected values without the IP floor). Don't unify them вЂ” the divergence is informative. **BUT `get_all_matchups` falls back to a raw value comparison when `result` is `None`** вЂ” essential because ESPN leaves per-category `result` unset for the *live, in-progress* matchup period until it closes.

### `roto_score_week` ranks by VALUE, not `result` (so the live current week populates)
`roto_score_week` builds the per-week roto table by ranking teams on each category's `value` (coerced numeric), NOT gating on ESPN's per-category `result`. This is deliberate: for the live matchup period ESPN leaves `result = None` on every category until the period closes, so a result-gated filter would drop the entire current week вЂ” leaving `get_all_roto` with no current-week rows, which makes `_all_roto_tied` True and **silently suppresses digest sections 4 + 4b (roto rankings)** mid-week. Ranking by value is identical for completed weeks (value + result both present; the ranking math never used `result`) and additive for the live week. `get_all_roto` uses `continue` (not `break`) on a failed week so one bad week can't truncate history.

### Category Pulse
- Tied cats use `TEXT` (#e2e8f0, white) for border/value/status вЂ” not `YELLOW`. Win=green, loss=red, tie=white.

Tie band half-width `0.5В·10^-dec` matches the display-precision tie test. Display-only вЂ” `classify_categories` untouched. `WIN %` per-card only; summary record stays point-estimate `в†’ proj`.

### Terminology: user-facing labels say "Matchup N", NOT "Week N"
The ESPN unit is a **matchup period** (`currentMatchupPeriod`, the roto/box-score unit), which is *usually* 7 days but is **14 days for the All-Star break and playoffs** вЂ” and the number (e.g. 15) is a matchup *index*, not a calendar-week index (a 2-week matchup still advances it by only 1). So every **display label** referring to that unit uses **"Matchup {n}"** (e.g. "Matchup 15 Roto Rankings", "Matchup 15", "Matchup at a Glance", "Starts This Matchup", "Opponent This Matchup", "Matchup N Recap/Highlights", Season Trajectory "by matchup", "of the Matchup"). Only **genuine 7-day rolling windows keep "week"**: "last 7 days", "7-day OPS", "L15 ERA", and the **Lineup Watch** which is deliberately the current *calendar* week (Monв†’yesterday, "week to date"/"so far this week") вЂ” distinct from the full matchup. Internal names (snapshot `Week` field, `weekly_results`, `week_end_str`, `current_week_num`, `prev_week`, `weekly_avgs`, filenames like `recap_week_N.html`, `.github/workflows/weekly-recap.yml`) are **NOT** renamed вЂ” display strings only. When adding a label for the matchup unit, write "Matchup", not "Week".

### Weekly matchup is MondayвЂ“Sunday (or MondayвЂ“SundayГ—2 for All-Star break)
`week_end_str` comes from `matchup_end_date` in the snapshot (not a hardcoded `today + 6 - weekday`). `fetch_data.get_matchup_dates(league)` reads `league.settings.matchup_periods` вЂ” a dict mapping period в†’ list of weekly scoring-period IDs (e.g. `{'15': [15]}` for 7 days, `{'16': [16, 17]}` for 14-day All-Star). Period length = `len(ids) * 7 days`. Stored in snapshot as `matchup_start_date`, `matchup_end_date`, `matchup_period_days`, `next_matchup_end_date`. Dates past `week_end_str` get a `NEXT WK` badge. **Bullet 2 scopes both `confirmed` and `thin_days` to `PSP_Date <= week_end`** so its count matches the KPI.

### End-of-matchup mode (`is_sunday`)
When `today >= matchup_end_date` (last day of the matchup period вЂ” not always a calendar Sunday for multi-week periods): header subtitle в†’ "Weekly Lookahead"; subject в†’ "Lookahead"; KPI в†’ "Starts Next Week" (counts starts after `week_end_str`); Category Pulse subtitle в†’ "Final stretch вЂ” week ends today"; Week at a Glance в†’ "Next Week Preview"; bullet 1 appends "вЂ” final"; bullet 2 shows next-period confirmed starts; bullet 3 shows best FA SP for next period. `next_week_end_str` comes from `next_matchup_end_date` in the snapshot (available in `build_email`). `classify_categories` and `build_category_pulse` accept `matchup_days=matchup_period_days` so `elapsed_frac` divides by 14 (not 7) for a 2-week period.

### Probable starters
Primary: two MLB API calls (range schedule в†’ batch hydrate) give **confirmed** starters (`PSP_Projected=False`), published only ~1вЂ“2 days out. A live-feed fallback exists if the batch returns nothing. **Unannounced slots are filled by a rotation projection that advances each team's rotation ORDER through its actual upcoming games** (`get_probable_starters` Step 3) вЂ” NOT the old `last_start + 6 calendar days` guess (which double-listed two projected SPs on the same team/day because each pitcher independently claimed its nearest open game, with nothing enforcing one-pitcher-per-game). The new walk: `_get_team_recent_starts(days_back=14)` returns each team's confirmed past starts chronologically в†’ build a rotation **queue** of members who started within `_ROT_RECENCY` (11 days вЂ” excludes IL/optioned arms with stale last-starts), ordered by last-start-date ascending (longest-rested = most "due" = front). Then walk the team's upcoming games in date order, assigning each **open** game to the front-of-queue pitcher and rotating them to the back (one game = one pitcher). Confirmed MLB entries act as **anchors**: their slot is skipped (already in `frames`) and the pitcher is rotated to the back to re-sync the cycle; a pitcher with a confirmed *future* start is skipped for earlier open games (`confirmed_date` guard) so they're never projected before a start ESPN already confirmed. **Two structural wins over +6-day:** (a) game-exclusive в†’ two projected SPs can NEVER share a team/day; (b) turns counted by games (not calendar days) honor off-days/doubleheaders and correctly surface legitimate two-start weeks. A pitcher can legitimately appear once confirmed + once projected (later date) вЂ” `_attach_start_lists` folds both into `PSP_Dates`; the scalar `PSP_Date` stays the earliest (confirmed sorts first). External sources rejected: FanGraphs (projected rotations) 403s; Rotowire/others have no free API вЂ” MLB StatsAPI is the only clean feed, so the improvement is a smarter model on data we already pull.

### HR% (`_hrp_cell`)
`HR_Probability` (computed in `fetch_data.compute_hr_probability` from barrel%, hard-hit%, launch angle, HR/AB, xwOBA, ISO, recent HR streak; в‰€ 0.05вЂ“0.31, a modeled per-game HR probability) is a color-coded `HR%` column in Roster Hot/Cold + FA Hitters via `_hrp_cell(row)`, with a hover `title` tooltip of drivers (Barrel% В· HardHit% В· EV В· xwOBA В· ISO). Green в‰Ґ 20%, yellow в‰Ґ 14%. Takes the full player row (Roster Hot/Cold stashes the season row as `srow`). **`compute_hr_probability` measures power SKILL, not availability** вЂ” it must NOT gate on `ESPN_Status` (an earlier gate zeroed out Judge/Trout/Buxton). Returns 0.0 only when there's no usable signal at all (shows "вЂ”"). `ISO = SLG в€’ AVG` (FP omits it).

### Week at a Glance pickups (`_roster_suggestion`) are roster-context aware вЂ” returns a LIST of bullets
**Superseded the old "best available hitter" bullet** (which was blind to positional need вЂ” it chronically told you to add an OF, the deepest FA pool, even while you were benching a masher there and near-last at catcher). Now returns a **list** of up to two bullets (build_week_overview `.extend`s them; still tolerates a bare string): 
- **BAT bullet** вЂ” upgrades my **weakest hitter position** where a real FA upgrade exists. Uses `positional_breakdown` results (passed in as `pos_data`) for per-position league `rank` + my `worst_player` + best `top_fa` (both carry `_pscore`). A position is a **NEED** if `rank` is bottom-third (`rank в‰Ґ n в€’ round(n/3) + 1`) AND `top_fa._pscore в€’ worst._pscore в‰Ґ _UPGRADE_MARGIN` (**3.0** вЂ” deliberately modest: at your weakest spot even a small bump matters; the goal is to fill the hole, not chase the biggest raw gap). Among needs, picks the **weakest** (highest rank), tie-broken by upgrade size. **SURPLUS positions are excluded**: top-third rank OR a position where `lineup_efficiency_current` shows I'm leaking bench production (`lineup_eff.bench` в†’ `leak_groups`). Fallback when no positional need: best FA hitter for a losing hitter cat, but still **not** at a surplus position. 
- **PITCH bullet** вЂ” fires when `lineup_eff.blowups` is non-empty (an active-slot implosion this week) **OR** I'm losing ERA/WHIP by a **non-tossup** margin (`classification[cat][1] != "tossup"`). Recommends a high-**floor** stabilizer from `fa_sp + fa_rp` gated on a real sample (`_pit_viable_min`) and good ratios (ERA в‰¤ 4.00, WHIP в‰¤ 1.25), ranked by `(xERA or ERA, WHIP)` ascending вЂ” a low-volatility arm, NOT a streamer that would worsen ratios.

Drops prefer **surplus** players (deep position / bench-leaker), sorted surplus-first then score-ascending; the two bullets take **distinct** drops (`_take_drop` / `_used_drops`); a `[surplus]` tag marks a drop-from-strength. `_move_tail` shows a green "roster spot open" free-pickup badge while `slots_left > 0` (decremented per add), else appends the drop. `_can_drop` still guards в‰Ґ 1 healthy player per position and never drops an IL-slot player. Wired in `build_email`: `_roster_suggestion(..., pos_data=pos_data, lineup_eff=(lineup_efficiency_current if not override_team else {}))` вЂ” the `--team` view passes `{}` (no daily-lineup data for other teams), so its pitch bullet fires only on ratio loss. The old SP `ratio_warn` guardrail was **removed** (the pitch bullet now picks a stabilizer by construction, so there's nothing to warn about; the bat bullet is hitters-only).

### Score cascade (`best_recent_p` / `best_recent_h`)
Built in `build_email` by merging `{**rec_p_fp, **p7, **p15, **p30}` (pitchers) and `{**rec_h, **h7, **h15, **h30}` (hitters) вЂ” later dicts win, so 30d FP > 15d FP > 7d FP > Baseball Ref. Passed to `_blend` and `positional_breakdown`.

### positional_breakdown viable filter
FA pool per position excludes benchies. SP: `GS >= _pit_viable_min("SP","GS")`. RP: `ESPN_GP >= _pit_viable_min("RP","GP") or IP >= _pit_viable_min("RP","IP")`. Hitters: `OPS > 0.200 or R+RBI > 5`. FA quality (`fa_quality`) = avg blended score of top-3 viable FAs. Scarcity: `< 50` scarce (RED), `< 60` moderate (YELLOW), `>= 60` deep (MUTED).

### Dynamic volume benchmarks (no hard-coded IP/AB/GS minimums)
"Full-time" thresholds are derived from the live snapshot each run so they scale with the season. Two builders, both called once at the top of `build_email`:
- `compute_ab_benchmarks(hitters)` в†’ `_AB_BENCH[window]` = `_AB_LEADER_FRAC` (0.62) Г— the window's p95 (leader) AB. Consumed by `_ab_opportunity_mult` in `hitter_score`. `_FULLTIME_AB` is a cold-start fallback only.
- `compute_pitcher_benchmarks(pitchers)` в†’ `_PIT_BENCH[(window, role)]` = leader IP/GS/GP (p95) per role, `_is_sp`-split. `_ip_reliability_mult` uses `_IP_RELY_FRAC` (0.20) Г— leader IP for the row's window+role. `_pit_viable_min(role, stat)` uses `_GS_VIABLE_FRAC`/`_GP_VIABLE_FRAC`/`_IP_VIABLE_FRAC` (0.17/0.30/0.38) Г— the season leader. `_PIT_FALLBACK` holds cold-start constants.

### Data-derived league averages (`_LG` / `compute_league_averages`)
Called once in `build_email` next to the benchmark builders; writes `_LG` with `ops` (full-time regulars), `team_ops` (mean opponent OPS faced), `team_k`, and starter `era`/`whip`/`k_pct`/`ip_per_start`/`barrel_allowed` from qualified YEAR rows. Consumers read `_LG.get(key) or <old literal>`. `qs_probability` stays calibrated because the intercept `38` and multipliers are fixed. fetch_data derives its own `LG_OPS` for wRC+. ## Scoring functions (send_digest.py)

- `_is_sp(r)` в†’ bool. Usage-based SP/RP detection (see gotcha).
- `_blend(r, score_fn, idx_recent, w=None)` в†’ blended score. `_BLEND_W = 0.35` (35% recent + 65% season) вЂ” single source for math + tooltip. `idx_recent` is `best_recent_p`/`best_recent_h`. Applies to hitters + SPs; RP `rp_score` never blended.
- `hitter_score(r, _parts=вЂ¦)` / `pitcher_score(r, _raw=вЂ¦, _parts=вЂ¦)` / `rp_score(r, _raw=вЂ¦, _parts=вЂ¦)` в†’ `_parts=True` returns `(components_dict, multiplier)`. Component insertion order == display order (single source for tap-to-expand).
- `_score_p(r, idx_recent=None)` в†’ canonical role-aware pitcher score. SP в†’ `_blend(r, pitcher_score, idx_recent)`; RP в†’ `rp_score(r)` unblended. Used by every pitcher Score display/sort.
- `_starts_this_week(r, today, week_end)` в†’ int. Upcoming starts within the matchup week (from `PSP_Dates`; falls back to scalar `PSP_Date`). Drives the `2-START` badge and best-FA-SP preference.
- `save_role_watch(pitchers, my_team, claimed)` в†’ `(emerging, fading)` (see gotcha).
- `classify_categories(matchup, weekly_avgs, days_elapsed, remaining_proj, matchup_days=7)` в†’ `{cat: (proj_res, tier)}` (see gotcha). Pass `matchup_days=matchup_period_days` for 2-week periods.
- `compute_weekly_std(roto, current_week)` в†’ per-team/per-cat stddev; `_cat_win_prob(pm, po, cat, sigma, remaining_frac)` в†’ `(p_win, p_tie)` (see Category Pulse gotcha).
- `opponent_week_intel(pitchers, hitters, opp_team, best_recent_h, today, week_end)` в†’ dict (starts, two-start pitchers, hot hitters) for the Opponent This Week block. None when `opp_team` empty.
- `pitcher_score(r, _raw=False)` в†’ 0вЂ“100. Components: K/WhiffPctile (28), ERA/xERA (28), WHIP (20), contact-quality/BarrelPct+xwOBA (0вЂ“12), SP role bonus 9вЂ“12 by GS. Small-sample penalty `s *= min(1.0, ip/20)`. Calibrated `s * A + C` with `(A,C) = _SCORE_CALIB["sp"]` (re-anchored live вЂ” see below). `_raw=True` returns pre-calibration.
- `rp_score(r, _raw=False)` в†’ 0вЂ“100. SVHD de-emphasized to ~15% (punt-saves). Components from ESPN season counts: SVHD (15), K (26), W (15), ERA/xERA (16), WHIP (12), IP/G (8), contact-quality (0вЂ“8). Calibrated `s * A + C` with `(A,C) = _SCORE_CALIB["rp"]`. My Relief Pitchers picks best dataset per player (YEAR в†’ 30 в†’ 15 в†’ 7).
- **Live SP/RP recalibration (approach A + small-pool guard) вЂ” `compute_score_calibration(pitchers)`:** the SP/RP calibration constants are **re-derived every run** from the snapshot's raw-score distribution (same p50в†’50 / p90в†’80 solve as `recalibrate_scores.py`, same qualified pool: SP = `_is_sp` + IP past `_IP_RELY_FRAC` of the season leader; RP = `_pit_viable_min` on GP or IP; both YEAR rows) and written to the module global `_SCORE_CALIB = {"sp":(A,C), "rp":(A,C)}`, which `pitcher_score`/`rp_score` read. No hand-paste, no CI job вЂ” the anchors track the season automatically (median/p90 pinned в†’ sub-point daily drift for most players; the elite tail, extrapolated past p90, wiggles ~1вЂ“2 pts most). **Called in `build_email` right AFTER `compute_pitcher_benchmarks`** (qualification + `_raw` scores read `_PIT_BENCH` вЂ” order matters). **SMALL-POOL FALLBACK:** a role whose qualified pool is below `_MIN_CALIB_POOL` (30) or degenerate (`p90 <= p50`) KEEPS the hand-tuned literals in `_SCORE_CALIB`'s default (SP 1.5070/-44.3346, RP 1.6543/-28.0645), so a thin/noisy early-season distribution can't warp the scale. `recalibrate_scores.py` is now a **manual inspection tool** (prints the live-derived constants) and the **source of the fallback literals** вЂ” update the `_SCORE_CALIB` defaults from it if the component mix changes materially. Hitter score is NOT live-recalibrated (still `s * 1.587 - 5.2`).
- `hitter_score(r)` в†’ 0вЂ“100. Prefers wRC+ over OPS. Uses xwOBA, sprint speed, Barrel%, ISO, HR_Probability. Opportunity multiplier (`_ab_opportunity_mult`): raw score scaled by AB vs a full-time benchmark (floored `_AB_FLOOR = 0.40`, capped 1.0) вЂ” a full-time hitter lands at 1.0 (no penalty). Calibrated `s * 1.587 - 5.2`. Displayed everywhere as `_blend(r, hitter_score, best_recent_h)`.
- `qs_probability(r)` в†’ 1вЂ“99. Calibrated league-avg ~38%, ace ~75%. Uses IP/G (not IP/GS).
- `_fmt_ip(ip_decimal)` в†’ baseball IP notation. `whole = int(d); outs = round((d-whole)*3); if outs>=3: whole+=1, outs=0`.
- `_proj_line_html(r)` в†’ `IP В· ER В· K` span. ER = `raw_er * opp_factor * park_factor`; `opp_factor = clamp(opp_ops / LG_team_ops, 0.80, 1.20)`; `park_factor` = 0.97 home / 1.03 away. K also opponent-adjusted via `Team_K_Value`. IP = `IP_per_G`.
- `hot_cold_cell(season_val, recent_val, вЂ¦, no_data_title=None, td_style=TDC)` в†’ `<td>` with colored recent stat + рџ”Ґ/в†‘/вќ„/в†“ icon vs season baseline. When recent is missing/zero and `no_data_title` is set, renders `вЂ”` with a dotted underline + hover tooltip. Optional `td_style` so the compacted pitcher tables' L15 ERA cell matches.
- `band_divider(label, color, anchor=вЂ¦)` в†’ full-width band boundary `<div>`.

### Hitter tactical badges (PWR / SB / BUY / SELL)
Glance flags next to a **hitter's** name, mirroring the SP badges вЂ” **display-only, never folded into any score**. One shared source: `hitter_badges(row, hit_pctile=None, cap=2)` in send_digest.py (dashboard imports `sd.hitter_badges`), which emits chips via `_hit_badge(text, color, title)` (translucent QS/5K+ style). Each carries a hover `title` naming the justifying stat, so a badge stays **anchored even in a table with no matching column** (e.g. Roster Hot/Cold shows no SB column). Priority order **SB в†’ PWR в†’ BUY/SELL**, capped at 2. Four flags (tunable module constants, calibrated to fire on ~10вЂ“25% of the qualified YEAR pool):
- **PWR** (`PURPLE`) вЂ” power/HR threat: `HR_Probability в‰Ґ _PWR_HRP_MIN` (0.23 вЂ” a notch above `_hrp_cell`'s 0.20 green tier, so only elite power; ~7% of the qualified pool). Title = `_hrp_driver_str(row)`.
- **SB** (`SILVER`, "Quicksilver") вЂ” genuine base-stealer: SB percentile `в‰Ґ _SB_PCTILE_MIN` (0.80) via the existing `_cat_pctile`/`build_cat_percentiles` SB pool, AND `SprintSpeed в‰Ґ _SB_SPEED_MIN` (27.0, skipped when SprintSpeed missing). Needs the `hit_pctile` pool вЂ” when `None` the SB badge silently doesn't fire.
- **в–ІBUY** (`GREEN`) / **в–јSELL** (`RED`) вЂ” Statcast expected-vs-actual regression read (skill-vs-luck), mutually exclusive: derive `SLG = ISO + AVG`; BUY = `xBAв€’AVG в‰Ґ _XREG_BA` (0.020) AND `xSLGв€’SLG в‰Ґ _XREG_SLG` (0.030) (unlucky в†’ positive regression coming); SELL = the same gaps inverted (riding luck в†’ regression risk). Requires `xBA/xSLG/AVG/ISO > 0`.

**Callers pass the `hit_pctile` pool** (built once via `build_cat_percentiles(_hit_pool, _FA_HIT_CATS)` on the qualified YEAR hitter pool вЂ” send_digest.py in `build_email`, dashboard.py in `build_context` as `ctx["hit_pctile"]`). **Digest sites:** Roster Hot/Cold (`build_hot_cold_section` takes a `hit_pctile` param), FA Hitters, Positional Breakdown (hitter branch only вЂ” gate on `p["ptype"]`). **Dashboard sites:** Hitting Hot/Cold, FA Radar hitters, Weakest Spots (hitter positions only). **When adding a hitter-listing surface**, drop `hitter_badges(row, hit_pctile)` after the name and make sure the caller has a `hit_pctile` pool in scope.

## Automation

- **GitHub Actions (three scheduled workflows, cron always UTC):** `daily-digest.yml` fires **06:00 UTC** (2 AM EDT) once daily (`python send_digest.py`); `daily-dashboard.yml` fires **06:00 UTC** (2 AM EDT) once daily (`python dashboard.py --refresh --email`); `weekly-recap.yml` fires Monday 15:30 UTC. **Cron = when the job FIRES, not when the email ARRIVES.** GitHub's scheduler is unreliable вЂ” delays run 1вЂ“4 h and in practice **~6 h** (measured: the 06:00 UTC digest lands ~8 AM ET) вЂ” and only ever push runs LATER, never earlier. So arrival в‰€ cron + ~6 h в†’ **both digest + dashboard land ~8 AM ET** (they share the 06:00 UTC cron вЂ” move both together). That ~6 h lag is also what keeps the fetch fresh: both morning jobs `--refresh`/fetch at run time and their 2 AM crons actually execute ~8 AM, well past ESPN's overnight stat/standings rollover. The only freshness risk is a rare FAST-scheduler day (1вЂ“2 h delay в†’ fetch ~3вЂ“4 AM, possibly mid-rollover); firing later (e.g. 4 AM EDT) removes that risk but pushes normal-day arrival ~2 h later, so we DON'T вЂ” user chose earlier arrival over that edge-case safety. The digest's old second run (15:00 UTC afternoon) was removed вЂ” once daily now. All three share repo secrets `GMAIL_APP_PASSWORD`, `ESPN_SWID`, `ESPN_S2`. Decision: sticking with GitHub Actions вЂ” do not migrate to cron-job.org unless revisited.
- **Fetch-time freshness badge:** the record and category standings are a point-in-time snapshot of ESPN's live box scores at fetch time. The header badge shows the fetch **time in ET** (`_fmt_refresh_time`): green вњ“ when data is from today, yellow вљ  when stale. `fetch_data.py` writes `refreshed_at` as tz-aware UTC (`datetime.now(timezone.utc)`) so the ET conversion is correct on CI (UTC) or local; `_fmt_refresh_time` converts to `America/New_York`, shows naive stamps as-is. `_data_fresh` keys off the date only.
- **Local runner:** `scripts/run_digest.bat` for manual runs. Captures console output (incl. tracebacks) to `logs/run_console.log`; the structured one-line-per-send record goes to `logs/digest.log` (written by send_digest.py). Kept in **separate files** вЂ” the old shared handle collided (`PermissionError` on Windows). The Python write is wrapped so a locked log can't crash a run that already sent.
