# CLAUDE.md

Guidance for Claude Code when working in this repo. Actionable rules only — background/rationale and completed-work forensics live in `NOTES.md`.

## Where things live — read before editing

CLAUDE.md holds universal + cross-cutting rules. Per-surface detail lives in `docs/*.md`,
loaded on demand. **Before editing a file below, read its doc first** — those docs are as
authoritative as this file:

| Editing… | Read first |
|---|---|
| `fantasy/trades.py`, `trade_lab.py`, or trade sections of `send_digest.py` | `docs/trades.md` |
| `fantasy/scoring.py`, `fantasy/analytics.py`, badges, Category Pulse / win-prob | `docs/scoring.md` |
| `dashboard.py` | `docs/dashboard.md` |
| `weekly_recap.py` | `docs/recap.md` |
| `fetch_data.py` (sources, seeding, probable starters, matchup dates, lineup efficiency) | `docs/fetch_pipeline.md` |

Rules that apply everywhere (names/merges, fields & sentinels, the shared derived-value
prelude, key data fields, digest section order, palette, automation) stay in this file.
**When you add a new surface/file, add a router row + a `docs/*.md`.** `@import` is
intentionally NOT used — it eager-loads at startup and would defeat the deferral.

## Commands

```bash

python send_digest.py --no-refresh             # use existing snapshot (fast — email-only changes)
python send_digest.py --dry-run                # save HTML to previews/, no email
python send_digest.py --dry-run --no-refresh   # instant preview, no network

python send_digest.py --dry-run --no-refresh --team "Houck Tuah"  # another team's digest (needs all_matchups)
python fetch_data.py                           # refresh data only → data/snapshot.json

python weekly_recap.py --dry-run --no-refresh  # instant recap preview → previews/recap_week_N.html
python bench_leakage.py                         # standalone daily-lineup audit (my team + last week's opponent → console)
python backtest_projections.py                  # walk-forward accuracy of the SP proj line (IP/ER/K) vs actual game logs → console
python backtest_projections.py --limit 30 --no-cache  # quick smoke test (small pool, no cache)
python backtest_winprob.py                      # walk-forward calibration of the category Win% model (_cat_win_prob) → reliability bins + Brier + ECE + sigma-inflation sweep (snapshot-only, no network)
python snapshot_schema.py                       # validate data/snapshot.json against the reader contract (ERROR/WARN report; exit 1 on error)
python data_coverage.py                         # SEMANTIC coverage/degradation report (Statcast %, recent-window counts, freshness, probable-starter confirmed%) → console; --strict exits 1 on LOW (CI gate)
python scripts/render_diff.py baseline          # refactor safety net: render all previews offline + save as golden baseline
python scripts/render_diff.py check             # re-render + byte-compare vs the baseline (normalizes Trade Lab builtAt; pins PYTHONHASHSEED). Use THIS, not GNU diff — diff emits degenerate whole-file hunks on the digest's >20KB lines
python dashboard.py                             # single-viewport dashboard → previews/dashboard_{team}.html (uses existing snapshot, no email)
python dashboard.py --refresh                   # refresh data (~60s) first, then write the dashboard
python dashboard.py --team "Houck Tuah"         # another team's dashboard (needs all_matchups)
python dashboard.py --email                     # also email the dashboard to yourself as an attachment (open in a device browser)
python trade_lab.py                              # interactive browser-only Trade Lab → previews/tradelab_{team}.html (uses existing snapshot)
python trade_lab.py --refresh                    # refresh data (~60s) first, then write the Trade Lab
python trade_lab.py --team "Houck Tuah"          # default the LEFT (my) side to another team
python trade_lab.py --partner "The BIG Dumpers" --give "Jake McCarthy" --get "JJ Wetherholt"  # PRELOAD a specific deal (partner + names, comma-separated); baked into DATA.preload so it survives the Windows file:// launch (URL #hash form also works: #partner=...&give=...&get=...)
pip install -r requirements.txt
```

No linter, no test suite. Verify by opening `previews/digest_preview_{team_slug}.html` or `previews/recap_week_N.html` in a browser. The dashboard (`previews/dashboard_{team_slug}.html`) is designed for a **1440×900 laptop with zero scrolling** — verify by opening maximized (or headless-screenshot at 1440×900) and confirming no scrollbars appear.

## Architecture

Three core scripts + interactive tools, one intermediate artifact. The digest, recap, dashboard, and Trade Lab all read the same snapshot:

**`send_digest.py` internals live in a `fantasy/` package (F5 split).** Bottom-up layers, each importing only lower ones; `send_digest.py` is a **re-export facade** (`from fantasy.<mod> import *`) so `import send_digest as sd; sd.<name>` and `from send_digest import <name>` keep working unchanged for dashboard/trade_lab/weekly_recap/CI — **no caller imports `fantasy.*` directly.**
- **`fantasy/ui.py`** — pure presentation LEAF (stdlib-only): color palette + `TH_S/TD_S/TDC`, `badge`/`_hit_badge`/`two_start_badge`, `band_divider`/`nav_bar`/`section_head`, logos, `score_reveal`+`_bd_uid`+`_BD_SEQ`+`_BD_TOGGLE_SCRIPT`, `hot_cold_cell`/`kpi_cell`, formatters (`_n`/`_ascii_lower`/`_fmt_ip`/`_st`/`v`/`_fmt_refresh_time`/`_fmt_game_time_et`).
- **`fantasy/scoring.py`** — role scorers (`pitcher_score`/`rp_score`/`hitter_score`/`_score_p`/`_blend`/`_is_sp`/`qs_probability`), the per-snapshot calibration state + builders (`compute_pitcher_benchmarks`/`compute_ab_benchmarks`/`compute_score_calibration`/`compute_league_averages`/`compute_xera_offset`), blowup/regression flags, the **season starter-skill badge** `sp_skill_badges` (+ `_sp_qs_season` + the `_QS_SEASON_MIN`/`_K_SEASON_MIN`/`_SP_SKILL_MIN_IP` consts — a pure row→HTML badge, so it lives here next to the other pitcher badges for the trade engine to reuse), and **`YEAR`** (single-sourced here). Imports ui. **Global-state rule (load-bearing):** the calibration dicts `_LG`/`_PIT_BENCH`/`_AB_BENCH`/`_SCORE_CALIB` are mutated **in-place** (`.clear()` + item-assign, never rebound), so the facade's star-import binding tracks their per-run updates across the module boundary. The one **rebound** global `_XERA_OFFSET` is co-located with its sole reader `pitcher_regression_flag`. **When you add a module global that a reader in another layer needs, either mutate it in-place OR keep it with its readers — never rebind a global that a different module reads via the facade.**
- **`fantasy/analytics.py`** — the layer above scoring: per-cat percentile/value helpers (`_cat_pctile`/`_cat_value`/`player_cat_strengths`), `positional_breakdown` + `team_category_ranks`, the tap-to-expand score-breakdown prose (`_hit_clauses`/`_sp_clauses`/`_rp_clauses`/`_score_narrative`/`_hitter_score_breakdown`/`_pitcher_score_breakdown`/`_badge_ctx_wrap`/`_hit_badge_context`/`_sp_skill_context`/`_hrp_driver_str`), the tactical **`hitter_badges`** renderer (moved down from send_digest — it only needs analytics-level `_hrp_driver_str`/`_cat_pctile`, so the trade engine can reuse it; a `regression=False` flag lets the trade cards skip the generic `$`/`▼`), `_on_il`, `_badge_name_key` (= `_name_key`), and the roto/position consts (`POS_GROUPS`/`POS_STARTERS`/`_LOWER_BETTER`/`_CAT_DISPLAY`/`_DL_STATUSES`/`_FA_HIT_CATS`/`_FA_RP_CATS`). Imports name_utils + ui + scoring. **No trade logic and no `build_*` section renderers** — a strict lower layer the trade engine and the section renderers both read (verified: zero analytics→trades edges).
- **`fantasy/trades.py`** — the trade engine + trade/pending-offer renderers: `find_trades`/`find_trades_combined`, the scarcity-weighted value currency (`_trade_value`/`compute_position_scarcity`/`_enrich_trade_player`/`_health_value_mult` — the last a severity-tiered IL discount on `_tval` (short IL a small haircut, severe/season-lost deep); `_il_strips_star` splits out `_tval_star` (the star-reach premium basis — healthy value for a short IL so an injured superstar isn't a freebie, discounted only when out for the season); and `_team_position_counts` excludes IL-slot bodies so an injured stud doesn't read as roster depth), the acceptance layer (`_need_mult`/`_trade_tilt`/`_star_premium`/`_deal_star_reach`/`_deal_star_surrender`/`_pending_verdict`/`_counter_suggestion`), the real-offer grader (`_grade_pending_trades`), the digest renderers (`build_trade_radar`/`build_pending_trades_section` + `_trade_player_line`/`_il_badge` — the injury chip that makes the `_health_value_mult` discount visible on a card, mirrored into Trade Lab via `_serialize`'s `badges` string — `/_verdict_pill`/`_tradelab_button`/`_trade_net_summary`), and the `_TRADE_*`/`_STAR_*`/`_NEED_MULT_*` consts + the **personal strategy overlay** `_TEAM_PUNT_CATS`/`_TEAM_TARGET_POS` (team-keyed; drop punted cats from my needs, add target positions incl. SP — my-perspective only, see `docs/trades.md`). Imports ui + scoring + analytics. **Rebound-global footgun (extends the scoring rule):** the cap globals `_TRADE_MAX_CARDS`/`_TRADE_PER_TEAM_CAP` are rebound and live here, so `trade_lab`'s Partner-Fit board can't raise them via `sd._TRADE_MAX_CARDS = 400` (that only rebinds the stale facade). It calls the facade-safe `_set_trade_caps(max_cards, per_team_cap)` hook (defined here, re-exported) which mutates them in *this* module's namespace where `find_trades` reads them. The other rebound global `_POS_SCARCITY` is co-located with its writer `compute_position_scarcity` and sole reader `_trade_value` (nothing outside this module reads it), so it needs no hook.
- Still in `send_digest.py` (top layer): the `build_*` section renderers, `build_email`, `send_email`, `main`, and the injury/matchup/FA/nav helpers. `name_utils.py` (canonical `_name_key`) is a separate top-level leaf shared with `fetch_data`. **Extraction is bottom-up complete for the reusable engine (ui→scoring→analytics→trades); `send_digest.py` is now ~4.8K lines (was ~7.3K), holding the section renderers + orchestration.**
- **Verify any change to the split with `python scripts/render_diff.py check`** (byte-identical is the contract).

**`fetch_data.py`** pulls from 5+ sources → `data/snapshot.json`:
1. FantasyPros HTML (`pd.read_html`) — pitcher/hitter stats across 4 ranges (7/15/30/season). **BOTH the pitcher AND hitter YEAR pools are WIDENED beyond FP** — see the off-FP seeding gotcha (`_seed_offfp_pitchers` / `_seed_offfp_hitters`): players present in ESPN (rostered/FA) but absent from the FP scrape (call-ups / struggling-veteran FAs who fell out of the top-300) get synthetic season rows seeded from ESPN's own breakdown, so they enter the snapshot instead of being dropped by the left-from-FP merge.
2. ESPN Fantasy API (`espn_api` + one raw endpoint) — rosters, FA list, roto box scores, standings, transactions, and **pending trade proposals** (`get_pending_trades` — raw `mPendingTransactions` view, see gotcha)
3. MLB Stats API — opponent OPS + today's game schedule & national TV (`fetch_todays_games`). **Probable starters come from ESPN's public scoreboard, NOT MLB** (full-week projection — see the Probable starters gotcha)
4. pybaseball — Statcast contact quality, expected stats, sprint speed, recent game logs

**`send_digest.py`** reads the snapshot, computes all derived metrics, builds the email via Gmail SMTP. Two parts: **inline body = "The Briefing"** (a short skimmable plain-English overview — NOT the full digest) + an attached `digest_YYYY-MM-DD.html` (the full render). All new digest features go in the attachment (`build_email`'s `body_parts`).
- **The Briefing (`render_briefing`)** — the inline email body. **⚡ Act today** items first (two-start/must-start SPs via `_starts_this_week`, ⚠ low-floor starts via `_is_blowup_risk`, emerging FA closers from `save_role_watch`, injury alerts, incoming pending trades, and a **📺 Tune in** teaser naming the single highest-overlap MLB game today via the `tune_in` param), then a one-line **This matchup** read (current record labelled `now` + projected record from `category_classification`, with ▲▼◆ + light prose) and a **Season** line (standing of N + luck read). The `lead_in` prose is **projection-framed** ("The projection leans your way…"), never "you're ahead here", so it can't read as a claim about the current record beside it when the two diverge. **Inline-styled only** (Gmail strips `<style>`). Built at the END of `build_email` inside a try/except; **`build_email` returns `(full_html, briefing_html)`** — if the briefing fails, `briefing_html=""` and `main` falls back to the full digest as the body. `send_email(body_html, attachment_html, subject, filename, extra_attachments)` takes body and attachment separately. `--dry-run` also writes `previews/briefing_preview_{slug}.html`.
- **Subject:** `⚾ {team} {Digest|Lookahead}[ + Dashboard] — {date}` — "+ Dashboard" appended under `--with-dashboard`.

**`data/snapshot.json`** is the schema contract between the scripts. ~1.2 MB, not committed. Numeric missing values are stored as `-1` (not `NaN`) after the merge pipelines run.

**`snapshot_schema.py`** guards that contract (stdlib-only, no deps). `validate_snapshot(snap)` → `(errors, warns)` (never raises); `assert_valid(snap)` raises `SnapshotValidationError` on any error; `report(errors, warns)` prints an ASCII summary. **ERROR** (fetch fundamentally failed → a reader crashes/reads garbage): missing spine key (`refreshed_at`/`my_team`/`standings`/`pitchers`/`hitters`/`roto`), an empty core list, wrong-typed spine key, unparseable `refreshed_at`. **WARN** (degraded but survivable): a missing/mistyped non-spine key, row-field coverage below 90% (`_ROW_FIELDS`; **`FantasyTeam` is intentionally excluded** — empty for the FA pool), or a NaN leak where the `-1` sentinel belongs. **Wired at both ends:** `fetch_data.main()` calls `assert_valid(...)` right before `json.dump` — a raise exits nonzero, so readers fall back to the last good snapshot instead of writing a broken one; the three readers call `validate_snapshot` right after `json.load` and `report` only on a hit (quiet on a clean snapshot, never blocks — catches drift under `--no-refresh`). All wiring is `try/except ImportError` so the validator is optional. CLI: `python snapshot_schema.py [path]`. **When you add/rename a top-level snapshot key or a required row field, update `_SPINE`/`_EXPECTED`/`_ROW_FIELDS` to match.**

**`data_coverage.py`** is the SEMANTIC complement to `snapshot_schema.py` (stdlib-only, no deps): schema guards STRUCTURE (missing keys, empty lists, NaN leaks); this measures how well the enrichment merges actually POPULATED the scoring fields, so a silent Statcast drop (e.g. WhiffPctile 92% → 60%) — which passes schema — is surfaced. `coverage_report(snap)` → structured dict; `format_report(rep)` → ASCII console report (per-field % bars + OK/WARN/LOW per group); `worst_status(rep)` → the worst group status for CI. Groups: **pitcher Savant** (`xERA`/`xwOBA_against`/`WhiffPctile`/`BarrelPctAllowed`/`HardHitPctAllowed` — **denominator excludes `Source=="ESPN"` off-FP seeded rows**, which are legitimately below Savant's qualifier so a call-up influx can't false-trip the badge), **hitter Savant** (`xwOBA`/`xBA`/`xSLG`/`SprintSpeed`/`Barrel_Pct`/`HardHit_Pct`), **hitter model** (`HR_Probability`/`wRCplus`), **FA injury-status match** (`FA_Matched` over the free-agent pool — the fraction of unrostered FantasyPros-ranked players ESPN's `free_agents()` pull actually returned a live status for; gates on the standard `_HEALTHY`/`_WARN` fractions since the `_FA_PULL_SIZE=1000` fix gets this to ~100% under normal conditions — a drop here means the FA pull size regressed or the ESPN endpoint degraded, the exact class of gap that let an IL10 free agent show as a clean pickup before that fix), **recent-form windows** (7/15/30 row counts), **probable starters** (confirmed% — info-only, legitimately low early in the week), and freshness. Presence is sentinel-aware (a real positive value; -1/0/NaN = missing). Thresholds are module constants (`_HEALTHY`=0.85 → OK, `_WARN`=0.70 → WARN, below → LOW; population-limited fields like SVHD are deliberately NOT coverage metrics). Wired into `daily-digest.yml` as a **non-blocking** post-send step (`|| true`, no `--strict`) that prints the report to the Actions log; `--strict` exits 1 on LOW for anyone who wants to gate on it. **Also surfaced in the digest email footer** via `send_digest.build_coverage_footer(snap)` — a compact one-line badge beneath "Data refreshed …": quiet/muted "Data coverage OK · Statcast ≥N% · recent windows full" on a healthy snapshot, colored + ⚠-prefixed ("Data partially degraded" YELLOW / "Data DEGRADED this run" RED) when a group is WARN/LOW. Wrapped in try/except so it degrades to `""` and never breaks the digest. **When you add a degradation-prone enrichment field, add it to the relevant `_*` group list.**

### Names & merges
- **Team name double-space:** `MY_TEAM_NAME = "Guerrero  Warfare"` (double space) in fetch_data.py matches ESPN exactly. `MY_TEAM = "Guerrero Warfare"` (single space) in send_digest is for display. Never normalize these to match each other.
- **Merge direction:** pitcher/hitter merges start from FantasyPros (left side); every enrichment (ESPN roster/FA, probable starters, Statcast) is a left-join that only fills NaNs on existing FP rows and never adds a row. Players outside the FP top 300 drop from short-range (7-day) views but appear in longer ranges. FantasyPros↔FantasyPros merges (`fp7`, season→all-rows enrich) stay exact. **EXCEPTION — off-FP pitcher seeding (`_seed_offfp_pitchers`, `build_pitcher_data`):** because the left-from-FP merge would silently drop any pitcher absent from the FP scrape (fresh call-ups, low-owned/struggling FAs), synthetic **YEAR** rows are built from ESPN's season breakdown (`get_pitcher_espn_svhd` — extended to also read `ERA`/`WHIP` + the club abbrev + raw `OUTS`) and **`pd.concat`'d onto `fp` BEFORE the coercion/merges**, so they flow through the identical probable-starter + Statcast + season-override pipeline. Gated on a real MLB line (`OUTS>0`, `GP>0`, `K>=0`, `ERA>=0`); deduped vs FP by `_name_key`. Seeded `IP` is written in **baseball notation** (`outs//3 + outs%3/10`) to match FP — a plain `outs/3` decimal would be mis-decoded by `innings_to_decimal` downstream. Every seeded row carries `Source="ESPN"` (FP rows get `Source="FP"`). The existing Savant merges **backfill xERA/whiff for free** on any seeded arm that qualifies (verified: ~99% of seeded rows, since most are established arms who merely fell out of the FP top-300). Thin-sample arms are excluded from `compute_score_calibration` (below the IP reliability floor) so they can't distort scores, and the projection's `_ERA_REG_PRIOR_IP=40` shrinkage regresses their line toward league ERA. **HITTERS are seeded the SAME way (`_seed_offfp_hitters`, `build_hitter_data`):** `get_hitter_espn_stats(league)` (a SEED-ONLY puller — never merged back onto existing rows, so no `ESPN_*` hitter columns pollute the snapshot, UNLIKE the pitcher `get_pitcher_espn_svhd` which doubles as the SVHD season-override source) reads ESPN's hitting breakdown (`AB/R/HR/RBI/SB` + `AVG/OBP/SLG/OPS`, batter Ks under ESPN's **`B_SO`** key → FP's hitter `K` column, plus `proTeam`), and synthetic **YEAR** rows are `pd.concat`'d onto `fp` BEFORE the merges. Gated on a real MLB line (`AB>0`); deduped by `_name_key`. No IP-notation issue (hitter counts are plain ints). Downstream wRC+ derives from the seeded `OPS`, ISO from `SLG−AVG`, and Statcast backfills for free where the bat qualifies. **Percentile note:** unlike the plan's premise, the `hit_pctile` AB-floor does NOT fully exclude seeds — most rescued bats are established-but-struggling vets with real AB (200+), so ~68 of them enter the pool and nudge the 80th-pctile badge thresholds down modestly (~1 HR / ~1 SB / 0.01 OPS). This is small, defensible (a more complete population), and the same accepted dilution as the pitcher `pit_pctile`. `data_coverage.py` excludes `Source=="ESPN"` rows from BOTH the pitcher- and hitter-Savant denominators AND the hitter-model group so a call-up influx (legitimately below Savant's qualifier / lacking `HR_Probability`/`wRCplus` inputs) can't false-trip the degradation badge.
- **`merge_on_name` / `_name_key`:** ESPN and FantasyPros names differ by accents and generational suffixes (`Luis García Jr.` vs `Luis Garcia`); an exact-string merge drops the roster link so a rostered player wrongly shows as a free agent. `merge_on_name` does the exact merge first, then fills still-unmatched rows via `_name_key` (accent-stripped, lowercased, `.`/`,`→space, trailing Jr./Sr./II–V removed, ≥ 2 tokens kept so a real name never collapses). Invariants: (a) exact matches always win — the fallback only fills NaNs; (b) a key is trusted only when it maps to a single player on **both** sides (the several MLB "Luis Garcia" pitchers stay ambiguous, never guessed). Wired into both roster+FA merges AND the Statcast/pybaseball merges. Older per-player `HITTER_NAME_PATCHES`/`PITCHER_NAME_PATCHES` still applied.
  - **ONE canonical `_name_key`, in `name_utils.py`** (a stdlib-only leaf module) — shared by `fetch_data`, `send_digest`, and `weekly_recap` (was 3 subtly-different copies that disagreed on periods/apostrophes/hyphens). `send_digest._badge_name_key` is now a **back-compat alias** of it (dashboard/docs still reference that name). It **preserves apostrophes/hyphens** (only `.`/`,` → space), so upstream scrape artifacts must be cleaned BEFORE keying — see `_fix_mojibake`.
- **Index-alignment invariant:** build the fill-loop keys `fkeys` from `merged["PlayerName"]`, NOT `fp["PlayerName"]` — `fp.merge()` resets to a clean RangeIndex, so keys from `fp` can be unalignable with the `missing` mask from `merged` → `IndexingError: Unalignable boolean Series`.
- **Statcast name matching:** `lf_to_name()` converts Baseball Savant "Last, First" → "First Last" AND strips accents (`Ramírez, José` → `Jose Ramirez`) so the merge against ASCII FantasyPros names succeeds. Without it, accented-name players silently lose all Statcast data.

### Fields & sentinels
- **PSP sentinel:** `PSP_Date = "1999-01-01"` = no upcoming start (set by `build_pitcher_data`'s `fillna` for pitchers absent from the probable-starters pull). `PSP_Projected = True` = an ESPN-projected start ≥ `_PROJECTED_MIN_DAYS_OUT` days out (vs a near-term one treated as confirmed) — see the Probable starters gotcha.
- **Two-start pitchers (`PSP_Dates`):** fetch_data preserves a list of ALL upcoming start dates per pitcher (`PSP_Dates` + parallel `PSP_HomeVAways`) via `_attach_start_lists` before the one-row-per-pitcher dedup. Scalar `PSP_Date`/`PSP_HomeVAway`/`PSP_Projected` remain the earliest start. `_starts_this_week(r, today, week_end)` counts entries within the matchup week. ≥ 2 starts Mon–Sun → bold **blue** (`ACCENT`) chip labelled just `2` (`two_start_badge()`) in FA SP + My Upcoming Starts, and preferred (secondary sort key, NOT a score change) in the Week-at-a-Glance best-FA-SP bullet. Never fold two-start into the 0–100 score.
- **B_SO is lower-is-better BUT a COUNTING stat (not a rate):** batter strikeouts sit in `_LOWER_BETTER` alongside ERA/WHIP. Affects Category Pulse bar direction and projection flip logic — fewer B_SO than the opponent is a win. **Do NOT add it to `_RATE_CATS`** — it accumulates over the week (values ~70), so `_project` must use the counting branch (`current + remaining·avg`, monotonically increasing), not the rate blend (which pulls a high current value *down* toward the weekly avg → a projection below the current count). Its `_CLOSE_THRESH` is on the counting scale (`8`, matching R/RBI/K), not the rate scale. `_RATE_CATS`/`_SEASON_RATE_CATS` are `{OPS, ERA, WHIP}` only — B_SO is the one lower-better cat that is NOT a rate.
- **`ESPN_OnIL`** (native python bool = `pl.lineupSlot == "IL" or pl.injured`) is captured in `get_pitcher_roster`/`get_hitter_roster` and broadcast to all rows. `lineupSlot` is the primary check (slot 17 in ESPN's POSITION_MAP); `pl.injured` is a fallback because ESPN's API sometimes omits `lineupSlotId` from the roster entry (e.g. Will Smith on the 60-day IL), leaving `lineupSlot = ''`. Keep it a **native bool**, NOT `.astype(bool)` → numpy `bool_`, which `json.dump(default=str)` stringifies to the truthy `"False"`.

### FA logic
- **FA exclusion:** players claimed today are found via today's `transactions` list; the *most recent* transaction per player wins. FA views + positional breakdown exclude all `_DL_STATUSES`.
- **FA RP requires SVHD ≥ 1:** `fa_relievers` gates on `(_n(r.get("ESPN_SVHD")) or _n(r.get("SVHD"))) >= 1`. Zero saves + zero holds all season = no role.
- **FA SP score floor (`_FA_SP_MIN_SCORE = 35`):** `fa_starters` drops any starter whose `_score_p` is below 35 (streamer-tier arms) BEFORE the top-12 slice. Applied inside `fa_starters` so all three consumers agree (the FA SP section, the Week-at-a-Glance best-FA-SP bullet, and `_roster_suggestion`'s pitch-stabilizer pool). Tune the one constant; the section subtitle + empty-state message interpolate it.
- **FA SP thinness-aware per-day display cap (`_FA_SP_PER_DAY_CAP = 3`, floor `_FA_SP_PER_DAY_MIN = 2`):** the FA SP **section** (NOT `fa_starters` — this is display-only, so the bullet + stabilizer pool still see the full quality pool) caps each day to `max(_FA_SP_PER_DAY_MIN, _FA_SP_PER_DAY_CAP − my_count)` arms, where `my_count` = my own starts that day (`my_starts_by_day`). A day I'm already covered on (2+ starts) shrinks **toward** its best arms but **always shows at least 2 when the day has them** (`_FA_SP_PER_DAY_MIN`, so a covered day is never a lone arm); a thin day (0 starts) shows the full 3. The `[:day_cap]` slice can't invent arms, so a day with only 1 available FA SP still shows 1. `day_pitchers` arrives score-desc so the slice keeps the top arms; the day header appends `(+N more)` when arms are hidden. Rationale: 7 moves/week means a 12-arm wall is noise — surface depth only where the schedule leaves me thin, but never so aggressively that a covered day hides a viable second option.
- **FA "Cats" column (`_cats_cell` / `player_cat_strengths` / `build_cat_percentiles`):** FA Hitters + FA Relievers show a **Cats** column (before Score) listing up to 3 roto cats the player is strong in (percentile ≥ 0.70 within a qualified YEAR pool; `_LOWER_BETTER` inverted). Cats in `need_cats` (my currently-losing ∪ tossup) render `ACCENT`; others `MUTED`. `_FA_HIT_CATS=[R,HR,RBI,SB,OPS]`, `_FA_RP_CATS=[SVHD,K,W,ERA,WHIP]`.

### Roster/drop rules
- **Never drop an IL-slot player (`_on_il`):** the league has **2 dedicated IL slots** that don't consume active/bench room, so dropping a player parked there frees nothing. `_on_il(r)` (tolerant of a stringified `"true"`) gates two drop paths: (a) `_can_drop` returns False (Week-at-a-Glance bullet 4); (b) `positional_breakdown`'s `worst_player` uses `drop_pool = [r for r in my_p if not _on_il(r)] or my_p`. It is lineup-SLOT-specific, NOT injury status — a DL player on the BENCH is `False` and stays droppable. Populated for all teams; only my rows affect my suggestions.
- **`_can_drop`** guards that every position keeps ≥ 1 healthy player.

### SP/RP role detection (`_is_sp(r)`)
Never use `"SP" in pos` or `gs > 3` alone. Priority chain: ESPN season GS/GP ratio (≥ 5 appearances) → dataset GS/G ratio (≥ 4 appearances) → IP/G → Position field. Thresholds: GS/G ≥ 0.80 → SP, ≤ 0.20 → RP; IP/G ≥ 4.5 → SP, < 2.5 → RP. Used by `pitcher_score`, `_score_p`, `fa_starters`, `fa_relievers`, My RP filter, `positional_breakdown`. **The single canonical usage-role fn** at every score/filter/trade site (fetch_data's `_espn_is_reliever` + `todays_games` `is_sp`=tonight's-confirmed-probable are a deliberately separate EVENT semantic, not usage).

### Matchup W-L uses ESPN's `result` field, NOT raw comparison
`get_all_matchups`/`get_all_prev_matchups` read `box.home_stats[cat]["result"]` (`WIN`/`LOSS`/`TIE`) as source of truth (→ `W`/`L`/`T`). Critical because ESPN applies a ratio-stat **innings-pitched minimum** (~25 IP) before ERA/WHIP count — the better WHIP under the IP floor **loses** (rationale + the 10-2 vs 11-1 bug → NOTES). **Intended consequence:** the current result can differ from the projected result (Category Pulse compares raw projected values without the floor) — don't unify, the divergence is informative. **Falls back to a raw value comparison when `result` is `None`** — essential because ESPN leaves per-category `result` unset for the *live* matchup until it closes.

### `roto_score_week` ranks by VALUE, not `result` (so the live current week populates)
Ranks teams on each category's `value` (coerced numeric), NOT ESPN's per-category `result`. Deliberate: ESPN leaves `result = None` on the live matchup until it closes, so a result-gated filter would drop the whole current week → `_all_roto_tied` True → **silently suppresses digest sections 4 + 4b** mid-week. Ranking by value is identical for completed weeks and additive for the live week. `get_all_roto` uses `continue` (not `break`) on a failed week so one bad week can't truncate history.

**Compacted 8-column pitcher tables (My Upcoming Starts + FA SP)** — kept narrow (they overflowed iPad width) by: (1) local tight style vars `_th`/`_tdc`/`_tds` = `TH_S`/`TDC`/`TD_S` with padding `10px→6px`, font `13px→12px` (via `.replace()`, scoped to these two blocks — shared styles untouched) + table inline `font-size:12px`; (2) Opp OPS folded into Matchup (9→8 cols). Keep the two tables identical: same padding/font, 8-column layout, `colspan="8"` banner rows, and **`score_reveal(...)` colspan arg = 8** (a stale 9 short-changes the breakdown row).

## Automation

- **GitHub Actions (two scheduled workflows + one manual, cron always UTC):** `daily-digest.yml` **06:00 UTC** once daily runs **`python send_digest.py --with-dashboard`** — ONE fetch, ONE email carrying BOTH the digest (inline body + `digest_*.html` attachment) AND the dashboard (`dashboard_*.html` attachment); `weekly-recap.yml` Monday 15:30 UTC. `daily-dashboard.yml` **schedule is RETIRED** (kept as `workflow_dispatch`-only for manual standalone dashboard sends). **Cron = when the job FIRES, not when it ARRIVES** — GitHub's scheduler runs ~6 h late in practice (only ever LATER), so the morning job lands ~8 AM ET and fetches well past ESPN's overnight rollover. All share repo secrets `GMAIL_APP_PASSWORD`, `ESPN_SWID`, `ESPN_S2`. Decision: stay on GitHub Actions — do not migrate to cron-job.org unless revisited. (Timing rationale → [[feedback-github-actions-delay]] memory.) `pocket-tradelab.yml` (workflow_dispatch + repository_dispatch, **no schedule**) refreshes + deploys the phone Trade Lab (see the trade_lab.py Pocket mode note).
  - **`--with-dashboard` (`send_digest.main`):** after building the digest, imports `dashboard` (deferred — dashboard imports send_digest at module load, so the import must be inside `main()` at call time to avoid a circular import), calls `dashboard.build_dashboard(snap, my_team)`, and passes `(dash_html, filename)` as `extra_attachments` to `send_email` (which takes an optional `extra_attachments` list of `(html, filename)` tuples appended after the digest attachment). **The dashboard build is wrapped in try/except** — a dashboard failure only drops its attachment; the digest still sends. Under `--dry-run` it also writes `previews/dashboard_{slug}.html`.
- **Fetch-time freshness badge:** the record and category standings are a point-in-time snapshot of ESPN's live box scores at fetch time. The header badge shows the fetch **time in ET** (`_fmt_refresh_time`): green ✓ when data is from today, yellow ⚠ when stale. `fetch_data.py` writes `refreshed_at` as tz-aware UTC (`datetime.now(timezone.utc)`) so the ET conversion is correct on CI (UTC) or local; `_fmt_refresh_time` converts to `America/New_York`, shows naive stamps as-is. `_data_fresh` keys off the date only.
- **Local runner:** `scripts/run_digest.bat` for manual runs. Captures console output (incl. tracebacks) to `logs/run_console.log`; the structured one-line-per-send record goes to `logs/digest.log` (written by send_digest.py). Kept in **separate files** — the old shared handle collided (`PermissionError` on Windows). The Python write is wrapped so a locked log can't crash a run that already sent.

## Environment notes

- fetch_data.py on Windows: ASCII-only in print/log strings (charmap encoding crashes on Unicode).
- ESPN credentials (ESPN_SWID, ESPN_S2) are env-only — `.env` locally (fetch_data.py loads dotenv), GitHub Actions secrets on CI. NEVER hardcode them in source: the repo is public. fetch_data.py exits with a clear error when they're missing.
